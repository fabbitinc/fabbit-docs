# 대표자 창업동기 전문성 스토리

이 문서는 대표자 스토리를 구조화해 발표와 사업계획서에 일관되게 반영하기 위한
기준 문서다. 원본 팩트와 사업계획서용 서술을 분리해서 관리한다.

## 1. 문제와의 연결

### 원본 팩트

- 첫 직장이 코너스톤 테크놀로지 (중소기업 대상 PLM 솔루션 회사, 5인 기업)
- BOM 관리, 프로젝트 관리 모듈 개발 담당
- 현장에서 목격한 문제:
  - 고객사마다 커스텀 기능을 만들어주면서 SI 회사처럼 변질됨
  - 설정 옵션이 100개 이상으로 불어남 → 누구도 쉽게 쓸 수 없는 제품
  - 쿼리 10초 등 기술 부채 누적
  - 성능 개선·구조 개선 제안을 근거와 해결방안까지 가져갔지만, 대표는 기능 개발 외에는 우선순위가 낮다고 판단
- 핵심 깨달음: 중소기업 PLM의 셀링포인트는 기능이 많은 게 아니라 **빠르게 도입하고 쉽게 쓰는 것**
- 최근 코너스톤 대표와 만나 토론 → 대표도 "네 말이 맞았던 것 같다, 한번 해봐라"라고 인정

### 사업계획서용 서술

> PLM 솔루션 기업에서 BOM 관리 모듈을 직접 개발하며, 중소 제조업의 제조 데이터 문제를 목격했습니다. 도면은 폴더에, BOM은 엑셀에, 공급사 정보는 담당자 머릿속에 갇혀 있어 검색·변경추적·의사결정이 불가능했습니다. 고객은 "빠른 도입과 쉬운 사용"을 원했지만, 고객사별 커스텀 개발이 반복되면서 제품은 복잡해지고 도입 장벽은 높아졌습니다. 근본 원인은 데이터가 구조화되지 않은 상태에서 관리 기능만 얹으려 한 것이었습니다.

## 2. 대표자 전문성

### 원본 팩트

- 학력: 컴퓨터공학 전공
- 경력 (총 3년 5개월):
  - 코너스톤 테크놀로지 (PLM 솔루션) 1년: BOM 관리·프로젝트 관리 모듈 개발
  - SI 회사 (데이터베이스 모니터링) 1년 반: 프론트엔드, 백엔드, 인프라 등 풀스택 담당
- 두 번째 회사에서 3년차(사원)임에도 팀장급 역할 수행:
  - 팀장이 프로젝트를 위임 → 실행계획·기획 검토·팀원(8명) 업무 분배·일정 관리 직접 수행
- 기술: 프론트엔드, 백엔드, 인프라, 배포까지 1인 풀스택 가능
- 자격: AWS 자격증
- 특성: 워커홀릭. 재직 시 "이렇게 일하는 사람 처음 본다"는 평가를 일관되게 받음

### 사업계획서용 서술

> 컴퓨터공학 전공 후 PLM 솔루션 기업에서 BOM 관리 모듈을 개발하며 제조업 도메인 지식을 확보했습니다. 이후 SI 기업에서 프론트엔드·백엔드·인프라를 아우르는 풀스택 역량을 쌓았으며, 8명 규모 프로젝트의 기획·실행·일정 관리를 주도한 경험이 있습니다. PLM 도메인 이해와 풀스택 개발 역량을 겸비하여, MVP 설계부터 구현·운영까지 1인 실행이 가능합니다.

## 3. 팀 구성

### 원본 팩트

| 역할             | 이름   | 배경                                                          | 강점                                                                          |
| ---------------- | ------ | ------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| 대표 / 설계·개발 | 문성하 | 코너스톤(PLM) → SI                                            | PLM 도메인 + 풀스택 개발 + 프로젝트 관리                                      |
| 영업·기술지원    | 남형우 | 고려대 생명공학과, SI 회사에서 DB 모니터링 제품 기술지원 담당 | 고객 커뮤니케이션에 강함. 이미 콜드콜·컨퍼런스 영업 실행 중. 기술 이해도 보유 |

- 남형우는 합류(팀원) 형태
- 역할 분담: 대표가 제품, 남형우가 시장(영업·고객 접점)
- 부족한 역할: 추후 도메인 전문가(제조업 현장 경력) 영입 필요

### 사업계획서용 서술

> 대표가 제품 설계·개발 전체를 담당하고, 팀원 남형우가 영업·고객 접점을 담당하는 **제품-시장 2인 체제**입니다. 남형우는 기술 제품의 고객 지원 경험을 보유하여 기술 이해를 바탕으로 한 고객 커뮤니케이션에 강점이 있으며, 이미 콜드콜 36건+, 산업 컨퍼런스 현장 영업을 실행하고 있습니다. 향후 제조업 현장 경력을 보유한 도메인 전문가 영입을 계획하고 있습니다.

## 4. 창업 동기

### 원본 팩트

- 퇴사 후 PLM 업계에서 느꼈던 문제를 코너스톤 대표와 다시 토론 → 대표가 "네 말이 맞았다, 해봐라"
- AI 발전으로 1인 풀스택 개발이 가능한 시대 → MVP·초기 사업모델까지 혼자 만들 수 있겠다는 판단
- 시작해보니 시장성도 확인됨: PLM 경쟁자가 많지 않고, 비링커는 PLM이 아닌 위탁생산으로 방향 전환

### 사업계획서용 서술

> PLM 업계에서 직접 경험한 문제의식을 바탕으로, AI 기술의 성숙과 제조 데이터 플랫폼 시장의 공백을 확인하고 창업을 결정했습니다. 기존 PLM은 데이터 연결 없이 관리 기능만 제공하고, AI 추출 도구는 추출(→Excel)에서 끝납니다. AI로 데이터를 쌓고, 쌓인 데이터 위에서 AI가 의사결정을 지원하는 플랫폼은 아직 없습니다.

## 5. 실행 의지와 계획

### 원본 팩트

- 12개월 목표:
  - 50인 이하 중소기업 타겟으로 초기 사업모델 빠르게 완성
  - 국내 검증 후 해외 진출 (동아시아 우선 — 북미보다 늦게 제조업 부상한 시장)
  - 개발 고도화보다 다국어 처리가 비용 대비 효과적 → 캐시카우 확보 우선
- 핵심 리스크: 영업
  - 아이템 자체는 나쁘지 않지만, 제품이 알아서 팔리는 수준의 기술력은 아님
  - 만들어놓고 영업 못하면 끝 → 영업 실행력이 사업 성패를 결정
  - 대응: 남형우가 영업 전담, 이미 콜드콜·컨퍼런스 실행 중

### 사업계획서용 서술

> **12개월 목표:** 50인 이하 중소 제조기업 대상으로 MVP 검증 및 유료 전환을 달성하고, 국내에서 검증된 제품을 동아시아 시장(일본·동남아)으로 확장합니다. 클라우드 SaaS 구조로 다국어 대응이 용이하여, 개발 고도화 대비 낮은 비용으로 해외 시장 진입이 가능합니다.
>
> **리스크 대응:** 가장 큰 리스크는 초기 고객 확보입니다. 이를 위해 영업 전담 팀원이 콜드콜·컨퍼런스·오픈톡방 등 다채널 영업을 이미 실행 중이며, 스마트공장 바우처·클라우드 바우처를 활용한 도입 장벽 해소 전략을 병행합니다.

## 6. AI 비전 (Scale-up용)

### 원본 팩트

- 대부분의 AI 서비스는 기존 데이터를 활용하는 수준 (MCP, Skills, RAG)
- Fabbit은 팔란티어처럼 제품 깊숙이 AI가 들어가 있는 서비스를 지향
- 사용자가 인지하지 못할 때도 AI가 활동하는 구조
- 도면 AI 분석이 가장 먼저 개발된 이유: 진입장벽 해소 (쉬운 도입)를 위한 전략적 선택

### 사업계획서용 서술

> Fabbit의 AI는 단순 보조 도구가 아닌 **제품의 핵심 엔진**입니다. 도면 업로드 시 AI가 BOM을 자동 생성하는 것은 시작점이며, 축적된 제조 데이터(부품·공급사·담당자)를 온톨로지 기반으로 연결하여 변경 영향 추적, 원가 분석, 리스크 감지까지 자동화하는 것이 궁극적 목표입니다. AI가 제품 깊숙이 내재되어, 사용자가 별도로 AI를 호출하지 않아도 의사결정에 필요한 인사이트를 제공하는 구조를 설계하고 있습니다.

## 7. 약점과 보강

### 약점 분석

| 약점 | 심각도 | 상세 |
|------|--------|------|
| 경력 3.5년, 스펙 약함 | 높음 | 지방대, 수상/특허 없음. 스펙으로는 설득력 부족 |
| 팀에 제조업 현장 경력자 없음 | 높음 | 대표는 PLM 솔루션 개발, 팀원은 DB 모니터링 기술지원 출신. 제조 현장을 아는 사람이 없음 |
| 고객 Pain 직접 목격 근거 약함 | 중간 | 코너스톤 재직 중 고객사 방문/도입 지원 경험 없음. 공급자(솔루션) 관점에서 문제를 본 것 |
| 12개월 해외 진출 비현실적 | 중간 | 국내 PMF 미검증 상태에서 해외 진출 언급은 평가위원에게 부정적 |
| 코너스톤 사명 언급 리스크 | 낮음 | 경쟁사 분석에 이미 포함. "5인 기업에서 1년"이 전문성으로 충분한가 의문 |

### 보강 전략

**전문성 (스펙 → 실행 증거로 전환):**
- MVP 구현 현황이 최고의 전문성 증빙: API 150개, 서버/웹/인프라 1인 구축
- "경력 10년"이 아니라 "이미 만들어서 돌아가고 있다"로 승부
- 사업계획서에 제품 스크린샷·아키텍처도를 첨부

**제조업 도메인 보강 — 자문 네트워크:**

| 자문위원 후보 | 커버 영역 | 상태 |
|-------------|----------|------|
| 코너스톤 대표 | PLM 도메인/업계 | 다음 주 미팅 예정, 자문 요청 필요 |
| 제조업 품질팀 지인 | 제조 현장/수요자 (PLM 실사용자) | 컨택 가능 (보안 이슈로 소속 비공개 가능) |
| 전 직장 기술마케팅 전무 | B2B SaaS 영업/GTM (IBM→메가존 등) | 컨택 가능 |

미팅 질문 리스트: [[40_타겟/인터뷰/자문-네트워크-미팅-준비]]

> [!info] 자문위원이 들어가면
> "팀에 제조업 경력자가 없다"는 약점이 "업계 전문가 자문 체계를 구축했다"로 전환됨.
> 사업계획서 팀 구성 항목에 "자문위원" 란이 보통 있음.

**고객 Pain 보강:**
- 코너스톤 대표/직원에게 고객 사례 확인 (다음 주 미팅 활용)
- 제조업 품질팀 지인에게 제품 피드백 + 현장 Pain 청취
- 2차 인터뷰 (10~49명 규모) 병행

**해외 진출:**
- 사업계획서에서는 12개월 목표를 국내 PMF로 한정
- 해외는 2~3년차 성장 비전으로 위치 변경

**코너스톤 사명:**
- 사업계획서에서는 "국내 PLM 솔루션 기업"으로 익명 처리 권장
- 발표 심사에서 질문 나오면 구두로 답변
