### 과제명

**AI 기반 중소 제조업 도면·BOM 관리 솔루션**

### 창업과제 소개

- 중소 제조업의 도면·BOM 데이터를 엑셀, 공유폴더, NAS에서 시스템으로
  이전·관리하는 솔루션
- MVP 핵심 기능: 고객 BOM 엑셀 헤더와 샘플 데이터를 LLM으로 분석 후
  Fabbit 데이터베이스 필드 매핑 규칙 자동 생성
- 동일 양식 엑셀 재업로드 시 매핑 규칙 기반 자동 적재 구조
- 기존 엑셀 양식 유지 가능, 초기 등록 부담 및 이전 부담 완화
- 핵심 타깃: 설계·생산·구매가 동일 데이터를 함께 사용하지만 전용 시스템
  없이 운영하는 10~49명 규모 제조기업

### 배경 및 필요성

- 실제 경쟁 구도 출발점: 타 소프트웨어보다 엑셀, 공유폴더, NAS
- 고객은 파일 증가 시 NAS 도입, 그러나 저장 위치 통합 외 최신본 식별,
  변경 승인, 배포 통제, 도면-BOM 정합성 해결 한계
- `공유는 되지만 관리가 안 되는` 시점에서 도면관리 솔루션 수요 발생
- 핵심 과제: 엑셀/NAS 운영 단계에서 시스템 도입 단계로 넘어가는 전환 장벽
  완화

### 현황 및 구체화 방안

- 현재 MVP 개발 완료 상태
- 핵심 동작: 고객 BOM 엑셀 속성을 AI로 해석해 시스템 필드에 연결하는
  매핑 기능
- 협약기간 집중 과제: 파일럿 운영, 유료 전환 검증
- 우선 해결 과제: 기존 속성명·분류 체계를 최대한 유지하는 매핑 기반
  마이그레이션 강화
- 목표: `도입할 수는 있지만 옮기기 어려운 문제` 해결

### 목표시장 및 사업화 전략

- 초기 목표시장: 경기도권 자동화기기, 특수장비, 시스템 제작업체
- 고객 확보 방식: 직접 영업, 현장 인터뷰 기반 파일럿 고객 발굴
- 확장 경로: 바우처 사업, ERP 벤더 협력
- 시장 판단: 공백 시장이 아닌 기존 도면관리 시장 내 진입
- 핵심 포지션: `엑셀/NAS 단계에서 실제 도입으로 넘어가는 구간` 공략

## 1. 문제인식 (Problem)

- 공식 양식의 `1-1. 창업과제 배경 및 필요성`, `1-2. 창업과제 목표시장(고객)
  현황 분석` 기준 작성

### 1-1. 창업과제 배경 및 필요성

- 제조 데이터 관리 현실: 엑셀, 폴더, 담당자 개인 경험 의존 구조
- 국내 제조업 537,580개 사업체 중 10~299명 규모 중소기업 77,389개
  수준(통계청 전국사업체조사, 2023)
- 현장 일반 상황: ERP는 보유, 도면과 BOM은 별도 폴더·엑셀 관리
- 외부 환경 악화 요인: 공급망 불안정, 설계 변경 빈도 증가
- 현장 리스크: 부품 변경 영향 범위 파악 지연, 구버전 도면 생산 사고 가능성
- 고객 구매 여정: `엑셀/개인PC -> 공유폴더 -> NAS` 순으로 버틴 뒤 변경관리·
  배포 통제 한계 시점에서 솔루션 검토
- 내부 검증 배경: 대표의 PLM 솔루션 기업 BOM 관리 모듈 개발 경험
- 기존 솔루션 한계: 기능은 충분하나 자사 양식·속성 체계 재정리 부담으로
  초기 도입 피로도 높음
- 핵심 문제 정의: `기능 부족`보다 `기존 방식에서 시스템으로 넘어오는 과정`
  자체의 높은 마찰 비용

```text
개인PC / 엑셀
      ↓
공유폴더
      ↓
NAS / 서버 저장
      ↓
최신본 혼선, 리비전 추적 한계, 변경 반영 누락
      ↓
도면관리 솔루션 검토
```

- {그래프 설명: 개인PC/엑셀 중심 운영에서 공유폴더, NAS로 저장 위치만 확장되지만
  최신본 식별·리비전 추적·승인/배포 통제 문제는 그대로 남는 흐름을 화살표형
  도식으로 표현}
![[SCR-20260315-opsc.png]]
- 자문·인터뷰 기반 현장 문제 요약
  - 자문 인터뷰: 고객이 가장 많이 호소하는 문제는 `도면관리`, 특히 최종 버전
    관리와 검토→승인→배포→결재 프로세스 부재
  - 2차 성공 인터뷰: 고객은 BOM 자동화보다 `도면 최신본 관리`와 `리비전 추적`
    문제를 더 크게 인식
  - 2차 성공 인터뷰: 여러 설계 툴을 함께 사용할수록 정합성이 무너지고, 최신
    승인본과 기준 리비전 추적이 어려워짐
  - 2차 성공 인터뷰: 예전에 만든 도면·모듈·기성품을 다시 찾지 못해 설계 자산이
    개인 기억과 폴더 구조에 묶이는 문제 확인
  - 설계 변경이 잦은 환경일수록 어떤 도면이 적용됐고 중간에 무엇이 수정됐는지
    추적이 어려워 운영 리스크 확대

- Fabbit이 정의한 핵심 문제 3가지

| 현장 문제 | 설명 |
| --- | --- |
| 최신본 관리 부재 | 도면이 여러 폴더와 개인 PC에 분산돼 유효본 즉시 식별 어려움 |
| 변경관리 미흡 | 검토·승인·배포 이력 부재로 변경 영향 추적과 책임 구분 어려움 |
| 마이그레이션 부담 | 기존 파일 구조와 속성 체계를 유지한 채 점진적으로 옮기기 어렵다는 도입 저해요인 반복 확인 |

- PLM 업계 자문 인터뷰(2026년 3월 13일) 재확인 사항: 고객의 최우선 문제는
  `도면관리`, 특히 최신 버전 관리와 검토·승인·배포 프로세스 부재
- 해결 접근: `기존 엑셀 속성의 AI 매핑 + 매핑 기반 마이그레이션`부터 단계적
  해결

### 1-2. 창업과제 목표시장(고객) 현황 분석

- 초기 목표시장 조건
  - `종사자 10~49명`
  - `매출 10억원 이상`
  - `금속가공(C25) / 기계장비(C29) / 전기장비(C28)` 업종
- 핵심 타깃 특성
  - ERP는 있으나 도면·BOM은 엑셀과 폴더로 별도 운영
  - 설계 인력 2~5명 수준의 소규모 조직
  - 생산·구매·품질이 동일 데이터를 함께 확인해야 하는 구조
- 실제 초기 경쟁상대: 대형 PLM보다 `지금 쓰는 방식`
- 고객의 본질적 요구: 더 큰 저장소보다 최신본 식별, 변경 이력, 배포 통제,
  기존 데이터 이전 부담 완화
- 시장규모는 보수적 기준으로 산정

| 항목 | 내용 |
| --- | --- |
| 세부 목표시장 | 경기도권 자동화기기, 특수장비, 시스템 제작업체 |
| 주요 고객 특성 | ERP는 있으나 도면/BOM은 엑셀과 폴더로 관리, 설계 인력 2~5명 규모 |
| 고객의 현재 대체수단 | 엑셀, 공유폴더, NAS, 메일, 메신저, 일부 기존 PLM |
| 고객 요구사항 | 최신본 표시, 변경 승인/배포 이력, 도면-BOM 연결, 쉬운 초기 등록 |
| 도입 저해요인 | 기존 데이터 정리 부담, 복잡한 초기 셋업, 교육 비용, 현장 적응 부담 |

```text
SOM  ━                     약   3.4억원
```

## 2. 실현가능성 (Solution)

- 현재 준비 상태와 협약기간 내 구체화 방안으로 구분 작성

### 2-1. 창업과제 현황 / 준비정도

- 현재 단계: 시제품(MVP) 단계
- 핵심 기능 구현 완료 상태
- 대표와 팀원이 함께 제품 기획, 서버, 웹, 운영 환경 구축
- 역할 구조: 대표는 기획 총괄, 팀원은 영업 전문 역할 중심 수행
- 개발 역량: 대표와 팀원 모두 풀스택 개발 역량 보유
- 고객 검증 상태: POC 참여 의향 2건 확보, 현장 방문 및 파일럿 협의 진행 중

| 항목    | 현황                                   |
| ----- | ------------------------------------ |
| 제품 단계 | MVP 개발 완료, 시제품 단계                    |
| 제품 상태 | 핵심 기능 구현 완료                          |
| 개발 방식 | 대표·팀원 협업 기반 풀스택 개발                   |
| 검증 현황 | 콜드콜 36건+, 산업 컨퍼런스 현장 영업, 후속 인터뷰 2건   |

### 2-2. 창업과제 실현 및 구체화 방안

- 협약기간 핵심 가치: `쉽게 시작할 수 있는 도입 경험` 구체화
- 제품 방향: 고객의 기존 엑셀 구조 폐기 유도가 아니라 부담 없는 전환 경험
  제공
- 핵심 기능인 AI 기반 BOM 매핑 동작 흐름

```text
┌──────────────────────┐
│ 1. BOM 엑셀 업로드   │
│ 헤더·샘플 데이터 입력 │
└──────────┬───────────┘
           │
           v
┌──────────────────────┐
│ 2. AI 매핑 생성      │
│ 필드 연결 규칙 자동화 │
└──────────┬───────────┘
           │
           v
┌──────────────────────┐
│ 3. 검수 후 자동 적재 │
│ 매핑 저장·재사용     │
└──────────────────────┘
```

- {플로우 설명: 고객이 기존 엑셀 양식을 그대로 업로드하면 AI가 헤더와 샘플
  데이터를 해석해 Fabbit 필드에 자동 매핑하고, 사용자는 결과를 한 번 검수한 뒤
  동일 양식 파일을 반복 업로드할 수 있는 구조. 핵심 메시지는 `기존 양식 변경 없이
  점진적 도입 가능`과 `고객별 상이한 엑셀 구조를 흡수하는 어댑터 역할`}
![[SCR-20260315-opoe.png]]
- 1단계: 고객 업로드 BOM 엑셀의 헤더와 샘플 데이터를 기반으로 컬럼 의미 분석
- 2단계: LLM이 고객 고유 속성명과 Fabbit 시스템 필드의 연결 규칙 자동 생성
- 3단계: 고객 검수 후 동일 양식 엑셀에 매핑 규칙 재적용, 데이터베이스 자동 적재

- 위 흐름 기반 협약기간 구체화 과제 3가지

1. **엑셀 속성 매핑 고도화**
   - LLM 프롬프트 및 후처리 규칙 개선
   - 매핑 정확도 향상
   - 반복 패턴의 업종별 템플릿 자산화
2. **매핑 기반 마이그레이션 강화**
   - 백엔드 등록 구조 전반을 매핑 시스템 중심으로 설계
   - 고객의 기존 속성명, 분류 체계, 엑셀 컬럼 구조 최대한 유지
   - Fabbit의 핵심 도입 차별점
3. **파일럿 운영 기반 개선**
   - 실제 고객 데이터 기반 업종별 템플릿·등록 규칙 정교화
   - 유료 전환 가능한 사용 시나리오 검증

- 고객 관점 경쟁 구도 비교

| 구분 | 엑셀 / 공유폴더 / NAS | AI 추출 도구 | 기존 PLM / 도면관리 | Fabbit |
| --- | --- | --- | --- | --- |
| 초기 사용성 | 익숙하지만 관리 한계가 큼 | 일부 자동화 가능 | 초기 셋업과 교육 부담이 큰 편 | 기존 방식과 유사한 진입 흐름 제공 |
| 데이터 입력 | 수동 정리 | 추출 후 별도 정리 필요 | 수동 등록 또는 자사 구조 적응 필요 | 기존 엑셀 속성을 AI로 해석해 시스템 필드에 매핑 |
| 변경관리 | 사실상 없음 | 없음 | 제공 | 향후 확장 영역 |
| 마이그레이션 | 해당 없음 | 제한적 | 자사 체계 적응 필요가 많은 편 | 기존 속성 체계를 최대한 유지하는 매핑 기반 도입 |
| 목표 포지션 | 현재 버티는 방식 | 보조 도구 | 정식 운영 시스템 | 엑셀/NAS 단계에서 정식 운영 시스템으로 넘어가는 진입 제품 |

- 기술 보호 방향
  - `엑셀 BOM 속성 해석 + 시스템 필드 매핑 + 매핑 기반 데이터 전환 흐름`
    고도화
  - 실제 파일럿 데이터와 업종별 매핑 템플릿의 반복 사용 가능한 내부 자산화

#### 3-1-1. 창업과제 비즈니스 모델

- 비즈니스 모델: `회사 단위 SaaS 과금 + 파일럿 기반 유료 전환`
- 과금 단위 판단 근거: 설계팀 전용 도구가 아닌 생산·구매·품질까지 함께
  사용하는 운영 데이터 성격
- 인당 과금보다 회사 단위 과금 적합

```text
Starter 가입
    ↓
실제 데이터 등록
    ↓
파일럿 운영
    ↓
Team / Business 유료 전환
```

- {그래프 설명: 퍼널 또는 단계형 화살표 그래프로 표현. `무료 진입 → 실제 데이터
  등록 → 파일럿 운영 → 유료 전환` 순서로 배치하고, 각 단계 아래에 핵심 전환
  포인트를 짧게 표기. 예: Starter는 `도입 장벽 제거`, 실제 데이터 등록은
  `실사용 개시`, 파일럿 운영은 `현장 검증`, 유료 전환은 `정식 운영 계약`}

| 구분 | 내용 |
