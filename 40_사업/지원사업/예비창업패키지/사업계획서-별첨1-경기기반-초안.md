## □ 창업 아이템 개요(요약)

| 항목               | 내용                                                                                                                                                                       |
| ---------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 명칭               | **Fabbit**                                                                                                                                                               |
| 범주               | B2B SaaS / 스마트제조 / 인공지능                                                                                                                                                  |
| 아이템 개요           | Fabbit은 중소 제조업의 도면·BOM 데이터를 엑셀, 공유폴더, NAS에서 구조화된 운영 데이터로 전환하는 웹 기반 제조 데이터 SaaS다. 고객의 기존 BOM 엑셀 헤더와 샘플 데이터를 AI가 해석해 시스템 필드 매핑 규칙을 자동 생성하고, 이후 동일 양식 업로드를 반복 가능한 구조로 전환한다. |
| 문제인식 (Problem)   | 중소 제조기업은 ERP 보유 여부와 별개로 도면과 BOM을 여전히 엑셀·폴더 중심으로 관리하는 경우가 많다. 이로 인해 최신본 식별, 리비전 추적, 변경 승인·배포 통제, 도면-BOM 정합성 관리가 수작업에 의존한다.                                                |
| 실현가능성 (Solution) | 현재 MVP 개발을 완료했고, 도면 관리, 부품관리, 승인·결재, 리비전 상태, 변경 이력, BOM 엑셀 업로드, AI 기반 필드 매핑, 매핑 규칙 재사용 기능을 구현했다. 협약기간에는 매핑 정확도 개선, 파일럿 운영, 유료 전환 검증에 집중한다.                               |
| 성장전략 (Scale-up)  | 초기에는 종사자 10~49명 규모 제조기업을 대상으로 `무료 Starter -> Team 유료 전환` 구조로 진입한다. Org와 Enterprise는 중장기 확장 플랜으로 운영하고, 이후 운영 데이터 AI 고도화, 동아시아 제조업 확장 검토, ERP·MES·PLC 연동 기반 운영 플랫폼으로 확장한다.           |
| 팀 구성 (Team)      | 대표는 제품 기획과 개발 총괄을 담당하고, 팀원은 영업과 고객 접점을 맡는다. 대표와 팀원 모두 풀스택 개발 협업 역량을 보유하며, PLM 업계·제조 현장·B2B SaaS GTM 자문 네트워크로 부족한 영역을 보완한다.                                               |

## 1. 문제 인식 (Problem)_창업 아이템의 필요성

### 1-1. 국내 시장 현황과 문제점

- 국내 제조업 537,580개 사업체 중 10~299명 규모 중소기업 77,389개 수준
- ERP 보유 여부와 별개로 도면·BOM은 엑셀, 공유폴더, NAS, 메신저 중심 관리
- 저장 위치는 중앙화되지만 최신본 식별, 변경 승인, 배포 통제, 리비전 추적은 미해결
- 저장은 되지만 관리가 안 되는 단계에서 도면·BOM 관리 솔루션 검토 발생

```text
개인PC / 엑셀
      ↓
공유폴더
      ↓
NAS / 서버 저장
      ↓
최신본 혼선, 리비전 추적 한계, 변경 반영 누락
      ↓
도면·BOM 관리 솔루션 검토
```

- {그래프 필요: 저장 위치는 바뀌지만 관리 문제는 해결되지 않는 흐름}
- {보강 필요: 국내 제조업 통계, 현장 운영 근거 출처 표기}

### 1-2. 현장에서 발생하는 손실

| 구분 | 현장 문제 | 발생 손실 |
| --- | --- | --- |
| 설계 | 도면 검색·버전 혼선 | 최신본 확인 지연, 검토 커뮤니케이션 비용 증가 |
| 설계 | BOM 수동 입력 및 정리 | 반복 입력 업무 증가, 엔지니어 시간 소모 |
| 설계/생산 연결 | 설계 변경 반영 지연 | 작업지시서와 실제 설계 정보 불일치 |
| 생산 | 불량 원인 역추적 어려움 | 어느 부품·버전·작업에서 문제가 발생했는지 확인 지연 |

- 핵심 문제: 개별 기능 부족보다 `기존 방식에서 시스템으로 넘어오는 과정` 자체의 높은 마찰 비용
- 기존 솔루션 한계: 자사 양식·속성 체계 재정리 부담, 초기 셋업과 교육 부담, 데이터 이전 피로도
- {보강 필요: 시간/비용/오류 수치 2~3개와 출처 삽입}
- {표 필요: 현장 손실 정량 근거 요약표}

### 1-3. 고객 검증 결과와 시급성

- 콜드콜 36건 이상 진행
- 산업 컨퍼런스 현장 영업 진행
- 후속 인터뷰 2건 완료
- 후속 인터뷰 2건 모두 POC 참여 의향 확인

| 검증 항목 | 확인 결과 | 시사점 |
| --- | --- | --- |
| 타깃 고객 | 1~9인 영세기업보다 10~49인 제조기업의 문제의식이 더 명확 | 초기 타깃은 10~49인, 매출 10억 이상 제조기업 |
| 문제 표현 | `PLM`보다 `도면관리`, `변경 반영`, `최신본 관리` 반응이 더 큼 | 제품 메시지는 도메인 용어보다 현장 문제 중심 구성 필요 |
| 반복 pain | 도면 최신본 관리, 리비전 추적, 변경 승인·배포 프로세스 부재 | 우선 기능은 도면·BOM 관리와 마이그레이션 부담 완화 |
| 고객 반응 | 후속 인터뷰 2건 모두 POC 참여 의향 확인 | 문제 인식이 추상적 공감 수준을 넘어 실제 검토 단계 진입 |

1. 설계 변경 빈도와 공급망 변동 증가로 변경관리 중요성 확대
2. 현장 숙련 인력 의존 구조 심화로 지식 유실 위험 확대
3. AI 활용 수요는 높아졌지만, 구조화된 제조 데이터 기반은 부족

- {표 필요: 인터뷰 원자료 요약표 - 업종 / 규모 / 현재 관리 방식 / 가장 큰 pain / POC 의향}
- {보강 필요: 인터뷰 인용 2개 내외 추가}

### 2-2. 차별성 및 경쟁력 확보 전략

- 핵심 차별성: `기존 엑셀 구조를 최대한 유지한 채 시스템으로 전환`
- 경쟁 구도: 기존 PLM과의 직접 비교보다 `현재 고객이 버티는 방식` 전체와의 비교

| 구분 | 엑셀 / 공유폴더 / NAS | AI 추출 도구 | 기존 PLM / 도면관리 | Fabbit |
| --- | --- | --- | --- | --- |
| 초기 사용성 | 익숙하지만 관리 한계 큼 | 일부 자동화 가능 | 초기 셋업과 교육 부담 큼 | 기존 방식과 유사한 진입 흐름 제공 |
| 데이터 입력 | 수동 정리 | 추출 후 별도 정리 필요 | 수동 등록 또는 자사 구조 적응 필요 | 기존 엑셀 속성을 AI로 해석해 시스템 필드에 매핑 |
| 변경관리 | 사실상 없음 | 없음 | 제공 | 리비전·이력 관리 기반 제공 |
| 마이그레이션 | 해당 없음 | 제한적 | 자사 체계 적응 필요 | 기존 속성 체계를 최대한 유지하는 매핑 기반 도입 |
| 목표 포지션 | 현재 버티는 방식 | 보조 도구 | 정식 운영 시스템 | 엑셀/NAS 단계에서 정식 운영 시스템으로 넘어가는 진입 제품 |

```text
1. BOM 엑셀 업로드
        ↓
2. AI가 헤더·샘플 데이터 해석
        ↓
3. 시스템 필드 매핑 규칙 자동 생성
        ↓
4. 사용자 검수 후 자동 적재
        ↓
5. 동일 양식 반복 업로드
```

- {플로우 필요: 기존 양식 유지 -> AI 매핑 -> 검수 -> 반복 업로드}
- {보강 필요: 고객 가치 지표 추가 - 등록 시간 단축, 수동 보정률, 업로드 성공률}

### 3-3. 비즈니스 모델(수익화 모델)

Fabbit은 `무료 진입 -> 실제 데이터 등록 -> 파일럿 운영 -> Team 유료 전환`
구조의 역할별 Seat 기반 Freemium 모델을 채택한다. 협약기간 및 초기 매출
추정은 현재 제공 가능한 `Starter`와 `Team` 플랜 기준으로 산정하고,
`Organization`, `Enterprise`는 중장기 확장 플랜으로만 반영한다.

| 플랜 | 월 요금 | 정책 |
| --- | --- | --- |
| Starter | 무료 | 최대 5명, 총 250MB, 월 100 AI 크레딧 |
| Team | Viewer 5,000원 / Collaborator 15,000원 / Full Seat 29,000원 | 기본 10GB, Full Seat당 +10GB |
| Organization | Viewer 5,000원 / Collaborator 25,000원 / Full Seat 59,000원 | 기본 100GB, Full Seat당 +50GB |
| Enterprise | 별도 문의 | 커스텀 정책, 대용량 운영, 맞춤 연동 |

```text
Starter 가입
    ↓
실제 데이터 등록
    ↓
파일럿 운영
    ↓
Team 유료 전환
```

- {그래프 설명: 퍼널 또는 단계형 화살표 그래프로 표현. `무료 진입 ->
  실사용 개시 -> 현장 검증 -> 정식 운영 계약` 순서로 배치}

- Team 플랜 기본 가정
  - 적용 대상: 8~30명 규모 초기 운영 조직
  - 좌석 구조: Viewer는 조회 중심, Collaborator는 협업·검토 중심, Full Seat는 등록·수정·관리 중심
  - 초기 매출 산정은 평균 유료 좌석 구성을 적용한 Team 플랜 기준 보수 계산

| 항목 | 내용 |
| --- | --- |
| Starter | 무료 체험 및 도입 검토용 진입 플랜 |
| Team | 초기 유료 전환 대상 핵심 플랜 |
| Organization | 다부서 협업 고객 대상 확장 예정 플랜 |
| Enterprise | 맞춤 연동 및 대용량 운영 대상 별도 협의 플랜 |

- Team 플랜 기준 보수적 매출 시나리오

| 연차 | 유료 고객 수 | 산정 기준 | 연매출 |
| --- | --- | --- | --- |
| Year 1 | 5개사 | Team ARPA 242만원 × 5개사 | 약 0.1억원 |

- 초기 3년 매출은 Team 플랜만 반영한 보수적 시나리오
- Org와 Enterprise는 출시 예정 플랜으로, 초기 매출 추정에는 포함하지 않음
