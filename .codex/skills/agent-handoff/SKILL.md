# agent-handoff

기획 문서는 사람 친화적이다. 이 스킬은 그 문서를 구현 에이전트가 바로 쓸 수
있는 작업 지시문으로 압축한다.

핸드오프 문서는 원본 기획을 대체하지 않는다. 원본은 `기획.md`와 `정책.md`고,
`handoff-*.md`는 파생 산출물이다.

## 입력

기본 입력은 아래 순서로 읽는다.

1. 기능 폴더의 `기획.md`
2. 기능 폴더의 `정책.md`
3. 관련 `screens/*.tsx`
4. 관련 `screens/*-*.md`
5. 관련 `flows/*.tsx`

모든 파일을 무조건 읽지 말고, 요청한 역할에 필요한 범위만 읽는다.

- FE 핸드오프: `기획.md`, 관련 화면 와이어프레임, 화면 상세 정책, 필요한 플로우
- BE 핸드오프: `정책.md`, `기획.md`의 범위/MVP, 필요한 플로우
- 둘 다 요청: 공통 사실을 먼저 정리하고 역할별 문서로 분리

## 출력

기능 폴더에 아래 파일을 만든다.

- 프론트엔드용: `handoff-fe.md`
- 백엔드용: `handoff-be.md`

사용자가 한쪽만 요청하면 해당 파일만 만든다.

템플릿은 필요할 때만 아래 참조 문서를 읽는다.

- FE 템플릿: `references/frontend-handoff-template.md`
- BE 템플릿: `references/backend-handoff-template.md`

## 작성 원칙

- 기획의 문장을 반복하지 말고 구현 가능한 작업으로 재구성한다.
- 범위와 비범위를 분리한다.
- 추정이 필요한 내용은 구현 지시로 쓰지 말고 `오픈 이슈`로 분리한다.
- 문서 간 충돌이 있으면 임의로 봉합하지 말고 `충돌/확인 필요` 섹션에 적는다.
- 완료 조건은 검증 가능한 문장으로 쓴다.
- FE와 BE에 공통인 사실은 같아야 한다. 상태 전이, 권한, MVP 범위가 문서마다
  다르면 핸드오프에서 먼저 정렬한다.

## FE 핸드오프에 꼭 들어갈 것

- 목표
- 이번 작업 범위
- 하지 말 것
- 대상 화면과 진입 경로
- 사용자 액션과 상태 변화
- API 가정 또는 필요한 백엔드 계약
- 로딩 / 빈 상태 / 오류 / 읽기 전용 / 권한 없음 / 잠금 상태
- 컴포넌트 또는 화면 단위 완료 조건
- 오픈 이슈

## BE 핸드오프에 꼭 들어갈 것

- 목표
- 이번 작업 범위
- 하지 말 것
- 핵심 유스케이스
- 도메인 규칙
- API / 입력 / 출력 계약
- 검증 규칙과 실패 조건
- 권한 / 잠금 / 감사로그 / 마이그레이션 영향
- 테스트 시나리오
- 오픈 이슈

## 작업 순서

1. 기능 폴더를 정한다.
2. 원본 기획에서 MVP 범위와 후속 범위를 먼저 분리한다.
3. 역할별로 필요한 화면/정책/플로우만 읽는다.
4. 문서 간 충돌을 찾고, 구현 가능한 기준선을 정리한다.
5. 템플릿에 맞춰 `handoff-fe.md`, `handoff-be.md`를 작성한다.
6. 마지막에 다음 세 가지를 점검한다.

- 구현 범위가 너무 넓지 않은가
- 추정 문장이 작업 지시로 섞이지 않았는가
- 완료 조건이 실제 검증 가능한가
