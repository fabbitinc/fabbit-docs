> **[RULE — 반드시 준수]**
>
> 1. 이 스킬은 **제조업 SaaS** 기획 전용이다. 모든 기획은 제조 도메인 개념(BOM, 리비전, 공정, LOT 등)을 기반으로 작성한다.
> 2. **상태 전이를 명시하라**: 엔티티의 상태와 전이 조건·실패 조건을 반드시 정의한다. 암묵적 상태 변경 금지.
> 3. **MVP는 좁게, 확장은 문서로**: 구현 범위(MVP)와 후속 범위를 항상 분리 표기한다. MVP에 없는 기능은 "후속" 태그로 남긴다.
> 4. **추적성은 설계에 남겨둔다**: 당장 구현하지 않더라도, 데이터 변경의 감사로그 구조는 기획 단계에서 고려하고 "후속" 표기한다.
> 5. **기획 산출물만 작성한다**: 코드 구현·DB 마이그레이션·인프라 설정은 이 스킬의 범위가 아니다.
> 6. **정본은 `기획.md`, `정책.md`, `flows/*.tsx`다**: 화면 책임은 `기획.md`, 공통 규칙은 `정책.md`, 업무 흐름은 `flows/*.tsx`를 기준으로 유지한다.
> 7. **와이어프레임 상세 정책은 필요한 곳에만 쓴다**: 기본 CRUD 화면은 요약 문서 1개 또는 무문서로 끝내고, 복잡한 화면만 영역별 상세 정책을 둔다.

# 기획 문서 작성 가이드

사용자의 기획 요청을 받아 `10_제품/기획/` 하위에 문서를 생성한다.

## 폴더 구조 규칙

```
10_제품/기획/
  {모듈}/                    # plm, mes, site 등
    {번호}_{기능명}/          # 예: 04_부품관리
      기획.md                # 기능 전체 기획서
      정책.md                # 기능 전체 정책서
      screens/               # 와이어프레임 + 화면별 상세 정책
        {화면명}.tsx          # 와이어프레임 React 컴포넌트
        {화면명}-요약.md      # 기본 CRUD 화면의 요약 정책 (선택)
        {화면명}-{정책명}.md  # 복잡한 화면의 상세 정책 (선택)
      flows/                 # 사용자/시스템 플로우
        {플로우명}.tsx        # 플로우 React 컴포넌트
```

### 번호 규칙

- 같은 모듈 내 기존 폴더의 번호를 확인하고 다음 번호를 부여
- 두 자리 숫자 + 언더스코어: `01_`, `02_`, ...

## 문서 우선순위 규칙

기획 산출물은 아래 우선순위로 관리한다.

| 산출물 | 역할 | 우선순위 |
|--------|------|----------|
| `기획.md` | 기능 목표, 화면 책임, 사용자 시나리오, 범위 | 최상위 |
| `정책.md` | 상태 전이, 리비전, 권한, 잠금, 검증, 오류 코드 | 최상위 |
| `flows/*.tsx` | 업무 흐름과 분기, 실패 조건의 기준 시각 자료 | 최상위 |
| `screens/*.tsx` | 화면 구조와 UI 배치 검토 자료 | 보조 |
| `screens/*-*.md` | 화면별 차이점과 UI 주의사항 | 보조 |

기획 산출물 간 충돌 시 아래 원칙을 따른다.

- 화면 책임과 범위는 `기획.md`를 우선한다.
- 공통 규칙과 검증은 `정책.md`를 우선한다.
- 업무 분기와 단계 흐름은 `flows/*.tsx`를 우선한다.
- `screens/*.tsx`, `screens/*-*.md`는 정본을 반복 설명하지 않는다.

## 기획.md 작성법

기능의 전체 기획서. 아래 구조를 따른다:

```markdown
---
문서상태: 초안
문서소유자:
최종수정일: { 오늘 날짜 }
관련문서:
  - "[[관련문서 경로]]"
---

# {기능명} 기획

{기능 개요 1~2문장}

## 1. 문제와 목표

## 2. 주요 사용자

## 3. 메뉴명과 진입 경로

## 4. 화면 구성

## 5. 핵심 시나리오

## 6. 사용자/시스템 플로우

## 7. 데이터 연결

## 8. MVP 범위

## 9. FE 구현 메모

## 10. 완료 조건
```

기획.md는 FE가 바로 구현에 들어갈 수 있어야 한다. 최소한 아래를 포함한다:

- 화면별 진입 경로와 주요 CTA
- 로딩 / 빈 상태 / 오류 / 읽기 전용 / 잠금 상태
- 화면에서 필요한 API와 데이터 의존성
- MVP와 후속 확장을 분리한 범위 정의
- 검증 가능한 완료 조건
- 화면 정책 문서 할당 기준
- 플로우별 검토 범위와 책임 분리

기획.md에 아래 원칙을 같이 명시한다.

- 어떤 문서가 SSOT인지
- 어떤 화면은 요약 정책만 두고, 어떤 화면은 상세 정책을 둘지
- `flows/*.tsx`에서 반드시 검토해야 하는 핵심 업무 흐름이 무엇인지

## 정책.md 작성법

기능의 전체 정책서. 비즈니스 규칙, 유효성 검증, 상태 전이 등을 정의한다.

```markdown
---
문서상태: 초안
최종수정일: { 오늘 날짜 }
---

# {기능명} 정책

## 1. 데이터 모델

## 2. 상태 관리

## 3. 권한

## 4. 유효성 검증

## 5. 비즈니스 규칙

## 6. API/이벤트 계약

## 7. BE 구현 메모

## 8. 테스트 시나리오
```

정책.md는 BE가 바로 구현에 들어갈 수 있어야 한다. 최소한 아래를 포함한다:

- 엔티티/필드 단위 제약
- 상태 전이 규칙과 실패 조건
- 권한 매트릭스
- 동시성/잠금/감사로그 정책
- MVP 적용 범위와 후속 정책의 분리
- 테스트해야 할 성공/실패 시나리오
- 화면별 보조 정책 문서는 이 문서의 예외/차이점만 다룬다는 원칙

## 와이어프레임 (screens/\*.tsx) 작성법

와이어프레임은 `.wireframe` 앱에서 렌더링되는 React 컴포넌트다.
공통 컴포넌트를 `@wire`에서 import하여 사용한다.

와이어프레임의 주 역할은 `메인 화면에서 로우 UI와 정보 구조를 빠르게 검토`
하는 것이다. 정책 설명을 메인 화면에 길게 적지 말고, 배치와 블록 구성이 먼저
읽히게 만든다.

### 사용 가능한 공통 컴포넌트

상세 API는 [wire-components.md](wire-components.md)를 참조한다.

- `ScreenHeader` — 화면 제목 + 설명
- `WireBlock` — 보라색 제목의 섹션 블록
- `Field` — 읽기 전용 key:value 필드
- `Badge` — 상태 배지
- `MockButton` — 버튼 (primary 옵션)
- `FormRow` — 폼 라벨 + 입력 행
- `MockInput` — 텍스트 입력 (locked 옵션)
- `MockSelect` — 드롭다운 (options 배열)
- `MockTextarea` — 텍스트에어리어
- `Row` — 그리드 행 (cols: 1/2/3)
- `ButtonGroup` — 버튼 그룹
- `FlowSection` — 플로우 묶음 섹션
- `FlowRow` — 플로우 단계 가로 배치
- `FlowStep` — 플로우 단계 카드
- `FlowArrow` — 단계 연결 표시
- `FlowNote` — 플로우 보조 메모

### 와이어프레임 작성 예시

```tsx
import {
  ScreenHeader,
  WireBlock,
  Field,
  Badge,
  MockButton,
  Row,
  ButtonGroup,
} from "@wire";

export default function 목록() {
  return (
    <div className="p-6 bg-white min-h-full">
      <ScreenHeader title="목록 화면" desc="전체 항목을 조회하는 화면" />
      <Row>
        <WireBlock title="검색 / 필터">
          <Field label="검색">통합 검색</Field>
          <Field label="필터">상태 · 유형</Field>
        </WireBlock>
      </Row>
      <Row>
        <WireBlock title="테이블">
          <Field>
            <Badge status="production" /> 샘플 데이터
          </Field>
        </WireBlock>
      </Row>
      <ButtonGroup>
        <MockButton primary>신규 등록</MockButton>
        <MockButton>내보내기</MockButton>
      </ButtonGroup>
    </div>
  );
}
```

### 메인 와이어프레임 작성 원칙

- 메인 화면은 설명 문장보다 `화면 구조`, `블록 순서`, `정보 묶음`, `CTA 위치`
  확인이 우선이다.
- 메인 화면은 `실제 로우 UI처럼 보여야` 한다. 화면 안에 스펙 메모나
  체크리스트를 직접 박아 넣지 않는다.
- `WireBlock` 제목과 주요 필드만으로 화면 역할이 이해되게 만든다.
- 정책 문장을 와이어프레임 하단에 길게 붙이지 않는다.
- 잠금, 권한, 저장 정책처럼 설명이 필요한 내용도 메인 화면에서는 상태값이나
  배지 수준으로만 표현한다.
- 공통 규칙 설명은 `정책.md`, 블록별 차이점 설명은 `screens/*-*.md`로
  분리한다.

금지 예시는 아래와 같다.

- `상태 피드백` 블록 안에 `Loading: ...`, `Locked: ...`, `Error: ...`를
  문장으로 나열하는 방식
- `저장 전 검증` 블록 안에 필수값, 중복 검사, 서버 오류 코드를 설명문으로
  적는 방식
- 화면 하단에 `Release와 상태 변경은 상세 화면에서 진행` 같은 정책 문장을
  길게 붙이는 방식

대신 아래처럼 표현한다.

- 잠금 상태는 `잠금 배너`, `읽기 전용 필드`, `비활성 저장 버튼`으로 보여준다.
- 오류 상태는 `상단 에러 배너`, `필드 하단 오류 메시지`, `버튼 disabled`로
  보여준다.
- 저장 중 상태는 `버튼 로딩 텍스트`, `스켈레톤`, `오버레이` 등 실제 UI
  요소로 보여준다.
- 상태에 따라 레이아웃이 크게 달라지면 `편집-잠금상태.tsx` 같은 별도 상태
  와이어프레임을 추가로 만든다.

예시:

- 편집 화면의 `잠금 상태` 블록은 `상태`, `보유자`, `잠금 시각` 정도만
  표시한다.
- `자동 해제 기준`, `관리자 강제 해제`, `읽기 전용 진입 규칙` 설명은
  `편집-잠금상태.md`와 `정책.md`에 둔다.
- 상세 화면의 `Release`, `상태 변경`, `삭제` 버튼은 위치와 존재만 보여주고,
  노출 조건은 `상세-액션.md`에 둔다.

## 상세 정책 (screens/{화면명}-{정책명}.md) 작성법

와이어프레임의 각 화면에 대한 세부 정책을 기술한다.
파일명은 `{화면명}-{정책명}.md` 형식이다 (예: `목록-검색필터.md`, `상세-기본정보.md`).

모든 화면에 기계적으로 만들지 말고, 아래 기준으로 선택적으로 작성한다.

### 기본 CRUD 화면

목록, 생성, 편집처럼 기본 CRUD 성격이 강한 화면은 아래 중 하나를 선택한다.

- `화면명-요약.md` 1개만 둔다.
- `기획.md`와 `정책.md`로 충분하면 화면 정책 문서를 생략한다.

### 복잡한 화면

상태 전이, 리비전, 승인, 연결 정보, 권한 분기, 예외 처리가 겹치는 화면만
영역별 상세 정책을 둔다.

예시:

- `상세-액션.md`
- `상세-리비전.md`
- `상세-연결정보.md`
- `편집-현재상태.md`
- `편집-잠금상태.md`
- `편집-저장상태.md`

간결하게 작성하되, 다음을 포함한다:

- 해당 영역의 동작 규칙
- 해당 블록의 역할
- 메인 와이어프레임에 최소 무엇을 보여줘야 하는지
- 필드별 유효성 검증
- 예외 케이스
- UI 상태 변화
- 로딩 / 빈 상태 / 오류 상태
- 권한별 노출/비노출, 읽기 전용 조건

반드시 지킬 원칙은 아래와 같다.

- 공통 규칙은 `정책.md`를 반복하지 않는다.
- 화면 문서는 차이점과 UI 주의사항만 쓴다.
- 기본 CRUD 화면은 문서 개수를 최소화한다.
- 메인 와이어프레임의 `WireBlock` 이름과 정책 문서 파일 주제가 대응되게 만든다.
- 메인 와이어프레임에는 구조만 남기고, 블록별 설명은 상세 정책 문서로 뺀다.
- 잠금, 저장, 상태 피드백처럼 한 블록만 따로 설명이 필요한 경우
  `화면명-블록명.md`를 별도 문서로 만든다.
- `Loading`, `Locked`, `Error` 같은 상태는 문서에서 설명하고, 와이어프레임은
  실제 배너/disabled/에러 메시지 위치가 어떻게 보이는지 시각적으로 보여준다.

## 플로우 (flows/\*.tsx) 작성법

플로우는 복잡한 사용자/시스템 분기를 시각화하는 TSX 컴포넌트다.
`.wireframe` 앱의 `플로우` 탭에서 렌더링된다.

다음을 우선 작성한다:

- 신규 생성 흐름
- 상태 전이 흐름
- 승인/리비전/잠금처럼 분기가 많은 흐름

플로우는 화면 픽셀 설계가 아니라 과정 설계에 집중한다. 각 단계에는 다음이 보여야 한다:

- 누가 시작하는지
- 어떤 조건에서 분기되는지
- 시스템이 무엇을 생성/검증하는지
- 실패 시 어디서 막히는지
- MVP에서 지원하는 흐름인지 후속 범위인지

플로우는 아래 기준으로 상세하게 유지한다.

- `기획.md`의 핵심 시나리오와 1:1로 대응시킨다.
- 등록, 승인, 상태 전이, 리비전 상승, 잠금 충돌처럼 업무 기준 이름을 쓴다.
- 버튼 배치보다 업무 분기와 실패 조건을 우선 설명한다.
- FE/BE handoff에서 바로 재사용할 수 있을 정도로 단계와 예외를 구체화한다.

## 작업 순서

1. 사용자 요청을 분석하여 모듈과 기능명을 결정
2. 기존 폴더 구조를 확인하여 번호 부여
3. `기획.md` 작성
4. `정책.md` 작성
5. `flows/` 하위에 핵심 업무 흐름별 `.tsx` 플로우 작성
6. `screens/` 하위에 화면별 `.tsx` 와이어프레임 작성
7. 화면별 정책 문서가 필요한지 먼저 판단
8. 기본 CRUD 화면은 `화면명-요약.md` 1개 또는 무문서로 처리
9. 복잡한 화면만 영역별 상세 정책 `.md` 작성
