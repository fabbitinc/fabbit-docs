# 구현 지시 프롬프트 생성

기획 문서(기획.md, 정책.md, screens/*.tsx, screens/*-*.md)를 읽고,
개발 에이전트(Claude Code / Codex)가 바로 실행할 수 있는 구현 지시 프롬프트를 .md 파일로 생성한다.

## 사용법

```
/implement plm/04_부품관리 front     # 프론트엔드 지시만
/implement plm/04_부품관리 back      # 백엔드 지시만
/implement plm/04_부품관리           # 프론트 + 백 모두
```

- `$ARGUMENTS[0]` = 기능 경로 (예: `plm/04_부품관리`)
- `$ARGUMENTS[1]` = 포지션 (`front` | `back` | 생략 시 둘 다)

## 작업 순서

### 1. 기획 문서 읽기

`10_제품/기획/$0/` 하위에서 다음 파일을 모두 읽는다:

- `기획.md` — 기능 개요, 사용자 스토리, 화면 구성
- `정책.md` — 데이터 모델, 상태 관리, 권한, 유효성 검증, 비즈니스 규칙
- `screens/*.tsx` — 와이어프레임 (화면 구조, 컴포넌트, 필드 목록)
- `screens/*-*.md` — 화면별 상세 정책

모든 파일을 빠짐없이 읽어야 한다. 파일이 없으면 사용자에게 알린다.

### 2. 포지션 결정

- `$1`이 `front` → 프론트엔드 프롬프트만 생성
- `$1`이 `back` → 백엔드 프롬프트만 생성
- `$1`이 비어있음 → 프론트엔드 + 백엔드 둘 다 생성

### 3. 프롬프트 생성

포지션별 템플릿을 참조하여 프롬프트를 작성한다:

- 프론트엔드: [templates/frontend.md](templates/frontend.md)
- 백엔드: [templates/backend.md](templates/backend.md)

**핵심 규칙:**
- 기획 문서의 내용을 **구체적으로** 반영한다 (추상적 지시 금지)
- 화면명, 필드명, 상태값, API 경로 등을 기획에서 그대로 가져온다
- 에이전트가 기획 원본을 다시 읽을 필요 없이 프롬프트만으로 작업 가능해야 한다
- 특정 도구(Claude Code / Codex) 종속 문법 없이 범용 마크다운으로 작성
- 리포 경로를 하드코딩하지 않는다 (프론트/백 별도 리포)

### 4. 파일 저장

프로젝트 루트에 저장한다:

- 프론트: `{기능명}-frontend-impl.md`
- 백엔드: `{기능명}-backend-impl.md`

기능명은 경로에서 번호 prefix를 제거한 이름이다.
예: `plm/04_부품관리` → `부품관리-frontend-impl.md`

### 5. 완료 보고

생성된 파일 목록과 각 프롬프트의 핵심 내용을 요약하여 사용자에게 보고한다.
