# Phase 1: MVP - The Auto-BOM Generator

> **목표**: "엑셀보다 편한 도면 관리자" - 도면 PDF를 던져주면, 토끼가 알아서 자재 내역(BOM)을 물어오는 것

---

## 1. 핵심 사용자 시나리오 (User Story)

### 상황
성수동의 한 임가공 업체. 김 부장은 거래처에서 받은 도면 파일 50개가 담긴 압축 파일을 받았다.

| Before | After (Fabbit) |
|--------|----------------|
| 엑셀을 켜고 도면을 하나씩 열어가며 "품번, 품명, 재질, 수량"을 타이핑한다. **(2시간 소요, 오타 발생)** | 압축 파일을 Fabbit에 드래그한다. **3분 뒤**, Fabbit(토끼)이 "50개 도면 분석 완료! 2개는 글자가 흐려서 잘 안 보여요, 확인해주세요"라고 알림을 보낸다. 확인 버튼만 누르면 엑셀 BOM이 완성된다. |

## 2. 기능 상세 명세 (Functional Requirements)

### 기능 A: 스마트 업로드 (The Hopper)

| 항목 | 설명 |
|------|------|
| **Drag & Drop** | PDF, DWG, DXF, PNG 포맷 지원 |
| **자동 중복 체크** | 파일명 동일 → 새 버전(Rev)으로 자동 인식, Version +1 |
| | 파일명 다르나 내용(Hash) 동일 → "중복 파일입니다" 경고 |

### 기능 B: AI 도면 파싱 (The Rabbit Ears) ⭐ 핵심 기술

> 기술적 난이도가 가장 높습니다. MVP에서는 **GPT-4o Vision API**를 활용하여 개발 속도를 높이는 것을 추천합니다.

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  PDF 업로드  │ ──▶ │ 이미지 변환  │ ──▶ │  AI 분석    │ ──▶ │  BOM 생성   │
│             │     │ + 전처리    │     │             │     │             │
└─────────────┘     └─────────────┘     └─────────────┘     └─────────────┘
```

| 단계 | 처리 내용 |
|------|-----------|
| **이미지 전처리** | PDF → 고해상도 이미지 변환 + 노이즈 제거 (OpenCV) |
| **표제란(Title Block) 탐지** | 도면 우측 하단(일반적 위치) 스캔 → "품번(Part No)", "품명(Name)", "재질(Material)" 키워드 주변 텍스트 추출 |
| **부품표(Parts List) 탐지** | 조립도(Assembly)의 테이블(표) 형태 인식 → Child Part 리스트 추출 |

### 기능 C: 스플릿 뷰 검증 (Verification UI)

> AI가 100% 완벽할 순 없습니다. **사용자가 신뢰할 수 있는 UX가 필수**입니다.

```
┌────────────────────────────────────────────────────────┐
│                    Fabbit Viewer                       │
├─────────────────────────┬──────────────────────────────┤
│                         │  📋 추출 데이터              │
│      📄 도면 뷰어        │                              │
│                         │  품번: ABC-001               │
│    [하이라이트 표시]     │  품명: 브라켓 ◀── 클릭 시    │
│         ⬆               │  재질: SUS304     하이라이트  │
│         │               │  수량: 2                     │
│    AI가 읽은 위치        │                              │
│                         │  [확인] [수정]               │
└─────────────────────────┴──────────────────────────────┘
```

| 기능 | 설명 |
|------|------|
| **화면 구성** | 좌측 [도면 뷰어] vs 우측 [추출 데이터 폼] |
| **Highlighting** | 우측 필드 클릭 → 좌측 도면에서 AI가 읽은 위치(Box)에 형광펜 표시 |
| **검증 효과** | 사용자가 "아, 이걸 읽었구나" 하고 검증 가능 |

## 3. 데이터베이스 설계 (DB Schema)

User - Project 구조 하에 핵심 3개 테이블:

### 3.1 Items (자재/품목)

> 물리적인 부품 그 자체

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | UUID | PK |
| `part_number` | VARCHAR | 품번 (식별자) |
| `name` | VARCHAR | 품명 |
| `spec` | VARCHAR | 규격/재질 |
| `category` | ENUM | 완제품 / 반제품 / 원자재 |

### 3.2 Documents (도면 파일)

> 업로드된 파일 정보

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `id` | UUID | PK |
| `item_id` | UUID | FK → Items (어떤 품목의 도면인지) |
| `file_path` | VARCHAR | S3/R2 저장 경로 |
| `version` | INTEGER | 1, 2, 3... (리비전 번호) |
| `status` | ENUM | Draft(초안) / Released(승인됨) / Obsolete(폐기됨) |
| `processed_data` | JSONB | AI가 추출한 원본 데이터 |

### 3.3 BOM_Structure (제품 구조)

> 조립 관계(Parent-Child) 정의

| 컬럼 | 타입 | 설명 |
|------|------|------|
| `parent_item_id` | UUID | FK → Items |
| `child_item_id` | UUID | FK → Items |
| `quantity` | DECIMAL | 소요량 |

### ERD

```
┌──────────────┐       ┌──────────────┐       ┌──────────────┐
│    Items     │       │  Documents   │       │ BOM_Structure│
├──────────────┤       ├──────────────┤       ├──────────────┤
│ id (PK)      │◀──────│ item_id (FK) │       │ parent_id(FK)│──┐
│ part_number  │       │ id (PK)      │       │ child_id(FK) │──┤
│ name         │       │ file_path    │       │ quantity     │  │
│ spec         │       │ version      │       └──────────────┘  │
│ category     │       │ status       │              │          │
└──────────────┘       │ processed_   │              └──────────┘
       ▲               │   data       │                  │
       │               └──────────────┘                  │
       └─────────────────────────────────────────────────┘
```

## 4. 기술 스택

| 구분 | 기술 | 설명 |
|------|------|------|
| **Backend** | Python (FastAPI) | 비동기 처리 우수, AI 라이브러리 최적 |
| **AI Engine** | LangChain + GPT-4o | 초기에 OCR 모델 직접 개발 X, API 활용 (비용 < 개발 속도) |
| **PDF Processing** | PyMuPDF (fitz) | PDF → 이미지 변환, 텍스트 레이어 추출 최고 성능 |
| **Image Processing** | OpenCV | 노이즈 제거, 이미지 전처리 |
| **Database** | PostgreSQL (Supabase) | 관계형 DB + 백엔드 API 자동화, MVP에 적합 |
| **Storage** | Cloudflare R2 | S3 호환 + Egress 비용 0원, 도면 뷰어에 최적 |
