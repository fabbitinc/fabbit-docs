# HANDOFF - 예비창업패키지 사업계획서 작성 운영 기준

이 문서는 `예비창업패키지` 사업계획서를 HWP 양식에 맞춰 작성하거나
다듬을 때 사용하는 작업 기준선이다. 사용자가 양식 캡처를 보내면, 이 문서의
우선순위와 금지 규칙에 따라 해당 칸에 바로 붙여 넣을 수 있는 문안을 만든다.

이 문서는 지원사업 작성 작업을 위한 운영용 handoff다. 현재 제품 정책, 플랜
정책, 사업 전략의 SSOT 본문이 아니라 사업계획서 편집 작업에서 참고 우선순위와
금지 규칙을 정하는 작업 문서로 사용한다.

## 문서 등급과 사용 범위

이 문서는 지원사업 제출 문안을 다듬기 위한 handoff이며, 상위 기준 문서를
대체하지 않는다. 제품과 사업의 현재 기준을 판단할 때는 정책 SSOT와 전략
문서를 먼저 보고, 이 문서는 그 기준을 사업계획서 문체로 번역하는 용도로 쓴다.

- 제품 세그먼트와 도입 기준은 `10_제품/정책/*`를 먼저 따른다.
- 플랜과 과금 기준은 `40_사업/정책/*`를 먼저 따른다.
- 이 문서는 지원사업 작성 맥락에서만 문장 우선순위와 표현 금지 규칙을 정한다.

## 목표

핵심 목표는 `40_사업/지원사업/예비창업패키지/사업계획서-별첨1-경기기반-초안.md`
를 기준 원본으로 삼아, 심사자가 읽었을 때 `범위가 과장되지 않고`,
`근거가 보이고`, `협약기간 내 실행 가능해 보이는` 사업계획서 문구로
정리하는 것이다.

이 문서는 원본 초안을 대체하지 않는다. 원본 초안은 사실과 서사의 저장소고,
이 핸드오프는 캡처 단위 작성 작업에서 무엇을 쓰고 무엇을 빼야 하는지 정하는
운영 문서다.

## 기준 원본 우선순위

작업 시 참고 우선순위는 아래와 같다. 서로 충돌하면 위 문서를 따른다.

1. `40_사업/지원사업/예비창업패키지/사업계획서-별첨1-경기기반-초안.md`
2. `40_사업/지원사업/예비창업패키지/피드백.md`
3. `40_사업/지원사업/예비창업패키지/사업계획서-별첨1-정렬본.md`
4. `40_사업/지원사업/예비창업패키지/사업계획서.md`
5. 인터뷰/검증 리포트, 정책 SSOT, 제품/사업 전략 문서

`사업계획서.md`는 장기 비전과 배경 설명은 풍부하지만, 과한 범위와 오래된
주장이 섞여 있다. 캡처 기반 작성에서는 직접 복붙하지 말고, 최신 초안과
피드백으로 다시 걸러서 사용한다.

## 현재 기준선

현재 문서 기준선은 2026년 3월 18일 상태를 따른다. 이번 제출본에서는
`쉽게 도입 가능한 도면·BOM 관리 SaaS`, `AI 기반 BOM 필드 매핑`,
`구조화된 관리`, `파일럿 검증`, `Starter -> Team 전환`이 핵심이다.

반대로 `온톨로지`, `Agentic AI`, `팔란티어식 비전`은 중장기 확장 설명에만
제한적으로 남긴다. 협약기간 내 달성 목표처럼 쓰면 안 된다.

## 핵심 메시지

이 사업계획서에서 반복해서 지켜야 할 메시지는 아래 다섯 가지다.

- 중소 제조기업의 실제 문제는 `도면·BOM을 저장은 하지만 관리하지 못하는 상태`
  에 있다.
- 고객의 진짜 경쟁상대는 기존 PLM이 아니라 `엑셀, 공유폴더, NAS 그대로 버티는
  방식`이다.
- Fabbit의 강점은 `기존 양식을 버리지 않고 시스템으로 넘어오게 만드는 진입
  경험`이다.
- 협약기간의 승부처는 `기능 확장`보다 `정확도 개선, 파일럿 운영, 유료 전환
  검증`이다.
- 심사자는 큰 비전보다 `8개월 안에 실제로 끝낼 수 있는가`를 더 본다.

## 작성 모드

이 문서는 과거처럼 `아직 안 쓴 칸을 처음부터 채우는 모드`가 아니다. 현재는
이미 작성된 초안을 HWP 양식 칸 크기와 심사 톤에 맞게 줄이고 다듬는
`편집 모드`가 기본이다.

사용자가 캡처를 보내면 다음 순서로 작업한다.

1. 캡처의 섹션명과 칸 구조를 확인한다.
2. 최신 원본 초안에서 같은 섹션의 핵심 문장을 추린다.
3. `피드백.md` 기준으로 과장, 비약, 근거 부족 표현을 제거한다.
4. HWP에 바로 붙여 넣을 수 있는 개조식 문안으로 다시 쓴다.
5. 같은 페이지의 다른 항목과 용어, 수치, 범위가 충돌하지 않는지 점검한다.

## 출력 형식

캡처 기반 응답은 아래 형식을 기본으로 한다.

- `○ 헤드라인`은 메시지 문장으로 쓴다.
- 하위 항목은 개조식 3~5개 이내로 제한한다.
- 한 줄이 너무 길면 두 문장으로 자르지 말고 짧은 명사형 문장으로 다시 쓴다.
- HWP에 바로 붙여 넣기 쉽게 불필요한 설명 문단은 제외한다.
- 필요하면 `이미지 캡션 문구`도 별도로 제안한다.

기본 톤은 아래와 같다.

- 명사형 종결
- 고객 언어 우선
- 근거 있는 주장만 사용
- 심사자 관점에서 방어 가능한 문장

## 반드시 지킬 규칙

모든 섹션에서 아래 규칙을 지킨다.

- 문제와 해결은 1:1로 연결한다.
- 협약기간 내 목표와 중장기 비전을 섞지 않는다.
- 이미 구현된 것, 협약기간 내 구현할 것, 이후 확장할 것을 분리한다.
- 수치가 들어가면 가능하면 출처 또는 산식이 바로 설명되게 쓴다.
- 고객 검증은 활동량보다 `누가 어떤 pain을 말했고 어떤 의향을 보였는지`가
  드러나게 쓴다.
- `PLM`, `MES`, `온톨로지`보다 `도면관리`, `변경 반영`, `최신본 관리`,
  `도입 장벽` 같은 현장 언어를 우선한다.

## 쓰지 말아야 할 것

다음 표현이나 주장은 기본적으로 금지한다.

- `95.6% PLM 미사용`
- 출처가 불명확한 `업무시간 25~30% 검색 낭비`
- 출처가 불명확한 `BOM 오류가 ECO의 30%`
- 출처가 불명확한 `엔지니어 연 52~260시간 BOM 작업`
- 협약기간 내 완료처럼 읽히는 `온톨로지`, `Agentic AI`, `Systems of Action`
- 고객이 잘 모르는 전문 용어를 전면에 내세운 제목
- `다 할 수 있다` 식의 과장된 로드맵

위 내용이 필요하면 `중장기 방향` 또는 `후속 확장` 수준으로만 약하게
정리한다.

## 사용 가능한 근거

반복 사용해도 되는 근거는 아래 범주 안에서만 쓴다.

- 국내 제조업 사업체 수, 스마트공장 관련 통계
- 콜드콜 36건+, 후속 인터뷰 2건, POC 의향 확인
- 저장 위치는 바뀌어도 최신본 식별, 리비전 추적, 변경 승인은 여전히 수작업
- 대표의 PLM 도메인 경험과 풀스택 구현 이력
- 현재 MVP 구현 범위
- 협약기간 내 목표인 정확도 개선, 파일럿 운영, 유료 전환 검증

정량 근거를 쓸 때는 아래처럼 해석을 보수적으로 유지한다.

- 큰 시장 숫자는 `시장 크기`를 보여주는 용도로만 사용
- 파일럿, 유료 전환, 매출 전망은 `보수적 가정`임을 드러낼 것
- 고객 검증은 `36건 연락`보다 `반복 pain과 POC 의향` 중심으로 쓸 것

## 충돌 관리

현재 관련 문서에는 몇 가지 충돌이 있다. 작업 중 임의로 섞지 말고 아래
기준선을 따른다.

### 가격 체계 충돌

`수익모델-정의.md`와 `사업계획서-별첨1-정렬본.md`에는 회사 단위 정액형
가격이 남아 있고, `사업계획서-별첨1-경기기반-초안.md`에는 역할별 좌석 기반
Starter/Team/Organization/Enterprise 구조가 들어 있다.

캡처 기반 작성에서는 더 최신인 `사업계획서-별첨1-경기기반-초안.md`를
우선한다. 다만 심사 양식 칸이 좁으면 아래 수준으로 축약한다.

- `무료 Starter -> Team 유료 전환`
- `초기 매출 추정은 Team 플랜 기준 보수 산정`
- `Organization/Enterprise는 중장기 확장 플랜`

### 범위 충돌

오래된 문서에는 `경량 MES`, `온톨로지`, `Agentic AI`, `해외 진출`,
`ERP·PLC 연동`이 한 번에 묶여 있다. 최신 기준선에서는 아래처럼 분리한다.

- 협약기간: AI 매핑 정확도 개선, 구조화 관리 안정화, 파일럿, 유료 전환 검증
- 중기 확장: 운영 데이터 AI 고도화, 동아시아 확장 검토
- 장기 확장: ERP·MES·PLC 연동, 고도화 AI 분석

### 팀 표현 충돌

자문 네트워크는 `확보 완료`처럼 단정하지 말고, 최신 초안 기준으로
`부족 영역을 자문 네트워크로 보완` 정도로 쓴다. 실명, 확정 표현, 과한
약속은 피한다.

## 섹션별 작성 기준

각 섹션은 아래 기준으로 압축한다.

### 1. 문제 인식

문제 인식은 `저장은 되지만 관리가 안 되는 상태`, `설계-생산 단절`,
`도입 장벽이 핵심`이라는 세 축으로 정리한다. 현장 pain은 3~4개만 남기고,
각 pain이 왜 비용과 혼선으로 이어지는지 짧게 붙인다.

### 2. 실현 가능성

실현 가능성은 `이미 구현한 것`, `협약기간 내 할 것`, `왜 가능한가`를
구분해서 쓴다. 이 섹션에서 가장 중요한 것은 비전이 아니라 `범위 통제`다.
협약기간 목표는 정확도, 사용성, 파일럿, 전환 검증 중심으로 묶는다.

### 3. 성장전략

성장전략은 `10~49명 제조기업`, `무료 진입`, `파일럿`, `Team 전환`,
`직접 영업 -> 사례 확산 -> 후속 채널` 흐름을 유지한다. TAM/SAM/SOM이나
매출은 숫자 자체보다 `산정 방식이 보이는가`를 우선한다.

### 4. 팀 구성

팀 구성은 대표의 도메인 적합성과 실행력, 팀원의 영업·고객 접점 역량,
부족 영역을 보완하는 자문 네트워크 순으로 정리한다. `1인 창업의 약점`이
보이지 않게 감추기보다, `작은 팀이지만 이미 구현했고 검증을 시작했다`는
식으로 방어한다.

## 현재 상태 요약

현재 기준 문서상, 예비창업패키지 별첨 1 초안은 전 섹션 골격이 이미
작성되어 있다. 따라서 이후 작업은 아래 성격이 된다.

- HWP 칸 크기에 맞춘 문장 압축
- 오래된 주장 제거
- 범위 축소 및 우선순위 재정렬
- 수치와 용어 통일
- 이미지 캡션과 표 문구 보정

즉, 다음 캡처 작업부터는 `새로 쓰기`보다 `정리하고 줄이기`가 기본이다.

## 캡처가 오면 바로 확인할 체크리스트

응답을 내기 전 아래 항목을 확인한다.

- 이 문장이 협약기간 내 달성 목표로 읽히는가
- 고객 언어로 바뀌었는가
- 숫자와 근거가 방어 가능한가
- 같은 페이지 다른 항목과 수치가 충돌하지 않는가
- 목록 수가 너무 많지 않은가
- 심사자가 `왜 지금, 왜 이 팀, 왜 이 범위`를 바로 이해할 수 있는가

## 다음 작업 입력 형식

사용자가 다음 중 하나를 보내면 바로 작업한다.

- HWP 양식 캡처
- 특정 섹션명
- 기존 문안과 수정 요청

출력은 아래 중 필요한 형태로 제공한다.

- HWP 붙여 넣기용 최종 문안
- 더 짧은 버전
- 표 안에 넣는 버전
- 이미지 캡션 버전
- 심사 리스크를 줄인 대체 문안
