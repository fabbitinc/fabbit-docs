# Fabbit 문서 인덱스

이 문서는 Fabbit `docs`의 루트 진입 문서다. 현재 구조에서 가장 중요한 목표는
`정책 SSOT`, `전략 문서`, `기획 문서`, `구현 현황`을 빠르게 구분해 찾을 수
있게 만드는 것이다.

문서를 읽을 때는 먼저 "무엇이 현재 기준인가"를 확인한 뒤, 그 기준을 설명하거나
적용하는 파생 문서로 내려가는 순서를 따른다.

루트에 있는 `HANDOFF.md`, `HANDOFF-사업계획서.md`, `TODO.md`는 현재 기준을
고정하는 본문이 아니다. 이 문서들은 작업 전달, 작성 운영, 임시 추적을 위한
운영 문서로 읽어야 한다.

## 1. 먼저 읽을 문서

현재 구조에서 가장 먼저 찾아야 할 문서는 아래 다섯 갈래다. 이 표는 루트에서
현재 SSOT와 핵심 진입점을 빠르게 찾기 위한 안내다.

| 목적 | 먼저 읽을 문서 | 의미 |
|------|----------------|------|
| 고객 타깃과 도입 기준 확인 | [[10_제품/정책/index]] | 현재 타깃 고객, 우선 세그먼트, 파일럿 중심 도입 기준 |
| 플랜과 과금 기준 확인 | [[40_사업/정책/index]] | 플랜명, 좌석, 스토리지, AI, 업그레이드 정책 SSOT |
| 제품 방향과 범위 확인 | [[10_제품/전략/index]] | 제품 정의, 로드맵, 문제-해결 구조, 경쟁 관점 |
| 사업 모델과 GTM 가설 확인 | [[40_사업/전략/index]] | 수익화, 시장 진입, 사업 실행 가설 |
| 실제 화면과 흐름 확인 | [[10_제품/기획/index]] | `site`, `app`, `plm`, `mes` 기획 문서군 진입점 |

구현 기준을 확인하려면 위 문서 다음에
[[20_개발/코드베이스-현황]]을 읽는다.

## 2. 현재 SSOT 구조

현재 `docs`는 아래 레이어로 읽는 것이 가장 안전하다. 같은 주제를 여러 곳에서
다룰 때는 아래쪽 문서가 아니라, 더 위 레이어의 기준 문서를 먼저 따른다.

| 레이어 | 주 역할 | 대표 진입 문서 |
|--------|---------|----------------|
| 시장 근거 | 문제 검증, 시장 규모, 인터뷰 근거 | `30_시장/*` |
| 정책 SSOT | 현재 운영 기준과 상태 전이 규칙 | [[10_제품/정책/index]], [[40_사업/정책/index]] |
| 전략 | 왜 이 방향으로 가는지와 사업/제품 가설 | [[10_제품/전략/index]], [[40_사업/전략/index]] |
| 기획 | 화면, 흐름, 공개 경험, 앱 경험, 기능 상세 | [[10_제품/기획/index]] |
| 개발 현황 | 코드베이스 기준 구현 상태와 API | [[20_개발/코드베이스-현황]] |

핵심 원칙은 간단하다. 정책은 현재 기준을 고정하고, 전략은 그 기준을 바탕으로
해석하며, 기획은 그 기준을 사용자 경험과 화면 흐름으로 번역한다.

## 3. 제품 문서 진입점

제품 영역은 전략, 정책, 기획이 서로 가까워 보이지만 책임이 다르다. 아래 표를
기준으로 원하는 문서군으로 들어가면 된다.

| 보고 싶은 것 | 진입 문서 | 비고 |
|--------------|-----------|------|
| 제품 정의와 차별점 | [[10_제품/전략/index]] | `아이템-정의`, `제품-로드맵` 중심 |
| 타깃 고객과 도입 기준 | [[10_제품/정책/index]] | 고객 세그먼트 SSOT |
| 공개 사이트 | [[10_제품/기획/site/index]] | 랜딩, 요금제, 신청 전환 |
| 앱 온보딩과 구독 | [[10_제품/기획/app/index]] | 가입, 워크스페이스 생성, 플랜 선택 |
| MES 기획 | [[10_제품/기획/mes/index]] | 생산 실행 관리 문서군 |
| 제품 기획 상위 구조 | [[10_제품/기획/index]] | `site/app/plm/mes` 전체 연결 관계 |

현재 `plm` 문서군은 상세 문서는 있지만 별도 상위 `index.md`가 없다. 따라서
당장은 [[10_제품/기획/index]]에서 구조를 확인한 뒤 `plm` 하위 문서로
들어가는 방식이 맞다.

## 4. 사업 문서 진입점

사업 영역도 전략과 정책을 분리해서 읽어야 drift를 줄일 수 있다. 특히 가격,
플랜, 업그레이드 규칙은 전략 문서가 아니라 정책 문서를 먼저 본다.

| 보고 싶은 것 | 진입 문서 | 비고 |
|--------------|-----------|------|
| 플랜, 좌석, 스토리지, AI 기준 | [[40_사업/정책/index]] | 현재 사업 정책 SSOT |
| 수익화와 ARR 가설 | [[40_사업/전략/index]] | `수익모델-정의` 중심 |
| 지원사업 문서 | `40_사업/지원사업/README.md` | 사업계획서와 공고 대응 |
| 발표 자료 | `40_사업/발표/*` | 발표 스크립트와 질의응답 |

## 5. 문서 배치 원칙

루트 인덱스에서 가장 자주 혼동되는 지점은 `정책`, `전략`, `기획`의 위계다.
새 문서를 추가할 때는 아래 기준으로 위치를 정한다.

- 현재 운영 규칙, 상태 전이, 플랜 기준을 고정하면 `정책`에 둔다.
- 시장 진입 논리, 수익 가설, 포지셔닝, 우선순위를 해석하면 `전략`에 둔다.
- 화면, 사용자 흐름, 정보 구조, 구현 전달 산출물이면 `기획`에 둔다.
- 서버나 웹의 실제 구현 상태, API, 라우트, 모듈 설명이면 `20_개발`에 둔다.
- handoff, todo, 임시 메모, 작업 기준선이면 루트 운영 문서로 두되 SSOT처럼
  쓰지 않는다.

정책 문서는 가능한 한 하나의 원문만 유지하고, 전략과 기획 문서는 그 정책을
복제하지 않고 링크로 참조하는 편이 맞다.

## 6. 루트 운영 문서

루트에는 자주 쓰이지만 등급이 다른 운영 문서도 있다. 아래 문서들은 작업 효율을
위한 보조 문서이며, 정책이나 전략 본문과 같은 등급으로 읽지 않는다.

| 문서 | 등급 | 사용 목적 |
|------|------|-----------|
| [[HANDOFF]] | 운영용 handoff | `.wireframe` 앱과 기획 작성 체계 관련 전달 메모 |
| [[HANDOFF-사업계획서]] | 운영용 handoff | 지원사업 사업계획서 작성 작업 기준 |
| [[TODO]] | 작업 추적 메모 | 자료수집, 검증, 작성 backlog 관리 |

이 문서들 안에 과거 기준선이나 임시 문구가 남아 있어도, 현재 기준 판단은 정책
SSOT와 상위 전략 문서에서 먼저 내려야 한다.
