### 온톨로지의 역할

- 현실 객체(비행기, 부품, 공장)와 비즈니스 논리를 노드·엣지로 모델링 → 디지털 트윈
- 서로 다른 시스템(ERP, CRM 등)의 데이터를 의미론적으로 연결 → 전사 SSOT
- What-if 분석: 부품 공급 지연 시 전체 공정 영향을 수초 내 파악, 150개+ 옵션 중 최적해 제시
- Systems of Action: 분석 → SAP 워크오더 자동 생성 등 실제 비즈니스 행동으로 연결

- 생산 일정·부품 인도·작업 지시서·품질 결함 데이터가 국가/팀별로 분산

## 3. 지멘스 Industrial Knowledge Graph

### Use Cases

| 적용 영역 | 효과 |
|---------|------|
| 가스 터빈 유지보수 | as-maintained BOM 생성 **비용 80% 절감** |
| 판매 견적·사양 검토 | 속도 **최대 90% 향상**, 영업 성공률 개선 |

| 유연 생산 | 다품종 소량 생산(Lot-size-one) 경제적 실현 |

### 온톨로지 설계

- 데이터 분류: 1D(구조화 IT), 2D(도면/문서), 3D(CAD) 통합
- 재사용 가능 온톨로지 라이브러리 (BOM, 건물 등 도메인별 표준 블록)
- 3대 기둥: **재료(Material) × 프로세스(Process) × 장비(Equipment)**
- 4단계 성숙도: 단일 사일로 → 사일로 간 통합 → 전사 지식 공장

### 기술 스택

- metaphactory 플랫폼 (의미론적 검색·시각화·온톨로지 관리)
- Amazon Neptune (그래프 DB)
- 기존 PLM(Teamcenter), MES와 연결하는 가교 역할

> 출처: ISWC 2018 Paper, metaphactory Case Study, CERN Workshop

## 6. BOM 관리에서 그래프 DB의 장점

| 비교 축 | 관계형 DB (SQL) | 그래프 DB |
|---------|----------------|----------|
| BOM 계층 처리 | 다수 JOIN 필요, 깊이↑ → 성능↓ | 노드-엣지 직접 연결, 깊이 무관 |
| 재귀적 쿼리 (where-used) | 복잡한 하위 쿼리 필요 | 네이티브 지원, 즉시 결과 |
| 스키마 변경 | 마이그레이션 필요 | 기존 구조 파괴 없이 확장 |
| 관계 추가 (부품-공급사-담당자) | 테이블 추가 + JOIN 증가 | 엣지 추가만으로 완료 |

### 부품-공급사-도면 연결 시 가능해지는 것

- **근본 원인 분석**: 결함 발생 → 부품·공급사·공정 역추적을 **수초 내 완료** (기존 수주)
- **공급망 가시성**: 공급사 중단 시 영향받는 완제품·고객 주문 즉시 파악
- **도면 데이터 통합**: 2D 도면(P&ID) 기호 추출 → ERP/PLM 구조화 데이터와 결합 → as-maintained BOM 자동 생성

## 7. PLM → 데이터 플랫폼 진화 트렌드

BeyondPLM 블로그, OpenBOM 자료 기반.

### 기존 PLM의 한계

- 고정 스키마 → 확장 시 마이그레이션 필수
- 문서(CAD 파일) 중심 → 부품 간 세밀한 관계/영향 분석 불가
- 데이터 사일로 → 전체 수명 주기 통합 시야 부재

### 진화 방향

1. **애플리케이션 → 데이터 중심**: 기능보다 데이터 연결성이 핵심 자산
2. **모놀리식 → 포스트-모놀리식**: 의미론적 모델링 레이어 위에 서비스가 플러그 앤 플레이
3. **디지털 스레드 실현**: 온톨로지 기반으로 설계→제조→서비스 전 데이터 끊김 없이 연결
4. **AI 융합**: 지식그래프 + LLM → 문서 자동 생성, 설계 최적화, AI 에이전트 의사결정

> [!info] Fabbit과의 연결
> Fabbit의 3단계 비전(온톨로지 기반 제조 데이터 플랫폼)은 이 PLM 진화 트렌드의 최전선에 있다.
> 차이점: 팔란티어/지멘스는 대기업 대상 → Fabbit은 중소 제조업 대상.
> AI 추출로 진입장벽을 없애고, 축적된 데이터를 온톨로지로 연결하는 접근은
> 중소기업에서 이 트렌드를 현실화하는 유일한 경로.

## 8. AI 활용 단계: Reactive → Agentic

온톨로지·지식그래프 위에서 AI가 어떻게 동작하는지는 3단계로 구분된다.
Fabbit의 차별점은 기존 KG+AI(Reactive)를 넘어 Agentic AI를 목표로 한다는 것이다.

### 3단계 구분

| 단계 | 방식 | 제조업 예시 | 비고 |
|------|------|-----------|------|
| **Reactive** | 사용자가 질문 → AI가 답변 | "이 부품 어디에 쓰여?" → where-used 결과 | 기존 KG+LLM 조합의 일반적 패턴 |
| **Proactive** | AI가 데이터 변화를 지속 모니터링 → 이상 시 선제 알림 | "부품 A의 단일 공급사가 납기 지연 3회 → 리스크 알림" | 팔란티어 Foundry AIP가 이 수준 |
| **Agentic** | 감지 → 대안 탐색 → 행동 초안까지 자동 | "리스크 감지 → 대체 공급사 3곳 추출 → 담당자에게 발주 초안 제시" | 팔란티어가 **Systems of Action**이라 부름 |

### 용어 출처와 업계 사용

- **Agentic AI**: 2024~2026 Gartner, McKinsey 등이 주요 트렌드로 선정. "AI가 자율적으로 판단·행동하는 패러다임"
- **Proactive AI / Proactive Intelligence**: 팔란티어, Databricks 등이 사전 감지·알림 맥락에서 사용
- **Continuous Intelligence**: Gartner (2019~). 실시간 데이터 스트림에서 지속적으로 인사이트를 추출
- **Systems of Action**: 팔란티어 고유 용어. 분석(Systems of Insight) → 행동(Systems of Action)으로의 전환을 강조. 분석 결과가 SAP 워크오더 생성, 발주, 알림 등 실제 비즈니스 행동으로 연결되는 것이 핵심

### Fabbit 포지셔닝

- **비전**: Agentic AI (감지→대안→행동 자동화)
- **현실**: Reactive부터 구현하여 데이터 축적 → Proactive → Agentic으로 단계적 진화
- **차별화 논리**: 기존 KG+AI 서비스는 Reactive에 머무름. 비링커는 AI 추출에서 끝남. 기존 PLM은 AI 자체가 없음. Fabbit은 온톨로지 기반으로 Agentic까지 가는 것이 궁극 목표

> [!info] 사업계획서에서의 사용
> - 차별점 용어: **Agentic AI** (비전 레벨)
> - 팔란티어 레퍼런스: **Systems of Action**
> - Reactive는 "기존 방식"으로 대비할 때만 언급
> - Proactive는 내부 단계 구분용. 외부 문서에서는 Agentic으로 통일
