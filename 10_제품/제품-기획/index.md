# 제품 기획

이 문서는 Fabbit 제품 기획 문서의 최상위 기준 문서다. 기존 `plm/` 폴더는
도메인 지식, 경쟁 비교, 로드맵, 참고 자료를 관리하고, `제품-기획/` 폴더는
실제 구현할 기능을 화면과 사용자 흐름 중심으로 설계하는 문서를 관리한다.

## 1. 문서 목적

제품 기획 문서는 "무엇을 왜 만들고, 어떤 화면과 동작으로 제공할지"를 정리하는
실행 문서다. 기능 정의, 사용자 흐름, 정책, 데이터 연결, MVP 범위를 이 문서군에
모은다.

- `plm/`: 도메인 이해, 제품 전략, 경쟁 환경, 로드맵 근거
- `제품-기획/`: 실제 화면과 기능을 설계하는 상세 기획
- `제품-기획/mes/`: MES 도메인 상세 기획

## 2. 제품 구조

Fabbit의 제품 구조는 아래 4개 층으로 본다.

| 층위 | 역할 | 현재 문서 기준 |
|------|------|----------------|
| **진입 레이어** | 도면·BOM AI 추출로 시스템 유입 | `아이템-정의`, `plm/제품-로드맵` |
| **설계 관리 레이어** | PLM 코어, BOM, 변경관리, 부품/공급사 | `plm/*` |
| **실행 관리 레이어** | 작업지시, 생산실적, 품질, 추적 | `제품-기획/mes/*` |
| **지능화 레이어** | 온톨로지, 자연어 조회, Agentic AI | `plm/온톨로지-리서치`, 후속 기획 예정 |

## 3. 우선순위 원칙

제품 기획은 아래 순서로 확장한다.

1. **데이터 기준본 확보**
   도면, BOM, 리비전, 부품 체계가 정리되지 않으면 이후 실행 데이터도 신뢰할 수
   없다.
2. **운영 가시성 확보**
   MES MVP는 "오늘 무엇을 얼마나 만들었는지"를 당일 안에 파악하는 수준을
   목표로 한다.
3. **피드백 루프 형성**
   불량, 품질, 작업 이력을 설계 데이터와 연결해야 통합 제품 가치가 생긴다.
4. **외부 연동과 자동화**
   ERP, 설비, AI 자동화는 앞단 데이터 축적 이후 확장한다.

## 5. 기능 문서 작성 기준

각 기능 문서는 아래 항목을 기본으로 유지한다.

1. **문제와 목표**
   사용자가 왜 이 기능을 필요로 하는지 적는다.
2. **사용자와 시나리오**
   누가 어떤 흐름으로 쓰는지 적는다.
3. **화면 기획**
   목록, 상세, 생성/수정, 상태 표시, 액션 버튼 구조를 적는다.
4. **기능 기획**
   필수 동작, 권한, 상태 전이, 검색/필터, 알림 등을 적는다.
5. **정책과 예외**
   수정 가능 시점, 삭제 기준, 충돌 처리, 잠금 규칙 등을 적는다.
6. **데이터 연결**
   어떤 엔티티와 연결되는지 적는다.
7. **MVP 범위와 후속 확장**
   지금 넣을 것과 나중에 넣을 것을 분리한다.

## 6. 현재 집중 영역

현재 제품 기획의 1차 집중 영역은 MES다. 이유는 아래와 같다.

- PLM 단독보다 현장 체감 가치가 빠르다.
- 작업지시, 생산실적, 불량기록만 있어도 엑셀 생산관리에서 벗어날 수 있다.
- 설계 데이터와 생산 데이터를 연결해야 Fabbit의 통합 포지셔닝이 성립한다.
