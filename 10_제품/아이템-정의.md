# 아이템 정의

4개 사업계획서가 동일한 아이템을 서술하기 위한 기준 문서.
사업별 양식에 맞춰 표현을 바꾸되, 이 문서의 정의를 벗어나지 않는다.

## 1. 한 문장 정의

**Fabbit은 중소 제조업의 설계·생산 데이터를 AI로 통합하는 제조 데이터 플랫폼이다.**

### 차별점

1. **AI 추출로 진입장벽 제거**: 도면·BOM을 AI가 자동 데이터화 → 엑셀에서 시스템으로 전환하는 첫 관문을 없앤다
2. **설계+생산 통합**: 도면·BOM 관리(PLM)와 작업지시·생산실적·품질추적(경량 MES)을 하나의 플랫폼에서 제공 → 소기업이 여러 시스템 없이 설계부터 생산까지 관리
3. **Agentic AI — 데이터에서 AI가 먼저 발견하고 행동한다**: 설계 데이터와 생산 데이터를 온톨로지로 연결하고, AI가 변경 영향·리스크·품질 이상을 사전 감지하여 대안 탐색·행동 초안까지 자동화한다

## 2. Problem

중소 제조업(10~299명)의 95.6%는 제조 데이터 관리 시스템 없이 엑셀·폴더·메일로 도면, BOM, 생산 현황을 관리한다.

### 설계 데이터 문제

| 문제                 | 고객 언어                              | 정량 근거                                                                                        |
| ------------------ | ---------------------------------- | -------------------------------------------------------------------------------------------- |
| BOM 수동 작성 시간 낭비    | "도면 보면서 Excel에 하나하나 옮기는 데 30분~1시간" | 엔지니어가 엑셀 BOM 관리에 주당 1~5시간 소요, 연간 52~260시간/인 [^bom-time]                                      |
| 도면 검색·버전 혼선        | "업무시간의 30%를 도면 찾는 데 쓴다"            | 엔지니어 업무시간의 25~30% 정보 검색에 소요 [^search-aiim] [^search-mckinsey]. 일부 조사에서는 최대 40% [^search-ihs] |
| BOM 오류 → 재발주/생산 중단 | "수량 잘못 넣어서 자재 다시 발주"               | BOM 오류가 전체 ECO의 최대 30% 차지. ECO 1건당 행정 오버헤드 평균 $1,500~3,000 (하류 비용 미포함) [^eco-aberdeen]       |
| 설계변경 반영 지연         | "설계 바뀌었는데 생산팀이 모르고 구버전으로"          | ECO가 리드타임·개발비 핵심 드라이버. 기업 설문에서 "월 1회 이상" ECO로 추가 비용·리워크 경험 [^eco-aberdeen]                   |
| BOM 기준본 부재         | "설계팀 BOM이랑 생산팀 BOM이 다르다"           | 부서 간 SSOT 없음                                                                                 |

### 생산 현장 문제

| 문제 | 고객 언어 | 정량 근거 |
| ---- | --------- | --------- |
| 생산 현황 파악 불가 | "오늘 뭘 몇 개 만들었는지 엑셀 정리해야 안다" | 소기업 MES 도입률 극히 낮음. 수기/엑셀로 생산실적 관리 |
| 불량 원인 역추적 불가 | "불량 나면 어떤 도면·부품이 문제인지 추적이 안 된다" | 설계 데이터와 생산 데이터가 분리되어 품질→부품 역추적 불가 |
| 설계→생산 단절 | "설계팀이 BOM 바꿔도 현장 작업지시서는 그대로" | PLM과 MES가 별도이거나 둘 다 없어서 설계 변경이 생산에 반영 안 됨 |

**출처:**

[^bom-time]: OpenBOM Blog, "The Real Cost of Managing BOMs in Spreadsheets"
[^search-aiim]: AIIM, "The Cost of Not Finding Information" — 지식노동자 업무시간의 약 30% 정보 검색 소요
[^search-mckinsey]: McKinsey Global Institute, "The Social Economy" — 상호작용 노동자 근무시간 19% 정보 추적
[^search-ihs]: IHS Engineering Productivity Infographic — 엔지니어 시간의 40% 정보 검색 (상한값, 25~30%가 보수적 범위)
[^eco-aberdeen]: Aberdeen Group (2차 자료 인용) — BOM 오류 = ECO 최대 30%, ECO당 $1,500~3,000. 원본 보고서 직접 확인 불가, CADTALK·OpenBOM 등에서 재인용

> [!info] 핵심
> 문제의 본질은 **"제조 데이터가 갇혀 있고, 설계와 생산이 단절되어 있다"**는 것이다.
> 도면은 폴더에, BOM은 엑셀에, 생산실적은 수기장부에, 공급사 정보는 담당자 머릿속에 흩어져 있어 검색·변경추적·품질 역추적·의사결정이 불가능하다.

## 3. 타겟 고객

### 핵심 타겟

| 항목 | 정의 |
|------|------|
| 업종 | C25(금속가공), C29(기계장비), C28(전기장비) 우선 |
| 규모 | 종사자 10~299명, 매출 10억+ |
| 조건 | CAD/도면 사용 + PLM/MES 미사용 + ERP는 사용 중이거나 도입 예정 |
| 의사결정자 | 대표 또는 설계/생산 팀장 |
| 실사용자 | 설계 엔지니어, 생산기술, 구매 담당 |

### 유료 전환 가능성 순위

1. **ERP는 있는데** BOM/도면은 엑셀·폴더, 생산실적은 수기인 업체 (Fabbit = ERP 업스트림 + 경량 MES)
2. 설계변경이 잦고 외주/협력사가 많은 업체 (도면 배포 + 생산 반영 이슈)
3. 품번/리비전 체계가 약해서 구매/생산 오류가 반복되는 업체
4. 스마트공장 바우처 수혜 기업 (PLM+MES를 하나로 해결)

### 제외 대상

- 1~9명 영세기업: 관리 capa 자체가 없음 (1차 인터뷰 검증)
- ERP로 모든 관리가 완결되는 곳: 추가 도구 니즈 없음
- 대기업: 이미 엔터프라이즈 PLM 사용 중

시장 규모 상세는 [[40_타겟/시장규모-산정]] 참고.

## 4. 핵심 가치 (Solution)

### 차별점 1: AI 추출 — 진입장벽 제거

도면(DWG/PDF) 업로드 시 AI가 부품표를 자동 인식하여 품명/재질/수량/규격을 추출, 시스템 내 구조화 BOM으로 자동 등록한다.

- 기존: 엔지니어가 도면을 보며 Excel에 수동 입력 (건당 30분~1시간)
- Fabbit: 업로드 → AI 자동 추출 → 구조화 등록 (즉시)
- 현재 구현: BOM 자동 분석 완료. 도면 표제란 인식은 고도화 예정.

> 데이터를 시스템에 넣는 비용이 0에 가까워야 중소기업이 제조 데이터 관리를 시작한다.

### 차별점 2: 설계+생산 통합 — 하나의 플랫폼에서 설계부터 생산까지

기존에 소기업이 설계 관리(PLM)와 생산 관리(MES)를 별도로 도입해야 했던 것을, Fabbit 하나로 해결한다.

| 영역 | 기능 | 기존 방식 | Fabbit |
|------|------|-----------|--------|
| 설계 (PLM) | 도면·BOM 관리, 버전/리비전, 변경관리(ECR/ECO) | 폴더+엑셀 또는 별도 PLM | 통합 제공 |
| 생산 (경량 MES) | 작업지시, 생산실적, 품질추적 | 수기장부+엑셀 또는 별도 MES | 통합 제공 |
| **연결** | 설계 변경 → 작업지시 자동 반영, 불량 → 부품/도면 역추적 | 불가능 (시스템 분리) | **온톨로지로 자동 연결** |

> 소기업에게 PLM과 MES를 따로 파는 건 비현실적이다. "설계부터 생산까지 한 곳에서"가 Fabbit의 포지션이다.

### 차별점 3: Agentic AI — 데이터에서 AI가 먼저 발견하고 행동한다

설계 데이터(부품, BOM, 도면)와 생산 데이터(생산실적, 품질, 공급사)를 그래프 DB(Apache AGE)로 연결한다. 기존 KG+AI가 "사용자가 질문하면 답하는" Reactive 방식이라면, Fabbit은 **AI가 데이터를 지속 분석하여 사전 감지·대안 탐색·행동 초안까지 자동화하는** Agentic AI를 목표로 한다.

| 단계 | 방식 | 예시 |
|------|------|------|
| **Reactive** (질의응답) | 사용자가 질문 → AI가 답변 | "이 부품 어디에 쓰여?" → where-used 결과 반환 |
| **Proactive** (사전 감지) | AI가 데이터 변화를 지속 모니터링 → 이상 시 선제 알림 | "부품 A의 단일 공급사가 납기 지연 3회 → 리스크 알림" |
| **Agentic** (자율 행동) | 감지 → 대안 탐색 → 행동 초안까지 자동 | "리스크 감지 → 대체 공급사 3곳 추출 → 담당자에게 발주 초안 제시" |

Fabbit의 궁극적 목표는 Agentic 단계이며, 팔란티어가 이를 **Systems of Action**이라 부른다.

**구체적 적용:**

- **변경 영향 추적**: 부품 변경 시 영향받는 제품·공급사·공정·작업지시를 자동 탐색하고 담당자에게 알림
- **품질→설계 피드백**: 생산 불량 데이터에서 문제 부품·도면을 역추적하여 설계 개선 제안
- **원가 이상 감지**: 공급사 단가 변동 → 관련 BOM 원가 영향을 선제 분석
- **리스크 감지**: 단일 소싱, 납기 지연 패턴, 품질 이슈 등을 AI가 사전 포착
- **행동 연결**: 감지된 리스크에 대해 대체 부품·공급사 후보 제안, ECR 초안 자동 생성

기반 인프라 구현 완료: `age_client`(그래프 DB), `ontology`(품목 분류), `llm_client`(자연어 쿼리).

### 기존 시스템과의 관계

Fabbit은 ERP나 전문 MES의 경쟁자가 아니다. 소기업에서 **빠져 있는 레이어를 채우는** 역할이다.

| 시스템 | 잘하는 것 | 약한 것 (Fabbit의 기회) |
|--------|-----------|----------------------|
| **ERP** | 구매·재고·회계·원가 트랜잭션 | 도면 버전 통제, E-BOM 정리, 변경 이력 |
| **전문 MES** | 대규모 생산라인 실시간 제어, PLC/SCADA 연동 | 고가·복잡, 소기업 도입 비현실적 |

- Fabbit은 **ERP 업스트림**(BOM 정리 → ERP로 전달)이자 **경량 MES**(작업지시·생산실적·품질)
- "ERP에 넣기 전에 BOM을 정리하고, 생산 현장 데이터를 수집하는 과정"이 지금은 엑셀 → Fabbit이 대체
- 소기업이 성장하여 전문 MES가 필요해지면, Fabbit의 설계 데이터를 연동하는 구조

> 제조 도메인 시스템 전체 흐름은 [[10_제품/plm/제조-도메인-시스템]] 참고.
> 경쟁사와의 상세 비교는 [[10_제품/plm/경쟁사-비교]] 참고.

## 5. 과금 방식

회사 단위 Freemium 모델. 상세는 [[30_영업/수익모델-정의]] 참고.

| 플랜 | 월 요금 | 멤버 | 스토리지 | AI |
|------|--------|------|---------|-----|
| Starter | 무료 | 5명 | 10GB | 포함 |
| Team | 24.9만원 | 20명 | 500GB | 포함 |
| Business | 59.9만원 | 50명 | 2TB | 포함 |
| Enterprise | 별도 협의 | 무제한 | 5TB+ | 포함 |

> [!info] Freemium + 회사 단위 과금의 이유
> 중소 제조업은 설계자(편집) 소수 + 생산/구매/품질(조회) 다수 구조.
> 인당 과금이면 조회자에게도 비용이 붙어 "설계팀만 쓰자"가 되고, 부서 간 SSOT가 깨진다.
> 제조 데이터 관리 시스템 도입률이 극히 낮은 시장에서 무료 Starter로 진입장벽을 제거하고,
> 팀 규모·데이터량 증가에 따라 자연스럽게 유료 전환.

## 6. 단계별 비전

| 단계 | 시기 | 역할 | 설명 |
|------|------|------|------|
| **1단계: AI 추출** | 현재 | 진입 장벽 제거 | BOM 자동 분석(구현 완료), 도면 인식(고도화 예정) |
| **2단계: 설계+생산 통합** | ~6개월 | 데이터 축적 | 계층형 BOM, 리비전 추적, 변경관리(ECR/ECO) + 작업지시·생산실적·품질추적(경량 MES) |
| **3단계: 온톨로지 플랫폼** | ~12개월 | 핵심 가치 | 설계 데이터+생산 데이터를 온톨로지로 연결 → AI 의사결정 지원. 팔란티어 Foundry의 중소 제조업 버전 |
| **확장: 외부 연동** | ~18개월 | 생태계 | ERP 연동, 협력사 포털, 전문 MES/센서 연동 → 설계→생산 피드백 루프 고도화 |

## 7. 왜 지금

1. **제조 데이터 관리 부재**: 중소 제조업 95.6%가 PLM 없이, MES 도입률도 극히 낮음 → 설계·생산 데이터가 체계적으로 관리되지 않는 블루오션
2. **AI 기술 성숙**: LLM 기반 도면 인식이 상용 수준에 도달. 기업 AI 사용 비율 2023년 8.7% → 2025년 20.2%로 급증 (OECD)
3. **클라우드/SaaS 수용성**: 제조업 10인 이상 클라우드 이용률 73.9%
4. **AI 추출 시장 검증**: 이 영역에서 투자 유치·정부 지원 사례가 나오고 있어 시장 존재가 입증됨
5. **통합 플랫폼 공백**: AI 추출 플레이어는 추출→Excel에서 끝나고, 기존 PLM/MES는 별도 제품. 설계+생산을 AI로 통합하는 플랫폼 영역이 비어 있다
