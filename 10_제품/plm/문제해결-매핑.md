# 문제해결 매핑

이 문서는 고객 문제, Fabbit 해결 방식, 기능, 성과 지표를 한 번에 연결한다.
제품 설명이 아니라 사업계획서 Solution 근거로 사용한다.

## 1. 매핑 기준

- 문제는 고객 언어로 작성한다.
- 해결 방식은 기능 설명이 아니라 결과 중심으로 작성한다.
- 성과 지표는 도입 전/후 비교가 가능한 지표만 사용한다.
- **비링커와 겹치는 영역은 명시**하고, Fabbit만의 차별 해결을 구분한다.

## 2. 경쟁사 해결 범위 비교

각 문제에 대해 기존 방식 / 비링커 / 기존 PLM / Fabbit이 어디까지 해결하는지 표시.

| 고객 문제                  | 엑셀/파일서버 | 비링커                           | 기존 PLM     | **Fabbit**         |
| ---------------------- | ------- | ----------------------------- | ---------- | ------------------ |
| P1. BOM 수동 작성 시간 낭비    | 미해결     | **해결** (AI→Excel)             | 미해결 (수동입력) | **해결** (AI→시스템 등록) |
| P2. 도면 검색·버전 혼선        | 미해결     | **해결**                        | **해결**     | **해결**             |
| P3. BOM 오류 → 재발주/생산 중단 | 미해결     | **부분 해결** (추출 오류↓, 관리 오류 그대로) | **해결**     | **해결**             |
| P4. 설계변경 반영 지연         | 미해결     | **미해결** (변경관리 없음)             | **해결**     | **해결**             |
| P5. BOM 데이터 SSOT 부재    | 미해결     | **미해결** (Excel 내보내기 후 분산)     | **해결**     | **해결**             |

> [!info] 핵심 인사이트
> P1·P2는 비링커도 해결한다 — 여기서는 Fabbit이 후발.
> P3·P4·P5는 비링커가 해결하지 못한다 — **Fabbit의 차별 영역**.
> 사업계획서에서는 P3~P5를 중심으로 차별성을 주장해야 한다.

## 3. 매핑 상세

### P1. BOM 수동 작성에 시간을 너무 많이 쓴다

- **고객 언어**: "도면 보면서 Excel에 품명·수량 하나하나 옮기는 데 30분~1시간씩 걸린다"
- **현행 방식**: 도면(PDF/CAD) → 눈으로 확인 → Excel 수동입력
- **비링커 해결**: 도면 업로드 → AI가 품명·재질·수량·후처리 추출 → Excel 다운로드 (3초)
- **Fabbit 해결**: 도면 업로드 → AI가 표제란·부품표 인식 → **시스템 내 BOM 자동 등록** (Excel 거치지 않음)
- **Fabbit 추가 가치**: 추출된 BOM이 시스템에 바로 들어가므로 별도 정리·재입력 불필요
- **기능**: 도면 AI 인식, BOM 자동생성, 표제란 파싱
- **성과 지표**:
  - 도입 전: BOM 작성 건당 30~60분
  - 도입 후: 건당 < 5분 (목표, 검수 포함)
  - 절감: 연간 엔지니어링 시간 __시간 (인터뷰 데이터로 보정 필요)

### P2. 필요한 도면을 찾는 데 시간을 너무 많이 쓴다

- **고객 언어**: "업무시간의 30%를 도면 찾는 데 쓴다" / "최신 버전이 어떤 건지 모르겠다"
- **현행 방식**: 공유폴더/NAS에서 파일명·폴더 기반 수동 검색
- **비링커 해결**: 클라우드 도면관리, 메타데이터 검색, 버전/이력 추적
- **Fabbit 해결**: 클라우드 도면관리 + BOM 연동 검색 (부품번호로 관련 도면 역추적)
- **Fabbit 추가 가치**: 도면과 BOM이 연결되어 있으므로 "이 부품이 쓰인 도면" 역방향 검색 가능
- **기능**: 도면 저장소, 메타데이터 관리, 버전관리, BOM↔도면 연계 검색
- **성과 지표**:
  - 도입 전: 도면 검색 건당 __분 (인터뷰 데이터 필요)
  - 도입 후: 건당 < 1분 (목표)
  - 버전 혼선 발생률: → 0건 (목표)

### P3. BOM 오류 때문에 재발주하거나 생산이 멈춘다 ⭐

- **고객 언어**: "수량 잘못 넣어서 자재 다시 발주했다" / "사양 틀려서 생산 라인이 멈췄다"
- **현행 방식**: Excel 수동입력 → 오타·수량오류 → 재발주, 납기 지연
- **비링커 해결**: AI 추출로 **입력 단계** 오류 감소. 단, 추출 후 Excel로 관리하므로 복사본 분산, 수정 이력 부재, 후속 오류는 그대로 발생
- **Fabbit 해결**: AI 추출 + **시스템 내 BOM 관리** → SSOT로 입력~관리 전 과정의 오류 감소
- **Fabbit 차별점**: 비링커는 "뽑아주기"까지, Fabbit은 "뽑은 후 관리"까지
- **기능**: BOM 자동생성, 구조화 BOM 관리 (계층·where-used), BOM 리비전 추적
- **성과 지표**:
  - 도입 전: 월간 BOM 오류 __건, 오류당 수정 __시간, 원가 3~5% 낭비
  - 도입 후: BOM 오류 __% 감소 (목표)
  - 재발주 비용 절감: 연간 __만원 (인터뷰 데이터로 산출)

### P4. 설계변경이 생겨도 반영이 며칠~몇 주 걸린다 ⭐

- **고객 언어**: "설계 바뀌었는데 생산팀이 모르고 구버전으로 만들었다"
- **현행 방식**: 이메일/전화 수동 통보 → 수일~수주 소요 → 누락 빈번
- **비링커 해결**: **미해결** — 변경관리(ECR/ECO) 기능 없음. 도면 이력은 추적하지만 변경 프로세스는 없음
- **Fabbit 해결**: ECR 등록 → 영향 범위 자동 표시 → ECO 승인 → BOM·도면 리비전 자동 반영
- **Fabbit 차별점**: 비링커와의 결정적 차이. 변경관리 워크플로우는 PLM 영역이며 비링커는 진입하지 않음
- **기능**: ECR/ECO 워크플로우, 변경 영향 분석 (where-used), 도면·BOM 리비전 연동
- **성과 지표**:
  - 도입 전: 설계변경 통보→반영 __일
  - 도입 후: < 1일 (목표)
  - 구버전 사용 사고: → 0건 (목표)

### P5. BOM 기준본이 어디에 있는지 모른다 ⭐

- **고객 언어**: "설계팀 BOM이랑 생산팀 BOM이 다르다" / "최신 BOM이 어떤 파일인지 모르겠다"
- **현행 방식**: 부서별 Excel 복사본 → SSOT 부재 → 부서 간 불일치
- **비링커 해결**: **미해결** — 도면은 클라우드에서 SSOT이지만, BOM은 Excel로 추출 후 분산 관리되므로 SSOT 불가
- **Fabbit 해결**: BOM이 시스템 내에 구조화 데이터로 존재 → 단일 기준본 유지
- **Fabbit 차별점**: "뽑아서 Excel로 주는 것"(비링커)과 "시스템 안에서 관리하는 것"(Fabbit)의 근본적 차이
- **기능**: 구조화 BOM 관리, BOM 리비전, 접근 권한 제어
- **성과 지표**:
  - 도입 전: 부서 간 BOM 불일치 월 __건
  - 도입 후: 0건 (SSOT 보장)

## 4. MVP 범위

> 인터뷰 결과 + MVP 성능 데이터로 보정 필요. 아래는 현재 기준 초안.

### 반드시 포함 (MVP)

| 기능 | 해결 문제 | 비링커 대비 | 우선순위 근거 |
|------|----------|-----------|-------------|
| 도면 AI 인식 → BOM 자동생성 | P1 | 동등 (table stakes) | 비링커와 최소 동등해야 시장 진입 가능 |
| 도면 저장소 + 버전관리 | P2 | 동등 | 기본 기능 |
| 구조화 BOM 관리 (계층, where-used) | P3, P5 | **차별** | 비링커가 못하는 영역, 핵심 차별점 |
| BOM 리비전 추적 | P3, P5 | **차별** | SSOT 보장의 핵심 |
| DWG/PDF 뷰어 | P2 | 동등 | 기본 기능 |

## 5. 사업계획서 활용 가이드

### Solution 항목 서술 프레임

1. **문제**: 제조 데이터가 갇혀 있다 → 검색·변경추적·의사결정 불가
2. **진입 전략**: AI 추출로 도입 장벽 제거 (P1·P2)
3. **PLM 관리**: 추출된 데이터를 시스템 안에서 축적 (P3·P4·P5)
4. **핵심 가치**: 축적된 데이터를 온톨로지로 연결 → 데이터 위에서 동작하는 AI (변경 영향·원가·리스크)

### 주의사항

- ~~"국내외 어디에도 없는 기능"~~ → 사용 금지
- AI 추출은 **진입 수단**으로 서술 (차별점이 아닌 기본 기능)
- 차별점은 **데이터 연결(온톨로지)과 데이터 기반 AI** 에 집중
