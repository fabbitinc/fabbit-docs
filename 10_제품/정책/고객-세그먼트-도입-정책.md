# 고객 세그먼트·도입 정책

이 문서는 Fabbit의 고객 세그먼트와 기본 도입 경로를 정의하는 현재 정책
문서다. `아이템-정의.md`와 `수익모델-정의.md`에 흩어진 타깃, 전환 우선순위,
CTA 원칙, 도입 기본안을 한곳에 모아 `docs` 기준 SSOT로 삼는다.

이 문서는 전략 문서의 근거를 정리해 현재 운영 기준으로 고정하는 역할을 맡는다.
외부 공개 페이지 기획이나 앱 내부 온보딩 상세는 이 문서를 직접 대체하지 않고,
이 문서를 참조해 파생 문서를 작성한다.

## 1. 문서 역할

이 문서는 현재 누구를 우선 공략하고, 어떤 경로로 도입을 열며, 공개 사이트와
앱 내부 플로우를 어디서 나누는지 정한다. 따라서 `누구에게 팔 것인가`와
`지금 어떻게 시작시킬 것인가`에 대한 기준은 이 문서를 먼저 따른다.

이 문서가 맡는 범위는 아래와 같다.

- 핵심 타깃과 확장 타깃 정의
- 유료 전환 우선순위
- 공개 사이트 CTA 원칙
- 가입, 파일럿 문의, 상담의 기본 도입 경로
- 공개 페이지와 앱 내부 플로우의 경계

이 문서가 직접 다루지 않는 범위는 아래와 같다.

- 플랜/좌석/과금 숫자 상세
- 공개 페이지 카피와 화면 구조
- 가입, 플랜 선택, 업그레이드의 앱 내부 UX
- 영업 운영 세부 절차와 CRM 처리 규칙

## 2. 핵심 타깃

Fabbit의 현재 핵심 타깃은 설계와 생산 데이터가 끊긴 중소 제조업이다. 특히
도면과 BOM을 엑셀, 폴더, 메일로 관리하지만 ERP는 이미 쓰고 있거나 도입을
검토 중인 조직을 우선 본다.

### 핵심 타깃

아래 고객군을 현재 우선 타깃으로 둔다.

| 항목 | 기준 |
|------|------|
| 업종 | `C25(금속가공)`, `C29(기계장비)`, `C28(전기장비)` |
| 규모 | 종사자 `10~49명` 우선, `50~299명` 확장 |
| 운영 조건 | CAD/도면 사용, PLM/MES 미사용 또는 약함, ERP는 사용 중이거나 도입 예정 |
| 의사결정자 | 대표, 공장 책임자, 설계/생산 팀장 |
| 실사용자 | 설계 엔지니어, 생산기술, 구매, 품질 담당 |

이 세그먼트는 도입 필요성이 충분히 크면서도, 대기업용 PLM/MES보다 가벼운
도입 구조를 받아들일 가능성이 높다.

### 확장 타깃

초기 핵심 타깃이 검증된 뒤 아래 고객군으로 확장한다.

- 종사자 `50~299명` 제조업 중 설계변경과 협력사 커뮤니케이션이 잦은 조직
- 스마트공장, 클라우드, AI 바우처 활용 가능성이 높은 조직
- 외주 생산 비중이 높아 기준본과 변경 반영 문제가 큰 조직

### 제외 또는 후순위 대상

아래 고객군은 현재 기본 도입 타깃에서 제외하거나 후순위로 둔다.

- `1~9명` 영세기업
- 이미 엔터프라이즈 PLM이 깊게 정착한 대기업
- ERP 안에서 설계, BOM, 생산 흐름이 사실상 완결된 조직
- 도면/BOM 사용 비중이 낮은 비이산형 제조업

## 3. 유료 전환 우선순위

모든 리드가 같은 전환 가능성을 갖는 것은 아니다. 현재는 사용량 확대보다
유료 전환 가능성이 높은 파일럿 고객을 먼저 확보하는 쪽이 중요하다.

현재 유료 전환 우선순위는 아래와 같다.

1. ERP는 있는데 BOM과 도면은 엑셀, 폴더로 관리하는 업체
2. 설계변경이 잦고 외주, 협력사 커뮤니케이션이 많은 업체
3. 품번, 리비전 체계가 약해 구매, 생산 오류가 반복되는 업체
4. 바우처를 활용해 초기 도입비 저항을 낮출 수 있는 업체

이 우선순위는 단순 관심도보다 `도입 pain의 명확성`, `파일럿 진행 가능성`,
`유료 전환 가능성`을 기준으로 한다.

## 4. 공개 사이트 CTA 원칙

공개 사이트는 앱 가입 진입과 보조 문의를 함께 가지는 채널이다. 현재 정책상
사이트의 메인 목적은 적합한 제조업 사용자를 `가입`으로 연결하고, 추가 검토가
필요한 리드를 `파일럿 신청` 같은 보조 문의로 받는 것이다.

따라서 공개 사이트 CTA 원칙은 아래와 같다.

- 1차 CTA는 `가입`으로 둔다.
- `파일럿 신청`은 보조 문의 경로로 둔다.
- `제품 데모 요청`은 현재 기본 CTA 세트에서 제외한다.
- `지금 결제`, `즉시 구매`는 기본 CTA로 쓰지 않는다.
- 공개 페이지는 플랜 전체를 노출할 수 있지만, 현재 제공하지 않는 플랜은
  비활성 상태로 분명히 표시해야 한다.
- 공개 페이지의 메인 가입 CTA는 앱 내부 회원가입과 플랜 선택 흐름으로
  이어져야 한다.

이 원칙은 랜딩, 요금제 페이지, 신청 폼 문서가 모두 공통으로 따라야 한다.

## 5. 기본 도입 경로

현재 Fabbit의 기본 도입 경로는 `가입 중심`이다. 파일럿 문의와 상담은 존재할
수 있지만, 메인 진입 경로를 대체하지 않는다.

### 기본 경로

현재 기본 도입 경로는 아래 순서로 둔다.

1. 공개 사이트에서 `가입` CTA로 진입한다.
2. 앱에서 회원가입, 워크스페이스 생성, 플랜 선택 흐름으로 이어진다.
3. 현재 제공 상태 기준으로 `Starter` 또는 `Team`을 시작 플랜으로 선택한다.
4. `Organization`, `Enterprise`는 노출되더라도 현재는 비활성으로 처리한다.

### 파일럿

파일럿 신청은 현재 보조 문의 경로다. 아래 상황에서 메인 가입 대신 사용할 수
있다.

- 가입 전에 운영 맥락을 먼저 확인해야 하는 경우
- 현재 제공하지 않는 `Organization`, `Enterprise` 도입 가능성을 문의하는 경우
- 내부 데이터, 업종, 운영 프로세스 검토가 필요한 경우

### 셀프서브

셀프서브는 별도 미래 가능성이 아니라, 현재 메인 가입 흐름에 포함된다. 다만
플랜 제공 범위는 모두 열려 있지 않다.

- 현재 메인 가입 흐름은 `Starter`, `Team` 중심으로 운영한다.
- `Organization`, `Enterprise`는 공개 노출은 가능하지만 현재 제공하지 않는다.
- 결제, 업그레이드, 구독 관리의 상세 UX는 앱 문서군에서 별도로 정리한다.

정리하면, 현재 정책에서 메인 진입은 `가입`이고, 제공 범위는 일부 플랜에 한해
열려 있다.

### 상담

상담은 아래 상황에서 기본 경로가 된다.

- 공개 페이지만으로는 업종, 운영 맥락 설명이 부족한 경우
- `Organization`, `Enterprise` 같은 현재 비활성 플랜 도입 가능성을 검토하는 경우
- 파일럿 신청 후 별도 검토가 필요한 경우

## 6. 공개 페이지와 앱 내부 플로우의 경계

현재 `docs`에서 가장 자주 섞이는 부분은 `site` 문서군과 앱 내부 플로우다.
이 경계를 먼저 고정해야 사이트 기획이 정책이나 구독 UX를 흡수하지 않는다.

### 공개 페이지가 맡는 범위

공개 페이지 문서군은 아래 범위까지만 맡는다.

- 랜딩 메시지와 섹션 구조
- 요금제 페이지의 공개 설명과 CTA
- 가입 진입 CTA와 파일럿 신청 CTA
- 신청 폼의 공개 경험
- 방문자가 보는 비교표, FAQ, 완료 메시지

### 앱 내부 플로우가 맡는 범위

앱 내부 문서군은 아래 범위를 맡는다.

- 회원가입과 워크스페이스 생성
- 플랜 선택과 좌석 배정
- 업그레이드와 구독 변경
- 청구, 한도, 설정, 내부 운영 정책

즉, 공개 페이지는 `신청 전까지`와 `신청 직후의 안내`를 맡고, 앱 내부 문서는
`계정 생성 이후 실제 사용과 구독 관리`를 맡는다.

## 7. 문서 적용 기준

앞으로 아래 문서들은 이 정책 문서를 먼저 참조해야 한다.

- `site` 문서군의 타깃 고객, CTA, 도입 경로 판단
- 사업계획서의 고객 세그먼트와 도입 전략 서술
- 영업 자료의 파일럿 우선 타깃 설명

반대로 아래 내용은 이 문서가 아니라 별도 기준을 참조해야 한다.

- 플랜, 좌석, 과금 숫자: 구독 정책 또는 서버 truth
- 공개 페이지 카피와 화면 구조: `site` 기획 문서
- 가입, 플랜 선택, 업그레이드 UX: 앱 온보딩, 구독 기획 문서

## 8. 기존 문서와의 관계

이 문서는 기존 전략 문서를 대체하기보다, 중복된 핵심 내용을 현재 정책
기준으로 정리한 문서다.

문서별 역할은 아래처럼 본다.

| 문서 | 역할 |
|------|------|
| `아이템-정의.md` | 문제, 가치, 포지셔닝의 상위 전략 근거 |
| `수익모델-정의.md` | BM, 매출 가정, 사업 시나리오 근거 |
| `고객-세그먼트-도입-정책.md` | 현재 타깃과 도입 경로의 운영 기준 |

따라서 기존 전략 문서에 같은 내용이 남아 있더라도, 현재 기준 판단은 이
문서를 우선 참조하는 방향으로 정리한다.

## Next steps

이 문서를 세운 뒤 이어서 정리할 항목은 아래와 같다.

1. `site/index.md`에 공개 페이지 문서군의 역할과 비역할을 반영한다.
2. `site` 문서군에서 셀프서브, 가입, 업그레이드 관련 표현을 정리한다.
3. 앱 온보딩, 구독, 업그레이드 문서군의 상위 진입 문서를 별도로 만든다.
