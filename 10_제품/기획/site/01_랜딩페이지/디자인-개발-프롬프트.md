# 랜딩페이지 디자인·개발 프롬프트

이 문서는 [[10_제품/기획/site/01_랜딩페이지/기획]]을 바탕으로 만든 파생
프롬프트 모음이다. 디자인 AI, 구현 AI, QA 보조 도구에 복사해서 쓰기 위한
작업용 문서이며, 현재 랜딩페이지 기획의 기준 문서가 아니다.

이 문서와 기준 기획 문서가 충돌하면 이 문서를 버리고
[[10_제품/기획/site/01_랜딩페이지/기획]]을 다시 기준으로 프롬프트를
재생성하는 편이 맞다. 또한 플랜/과금, 타깃 고객, 구현 범위 같은 현재 운영
기준은 각각 [[40_사업/정책/플랜-과금-정책]],
[[10_제품/정책/고객-세그먼트-도입-정책]],
[[20_개발/코드베이스-현황]]을 먼저 따른다.

## 1. 문서 역할

이 문서는 아래 용도로만 사용한다.

- 디자인 AI에 랜딩페이지 방향을 빠르게 입력할 때
- 프론트엔드 구현 AI에 섹션별 요구사항을 전달할 때
- QA나 카피 수정용으로 문맥을 재주입할 때

반대로 아래 용도로 사용하면 안 된다.

- 현재 랜딩페이지 확정 기획 문서로 인용
- 정책 SSOT를 대신하는 근거 문서로 사용
- 구현 범위와 준비중 범위를 판정하는 원문으로 사용

## 2. 사용 전에 먼저 확인할 기준

프롬프트를 복붙하기 전에 아래 문서를 먼저 확인한다.

1. [[10_제품/기획/site/01_랜딩페이지/기획]]에서 현재 섹션 구조와 CTA 원칙을 확인한다.
2. [[10_제품/기획/site/index]]에서 `site` 문서군의 범위와 경계를 확인한다.
3. [[10_제품/정책/고객-세그먼트-도입-정책]]에서 현재 타깃 고객과 도입 경로를 확인한다.
4. [[40_사업/정책/플랜-과금-정책]]에서 가격과 플랜 관련 현재 정책을 확인한다.
5. [[20_개발/코드베이스-현황]]에서 구현된 화면과 준비중 범위를 확인한다.

즉, 이 문서는 현재 기준을 요약해서 LLM 입력 형식으로 바꿔 놓은 파생 작업물일
뿐이다.

## 3. 사용 원칙

아래 원칙을 먼저 읽고 프롬프트를 사용한다.

- 프롬프트 안의 문장은 기준 기획을 압축한 작업용 표현이다.
- 현재 기획과 충돌하는 문구가 보이면 이 문서가 아니라 기준 기획 문서를 따른다.
- 구현되지 않은 기능은 운영 중처럼 표현하지 않는다.
- 가격, 가입, 업그레이드 정책은 프롬프트 안에서 임의로 확정하지 않는다.
- 외부 레퍼런스는 구조 리듬 참고용으로만 쓰고, 카피와 레이아웃을 복제하지 않는다.

> [!warning] 해석 원칙
> 이 문서는 복붙용 프롬프트 묶음이다. 현재 제품의 확정 메시지, 확정 가격,
> 확정 구현 범위를 판정하는 원문이 아니다.

## 4. 공통 컨텍스트 프롬프트

아래 프롬프트는 다른 프롬프트 앞에 공통으로 붙이는 기본 컨텍스트다. 사용 전에
반드시 최신 기획과 구현 상태에 맞게 문구를 다시 확인한다.

```text
당신은 B2B SaaS 랜딩 페이지를 설계하는 시니어 디자이너이자 프론트엔드 협업자다.

프로젝트: Fabbit
제품 정의: 중소 제조업의 설계·생산 데이터를 AI로 통합하는 제조 데이터 플랫폼
핵심 진입 문제: 도면은 폴더, BOM은 엑셀, 설계 변경은 메신저/메일로 전달되어 기준본이 흔들린다.
핵심 가치:
1. 도면 업로드만으로 BOM을 구조화한다
2. 추출 후에도 시스템 안에서 BOM과 변경을 관리한다
3. 설계부터 생산 연결까지 확장 가능한 데이터 기준본을 만든다

타깃:
- 10~49명 우선, 50~299명 확장
- 금속가공, 기계장비, 전기장비
- ERP는 있지만 도면/BOM은 엑셀·폴더로 관리하는 팀

피해야 할 것:
- PLM 용어를 첫 화면 전면에 배치
- AI 추출만을 유일한 차별점처럼 과장
- 구현되지 않은 기능을 운영 중처럼 표현
- 외부 레퍼런스 스타일 직접 복제

원하는 결과:
- 신뢰감 있는 B2B SaaS 톤
- 제조업 맥락이 느껴지는 구조적 디자인
- 제품 화면 중심의 설득
- 무료 파일럿 신청과 제품 데모 요청으로 전환
```

## 5. 화면 단위 프롬프트

이 섹션은 화면 수준 결과물을 빠르게 생성하고 싶을 때 사용한다. 각 프롬프트는
기준 문서를 대체하지 않으므로, 결과물을 받은 뒤 기준 기획과 다시 대조해야
한다.

### 5.1 메인 랜딩 전체 디자인 프롬프트

```text
아래 조건으로 Fabbit 메인 랜딩 페이지를 디자인해 주세요.

[목표]
- 첫 방문자가 5초 안에 "도면/BOM 관리와 설계-생산 연결을 돕는 제품"이라고 이해하게 만들기
- 무료 파일럿 신청 또는 제품 데모 요청으로 전환시키기

[레퍼런스 방향]
- 강한 히어로 → 대표 시나리오 스포트라이트 → 핵심 기능 묶음 → 범위/지원 영역 표시 → 신뢰/제품 증거 → 사례 또는 파일럿 검증 → 마지막 CTA 흐름
- 시각 스타일과 카피는 Fabbit 기준으로 재해석

[브랜드 방향]
- Safety Orange + Fast Blue 중심
- 배경은 얇은 데이터 라인이나 도면 그리드 느낌을 약하게 사용
- 캐릭터는 전면 사용하지 않음

[필수 섹션]
1. 헤더
2. 히어로
3. 대표 시나리오 스포트라이트
4. 문제 제시
5. Before/After
6. 해결 방식 3단계
7. 현재 제품 증거
8. 적용 데이터 / 업무 범위
9. 도입 대상
10. 요금 / 도입 방식
11. FAQ
12. 파일럿 검증 / 사례 placeholder
13. 마지막 CTA

[카피 톤]
- 복잡한 제조 IT 용어보다 고객 업무 언어 사용
- 짧고 명확한 문장
- 과장 금지

[CTA]
- 1차: 무료 파일럿 신청
- 2차: 제품 데모 요청

[제약]
- 구현 완료와 준비중 범위를 최신 코드베이스 기준으로 다시 확인
- 미구현 기능은 반드시 준비 중으로 구분

[산출물]
- 데스크톱 기준 전체 랜딩 레이아웃
- 각 섹션의 역할 설명
- 추천 타이포 스케일
- 색상/간격 시스템
- CTA 위치와 반복 전략
```

### 5.2 모바일 랜딩 전체 디자인 프롬프트

```text
Fabbit 메인 랜딩 페이지를 모바일 우선으로 다시 설계해 주세요.

[목표]
- 데스크톱 구조를 축약하지 말고 모바일에서 더 빠르게 이해되도록 재배열
- 히어로, 대표 시나리오, 제품 증거, CTA가 첫 3스크린 안에 들어오도록 구성

[유의점]
- 메뉴는 햄버거로 단순화
- 히어로 비주얼은 핵심 화면 1장 + 짧은 배지로 정리
- 카드형 섹션은 세로 스택 우선
- 가격표와 FAQ는 접고 펼치는 구조 허용
- 마지막 CTA는 sticky button 또는 하단 고정 CTA 여부까지 함께 제안

[산출물]
- 모바일 390px 기준 와이어프레임
- 각 섹션 축약 규칙
- 버튼 우선순위
- 스크롤 흐름 요약
```

### 5.3 리드 폼 / 신청 화면 프롬프트

```text
Fabbit 랜딩 페이지의 전환용 신청 폼 화면을 설계해 주세요.

[목표]
- 단순 문의 폼이 아니라 리드 선별 폼으로 설계
- 작성 피로는 낮추고, 파일럿 적합도는 판단 가능하게 만들기

[필수 입력 항목]
- 회사명
- 업종
- 직원 수
- 역할
- 현재 관리 방식
- ERP 사용 여부
- 가장 큰 문제
- 연락처

[구조]
- 상단: 왜 이 정보를 받는지 설명
- 중단: 2단계 또는 3단계 폼
- 우측 또는 하단: 파일럿 대상 안내 카드

[톤]
- 영업 압박처럼 보이지 않게
- 파일럿 적합 여부를 빠르게 확인하고 데모를 준비하기 위한 정보라는 맥락 전달

[산출물]
- 폼 레이아웃
- 필드 우선순위
- 에러/완료 상태
- 제출 완료 화면 문구
```

## 6. 메뉴 / IA 프롬프트

이 섹션은 랜딩의 내비게이션 구조를 빠르게 실험할 때 쓴다. 최종 채택 여부는
반드시 기준 기획 문서의 CTA 구조와 다시 맞춰 본다.

### 6.1 헤더 메뉴 설계 프롬프트

```text
Fabbit 메인 랜딩 페이지의 헤더 메뉴 IA를 설계해 주세요.

[조건]
- 메뉴 수는 6개 이하
- 스크롤 랜딩 기준
- 첫 방문자에게 필요한 정보만 남길 것

[필수 후보]
- 문제
- 핵심 기능
- 도입 대상
- 요금
- FAQ
- 무료 파일럿 신청

[판단 기준]
- B2B 구매자 시선에서 정보 탐색 순서
- 스크롤 이동 편의성
- 모바일 햄버거 메뉴 대응

[산출물]
- 최종 메뉴 구조
- 데스크톱/모바일 차이
- sticky header 동작 제안
```

## 7. 섹션 단위 디자인 프롬프트

이 섹션은 특정 블록만 빠르게 다시 써 보고 싶을 때 사용한다. 결과를 바로
확정안으로 보지 말고, 기준 기획의 해당 섹션과 대조해야 한다.

### 7.1 히어로 섹션 프롬프트

```text
Fabbit 랜딩 페이지의 히어로 섹션만 집중해서 디자인해 주세요.

[핵심 메시지]
도면·BOM 관리부터 생산 연결까지, 엑셀 없이 한 번에

[서브 메시지]
도면 업로드만으로 BOM을 구조화하고, 설계 변경과 현장 실행이 이어지는 기준본을 만든다.

[필수 요소]
- 헤드라인
- 서브카피
- 보조 배지 4개 내외
- 기본 CTA: 무료 파일럿 신청
- 보조 CTA: 제품 데모 요청
- 실제 제품 화면 합성 이미지

[원하는 인상]
- 무거운 제조 SI 느낌보다 빠르고 명확한 SaaS 느낌
- 지나치게 일반적인 스타트업 랜딩처럼 보이지 않기
- 구조와 신뢰가 느껴질 것

[금지]
- 추상 이미지 남발
- 과장된 AI 이미지
- 구현되지 않은 화면을 메인 히어로처럼 제시
```

### 7.2 대표 시나리오 스포트라이트 프롬프트

```text
히어로 바로 아래에 들어갈 Fabbit 대표 시나리오 스포트라이트 섹션을 디자인해 주세요.

[의도]
- 방문자가 "이 제품은 도면 업로드 후 BOM이 시스템에 쌓이는 경험을 가장 먼저 보여주는구나"라고 이해하게 만들기

[핵심 문장]
도면 업로드만 하면 BOM이 시스템에 쌓입니다
```

## 8. next steps

이 문서는 파생 프롬프트 묶음이므로, 기준 기획이 갱신되면 함께 재검토해야 한다.

- 랜딩페이지 기획이 바뀌면 이 문서를 그대로 믿지 말고 다시 생성한다.
- 구현 범위가 바뀌면 프롬프트의 제약 문구를 먼저 고친다.
- 실제 handoff가 필요하면 프롬프트 문서가 아니라 별도 handoff 문서로 분리한다.
