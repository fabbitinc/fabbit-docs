# 랜딩페이지 기획

이 문서는 Fabbit 공개 사이트의 메인 랜딩 페이지를 정의한다. 랜딩 페이지의
목적은 브랜드 소개 자체가 아니라, 초기 타깃 고객에게 Fabbit의 문제 해결
범위를 짧게 설명하고 파일럿, 데모, 상담 전환을 만드는 것이다.

이 문서는 공개 첫 화면의 메시지와 섹션 구조를 다룬다. 상세 가격 숫자,
회원가입, 플랜 선택, 업그레이드, 운영 후속 처리 같은 앱 내부 또는 정책
영역은 이 문서에서 직접 소유하지 않는다.

## 1. 문서 역할과 범위

랜딩 문서는 공개 방문자가 스크롤하며 이해해야 하는 질문에만 답한다. 즉,
`왜 Fabbit이 필요한가`, `어떤 문제를 푸는가`, `어떤 팀에게 맞는가`,
`어떤 CTA로 시작하는가`를 설계하는 문서다.

이 문서가 맡는 범위는 아래와 같다.

- 히어로 메시지와 첫 화면 카피
- 문제 제시와 해결 구조
- 공개 가능한 제품 증거와 도입 대상 표현
- 랜딩 안의 CTA 종류와 배치
- 요금제 페이지, 신청 폼으로의 연결 방식

이 문서가 맡지 않는 범위는 아래와 같다.

- 플랜, 좌석, 스토리지, AI 정책의 상세 숫자
- 셀프서브 가입과 워크스페이스 생성 흐름
- 앱 내부 플랜 선택과 업그레이드 UX
- 리드 분류 운영 규칙과 CRM 처리
- 이벤트 스키마, 상태 키, 구현 명세

상세 가격 구조는 [[10_제품/기획/site/03_요금제페이지/기획]]과
[[40_사업/정책/플랜-과금-정책]]을 따른다. 공개 신청 경험은
[[10_제품/기획/site/02_신청-전환-흐름/기획]]에서 별도로 다룬다.

## 2. 문제와 목표

랜딩 페이지는 아래 문제를 해결해야 한다.

1. `PLM`이나 `MES` 같은 제품 분류어만으로는 타깃 고객이 가치를 이해하기 어렵다.
2. Fabbit의 차별점은 AI 추출 단독이 아니라 구조화와 연결인데, 이 범위를 짧은
   스크롤 안에서 전달해야 한다.
3. 공개 레퍼런스가 많지 않은 초기 단계이므로, 현재 구현 화면과 문제-해결
   구조를 중심으로 신뢰를 만들어야 한다.
4. 현재 기본 전환 경로는 셀프서브 가입이 아니라 파일럿, 데모, 상담이다.

페이지 목표는 아래와 같이 둔다.

| 목표 | 설명 | 성공 기준 |
|------|------|-----------|
| 문제 공감 형성 | 방문자가 자사 문제와 겹친다고 느끼게 한다 | 문제 섹션 도달률, 스크롤 깊이 |
| 포지셔닝 전달 | Fabbit이 단순 추출 도구가 아니라는 점을 이해시킨다 | 핵심 기능 섹션 체류 시간 |
| 타깃 선별 | 적합 고객이 스스로 해당 여부를 판단하게 한다 | CTA 클릭 전 도입 대상 섹션 도달 |
| 전환 확보 | 파일럿, 데모, 상담 전환을 만든다 | CTA 클릭률, 폼 진입률 |

## 3. 주요 사용자

현재 랜딩 페이지는 고객 세그먼트 정책 문서의 핵심 타깃을 그대로 따른다.
즉, 설계와 생산 데이터가 끊긴 중소 제조업을 우선 겨냥한다.

| 사용자 | 현재 문제 | 랜딩에서 확인하려는 것 | 권장 CTA |
|--------|-----------|------------------------|----------|
| 대표 / 공장 책임자 | 도면과 BOM이 엑셀, 폴더에 흩어져 운영이 불안정 | 도입 가치와 시작 방식 | `무료 파일럿 신청` |
| 설계 / 생산 팀장 | 설계 변경이 현장에 늦게 반영되고 기준본이 흔들림 | 실제로 어떤 문제를 줄이는지 | `제품 데모 요청` |
| 설계 엔지니어 / 생산기술 | 수동 입력, 검색, 버전 혼선이 반복됨 | 화면이 업무를 얼마나 줄이는지 | `제품 데모 요청` |

우선 공략 세그먼트는 아래와 같다.

- 업종: `C25(금속가공)`, `C29(기계장비)`, `C28(전기장비)`
- 규모: 종사자 `10~49명` 우선, `50~299명` 확장
- 조건: ERP는 있거나 도입 예정, PLM/MES는 없거나 약함
- 니즈: 도면과 BOM은 엑셀, 폴더로 관리하고 설계변경과 생산 반영이 끊겨 있음

세그먼트와 도입 원칙의 최종 기준은
[[10_제품/정책/고객-세그먼트-도입-정책]]을 따른다.

## 4. 내비게이션과 CTA 구조

랜딩 페이지는 1페이지 스크롤 구조를 기본으로 한다. 목적은 한 페이지 안에서
문제, 해결, 증거, 도입 대상, 전환을 닫는 것이다.

### 권장 헤더 구조

- 로고
- `문제`
- `핵심 기능`
- `도입 대상`
- `요금`
- `FAQ`
- 우측 CTA: `무료 파일럿 신청`

### 권장 1차 메시지

| 구분 | 권장 문구 | 사용 위치 |
|------|-----------|-----------|
| H1 | `도면·BOM 관리부터 생산 연결까지, 엑셀 없이 한 번에` | 히어로 |
| 서브카피 | `Fabbit은 중소 제조업의 설계·생산 데이터를 AI로 연결하는 제조 데이터 플랫폼이다. 도면 업로드만으로 BOM을 구조화하고, 설계 변경과 현장 실행이 이어지는 기준본을 만든다.` | 히어로 |
| 짧은 소개 | `설계부터 생산까지 연결하는 제조 데이터 플랫폼` | 헤더, 메타 설명, 공유 카드 |

### CTA 원칙

- 1차 CTA는 `무료 파일럿 신청`
- 2차 CTA는 `제품 데모 요청`
- `지금 가입`, `지금 결제`, `즉시 구매`는 쓰지 않는다
- `Starter 무료`는 보일 수 있지만, 랜딩의 기본 전환 경로를 대체하지 않는다
- 상세 가격 정책은 랜딩 안에서 풀지 않고 요금제 페이지로 넘긴다

## 5. 핵심 시나리오

### 시나리오 1. 차가운 유입 → 파일럿 신청

1. 방문자가 검색, 커뮤니티, 소개 링크로 유입한다.
2. 히어로에서 `도면·BOM 관리` 문제 해결 서비스임을 즉시 이해한다.
3. 문제 섹션과 Before/After 흐름을 보고 자사 운영 방식과 겹친다고 느낀다.
4. 현재 제품 증거와 도입 대상 섹션을 확인한 뒤 `무료 파일럿 신청`을 누른다.

### 시나리오 2. 문제 공감은 있지만 확신이 부족한 경우

1. 방문자가 기능 섹션에서 AI 추출, 구조화 BOM, 변경 추적을 확인한다.
2. 현재 구현 범위와 다음 단계 구분을 보고 과장되지 않았다고 느낀다.
3. 요금과 도입 방식은 요약만 확인하고, 더 자세한 정보는 요금제 페이지나
   데모를 통해 확인한다.
4. `제품 데모 요청`으로 전환한다.

### 시나리오 3. 재방문 리드 → 상담 전환

1. 이전에 소개를 들은 방문자가 직접 URL로 재방문한다.
2. 도입 대상과 공개 가능한 제품 범위를 빠르게 훑는다.
3. 적합하다고 판단하면 파일럿, 데모, 상담 전환 CTA를 누른다.

## 6. 화면 기획

### 6.1 히어로

첫 화면은 제품 분류보다 고객 문제와 해결 범위를 먼저 이해시키는 데 집중한다.

- 헤드라인:
  `도면·BOM 관리부터 생산 연결까지, 엑셀 없이 한 번에`
- 서브카피:
  `도면 업로드만으로 BOM을 구조화하고, 설계 변경과 현장 실행이 이어지는
  기준본을 만든다. Fabbit은 중소 제조업을 위한 제조 데이터 플랫폼이다.`
- 보조 배지:
  - `AI 도면 분석`
  - `구조화 BOM`
  - `변경 추적`
  - `생산 연결`
- CTA:
  - 기본: `무료 파일럿 신청`
  - 보조: `제품 데모 요청`

### 6.2 대표 시나리오 스포트라이트

히어로 직후에는 Fabbit의 가장 강한 진입 시나리오 하나를 다시 강조한다.

- 권장 제목:
  `도면 업로드만 하면 BOM이 시스템에 쌓입니다`
- 권장 설명:
  `PDF/DWG 도면을 업로드하면 AI가 부품표를 읽고, 결과를 엑셀 파일이 아니라
  Fabbit 안의 구조화 BOM으로 남긴다. 추출 이후의 관리까지 한 흐름으로
  이어진다.`
- 권장 CTA:
  `제품 데모 요청`

### 6.3 문제 제시 섹션

문제 섹션은 방문자가 `우리도 이렇다`라고 느끼게 하는 데 목적이 있다.

권장 제목:
`문제는 도면과 BOM이 따로 노는 순간 시작됩니다`

노출할 핵심 문제는 아래 4개로 압축한다.

| 문제 | 고객 언어 | 근거 방향 |
|------|-----------|-----------|
| BOM 수동 입력 | "도면 보면서 엑셀로 다시 치는 데 너무 오래 걸린다" | 반복 수작업 |
| 도면 검색/버전 혼선 | "최신본 찾는 데 시간이 다 간다" | 기준본 부재 |
| 설계변경 반영 지연 | "현장은 구버전으로 생산하고 있다" | 변경 추적 부재 |
| 생산 피드백 단절 | "불량이 나도 어떤 부품, 도면 문제인지 바로 못 찾는다" | 설계-생산 연결 부재 |

### 6.4 Before / After 섹션

- Before:
  `도면은 폴더`, `BOM은 엑셀`, `생산실적은 수기`, `공급사 정보는 담당자 기억`
- After:
  `AI 자동 추출 → 구조화 BOM → 변경 추적 → 생산 연결`
- 한 줄 요약:
  `데이터를 뽑는 것에서 끝나지 않고, 같은 시스템 안에서 관리하고 연결한다`

### 6.5 해결 방식 3단계 섹션

랜딩은 제품 전체를 단계적으로 이해시키되, 현재 구현과 비전을 섞어 말하지
않아야 한다.

| 단계 | 방문자 언어 | 상태 | 랜딩 표현 |
|------|-------------|------|-----------|
| 1단계 | `AI가 도면에서 BOM을 자동으로 만든다` | 구현 | 실제 화면 중심 |
| 2단계 | `뽑은 BOM을 시스템 안에서 관리하고 변경을 추적한다` | 구현/고도화 | 기능 카드와 화면 |
| 3단계 | `설계와 생산 데이터를 연결해 운영 흐름을 잇는다` | 방향성 | 준비 중 또는 다음 단계 표시 |

### 6.6 현재 제품 증거 섹션

제품 증거 섹션은 실제로 구현된 화면 중심으로 구성한다.

| 화면 | 전달 메시지 | 현재 상태 |
|------|-------------|-----------|
| AI 도면 인식 결과 | 비정형 도면을 바로 데이터로 바꾼다 | 구현 |
| BOM 트리 / 부품 탐색 | 엑셀이 아니라 시스템 안에서 구조화 관리한다 | 구현 |
| 변경 요청 / 이슈 화면 | 설계 변경을 추적 가능한 흐름으로 관리한다 | 구현 |
| 생산 연결 개념 화면 | 설계 기준본이 생산으로 이어질 방향을 보여준다 | 준비 중 또는 다음 단계 |

### 6.7 적용 데이터 / 업무 범위 띠

이 영역은 기술 스택보다 현재 다루는 제조 데이터와 업무 범위를 빠르게
인지시키는 데 목적이 있다.

현재 노출 가능한 항목:

- `DWG/PDF 도면`
- `부품`
- `BOM`
- `공급사`
- `변경 요청`
- `프로젝트`

후속 확장 항목:

- `작업지시`
- `생산실적`
- `품질`

### 6.8 도입 대상 섹션

권장 제목:
`이런 팀이면 Fabbit이 맞습니다`

추천 카드:

- `ERP는 있는데 도면과 BOM은 엑셀, 폴더로 관리하는 팀`
- `설계변경이 잦고 협력사, 외주 커뮤니케이션이 많은 팀`
- `품번, 리비전 체계가 약해 재발주와 구버전 이슈가 반복되는 팀`

함께 노출할 비적합 대상:

- `1~9명 영세기업`
- `이미 대기업 PLM이 깊게 정착한 조직`

### 6.9 요금 / 도입 방식 요약 섹션

랜딩의 요금 섹션은 상세 가격표가 아니라 `어떻게 시작하는가`를 요약하는
역할만 맡는다.

- 핵심 메시지:
  `무료 Starter 소개는 가능하지만, 현재 기본 도입 경로는 파일럿과 데모다`
- 노출 수준:
  - `무료로 시작 가능한 구조`
  - `역할 기반 확장 가능성`
  - `상세 가격과 비교는 요금제 페이지에서 확인`
- CTA:
  - `요금제 자세히 보기`
  - `무료 파일럿 신청`

### 6.10 FAQ 섹션

초기 FAQ는 아래 질문을 우선 다룬다.

1. `ERP를 이미 쓰고 있어도 필요한가요?`
2. `PLM이나 MES를 몰라도 쓸 수 있나요?`
3. `어떤 파일 형식을 다루나요?`
4. `무료 파일럿은 어떻게 진행되나요?`
5. `생산 기능은 지금 어디까지 가능한가요?`

### 6.11 파일럿 검증 / 사례 섹션

공개 고객 사례가 충분하지 않은 동안에는 고객 로고보다 `어떤 팀을 먼저
검증하는가`를 정직하게 보여주는 편이 낫다.

- 권장 제목:
  `이런 제조팀부터 먼저 검증합니다`
- 권장 카드:
  - `금속가공`
  - `기계장비`
  - `전기장비`
- 사례 확보 이후 교체 항목:
  - 고객 로고
  - 전후 수치
  - 짧은 인용문

### 6.12 마지막 CTA 섹션

- 제목:
  `엑셀과 폴더로 관리하던 도면·BOM을 한 번에 정리해보세요`
- 기본 CTA:
  `무료 파일럿 신청`
- 보조 CTA:
  `제품 데모 요청`
- 보조 문구:
  `10~49명 제조팀, ERP 사용 중, 도면과 BOM을 엑셀로 관리 중이라면 우선 대상입니다`

## 9. 콘텐츠 기획

### 필수 자산

- AI 도면 인식 결과 스크린샷
- BOM 트리, 부품 탐색 스크린샷
- 변경 요청 또는 이슈 화면 스크린샷
- Before/After 흐름 다이어그램
- 파일럿 대상과 공개 범위를 설명하는 카드

### 있으면 좋은 자산

- 60~90초 제품 데모 영상
- 대표자 한 줄 신뢰 문구
- 인터뷰 기반 고객 언어 1~2개
- 추후 파일럿 확보 후 고객 로고와 전후 수치

### 카피 규칙

- 첫 화면에서는 `PLM`, `MES`보다 `도면`, `BOM`, `변경`, `생산`을 앞세운다.
- `AI가 다 해준다` 같은 과장 표현은 피한다.
- `온톨로지`, `Agentic AI`는 비전 설명에서만 제한적으로 쓴다.
- `뽑아주기`보다 `뽑은 뒤에도 같은 시스템에서 관리`를 강조한다.

## 10. 정책과 예외

- 실제 구현 전 기능은 반드시 `준비 중`, `다음 단계`, `로드맵`으로 구분한다.
- 고객사 로고, 후기, 절감 수치는 확보 전까지 임의로 넣지 않는다.
- `무료 가입`을 전면 CTA로 두지 않는다.
- 가격과 구독 정책은 랜딩 안에서 다시 정의하지 않는다.
- 외부 레퍼런스 사이트는 섹션 리듬만 참고하고 문구 구조를 직접 복제하지 않는다.
