# 요금제 페이지 기획

이 문서는 Fabbit 대외 사이트의 요금제 페이지를 정의한다. 요금제 페이지는
랜딩 페이지에서 문제 공감과 제품 소개를 마친 방문자가, 실제 도입 방식과
플랜 차이를 이해한 뒤 `가입`으로 전환하도록 돕는 역할을 맡는다. 파일럿
신청은 메인 전환을 보완하는 보조 문의 경로로만 둔다.

가격, 플랜 한도, 스토리지, AI 과금 정책은 사업 문서가 아니라 서버 구현을
기준으로 해석한다. 현재 기준 소스는 `WorkspacePlanType`과 공개 API 스키마
`/api/v1/auth/plans`다.

## 1. 페이지 목적

요금제 페이지는 가격만 보여주는 화면이 아니다. Fabbit의 과금 단위, 플랜별
권장 대상, 현재 제공 범위, 도입 절차를 명확히 설명해 `이 팀이 지금 어떤
경로로 시작해야 하는가`를 판단하게 만든다.

- `Starter 무료`를 진입점으로, 유료 플랜은 `플랜 + 좌석` 구조로 운영된다는
  점을 이해시킨다.
- `Viewer`, `Collaborator`, `Full` 좌석이 역할 기반으로 나뉜다는 점을
  빠르게 이해시킨다.
- `Team`, `Organization`, `Enterprise`의 스토리지와 AI 정책 차이를 비교할 수 있게
  만든다.
- 전체 요금과 과금 규칙을 공개하고, 요금 계산기로 예상 비용을 바로 계산하게
  한다.
- 메인 CTA가 `가입`임을 분명히 하고, `파일럿 신청`은 보조 문의로만 둔다.
- 랜딩에서 해소되지 않은 가격, 도입, 기능 범위 질문을 FAQ로 마무리한다.

## 2. 랜딩 페이지와 역할 분리

랜딩 페이지와 요금제 페이지는 방문자의 질문 단계가 다르다. 랜딩이
`왜 필요한가`를 설명한다면, 요금제 페이지는 `어떻게 시작하고 얼마까지
공개되는가`를 설명한다.

| 구분 | 랜딩 페이지 | 요금제 페이지 |
|------|-------------|----------------|
| 핵심 질문 | 왜 Fabbit이 필요한가 | 우리 팀은 어떤 플랜으로 어떻게 시작하나 |
| 주된 역할 | 문제 제시, 제품 이해, 신뢰 형성 | 가격 이해, 플랜 비교, 도입 판단 |
| CTA | 가입, 파일럿 신청 | 가입, 파일럿 신청 |
| 정보 깊이 | 요약형 | 비교형, 의사결정형 |

## 3. 진입 경로

요금제 페이지는 독립 URL을 가지되, 랜딩과 자연스럽게 연결되어야 한다.
진입 의도가 상대적으로 뜨거운 방문자를 받는 페이지이므로, 헤더와 하단 CTA
모두에서 접근할 수 있어야 한다.

- 랜딩 헤더의 `요금` 메뉴
- 랜딩 `요금 / 도입 방식` 섹션의 자세히 보기 링크
- 소개 메일, 영업 자료, 제안서 내 직접 링크
- 검색 유입 대비 메타 타이틀과 요약 카피 별도 구성

## 4. 핵심 메시지

페이지 전체는 가격표보다 과금 구조를 먼저 이해시키는 순서가 낫다. 방문자가
`왜 플랜만이 아니라 좌석 타입까지 보아야 하는가`를 이해해야 가격도 정확히
읽힌다.

- 메인 메시지:
  `무료 Starter로 시작하고, 역할별 좌석으로 확장하는 제조 데이터 워크스페이스`
- 서브 메시지:
  `Fabbit은 플랜 위에 Viewer, Collaborator, Full 좌석을 조합해 팀 운영
  방식에 맞는 과금 구조를 만듭니다`
- 도입 메시지:
  `Starter는 바로 가입해 시작하고, 파일럿이 필요한 팀은 보조 문의 경로를
  통해 운영 범위를 함께 확인합니다`

## 5. 권장 화면 구조

요금제 페이지는 아래 순서로 구성한다. 방문자가 위에서 아래로 읽으며 가격,
대상, 도입 방식, FAQ를 한 번에 닫을 수 있어야 한다.

### 5.1 히어로

첫 화면에서는 숫자보다 구조를 먼저 제시한다. `무료로 시작할 수 있지만,
유료 플랜은 좌석 단위로 확장되는 서비스`라는 인상을 만들어야 한다.

- 제목:
  `무료로 시작하고, 역할별 좌석으로 확장하는 제조 데이터 워크스페이스`
- 보조 문구:
  `Starter는 최대 5명까지 무료로 시작하고, Team·Organization·Enterprise는
  Viewer, Collaborator, Full 좌석 조합으로 운영 범위를 확장합니다`
- 기본 CTA:
  `가입`
- 보조 CTA:
  `파일럿 신청`
- 보조 배지:
  `Starter 무료`, `좌석 기반 과금`, `전체 요금 공개`

### 5.2 플랜 카드

플랜 카드는 `플랜 이름`보다 `과금 방식`이 먼저 보이게 구성해야 한다. 각 카드에는
좌석 단가, 스토리지 정책, AI 정책, 활성 여부, 권장 CTA를 담고 세부 기능 범위는
비교표, 계산기, FAQ로 넘긴다.

| 플랜 | 공개 가격 구조 | 멤버 정책 | 스토리지 정책 | AI 정책 | 노출 상태 | CTA |
|------|----------------|-----------|---------------|---------|-----------|-----|
| Starter | 무료 | 최대 5명 | 총 250MB | 월 100 크레딧 포함, 포함형만 사용 | 활성 | 가입 |
| Team | Viewer 5,000원 / Collaborator 15,000원 / Full 29,000원 | 서버 기준 상한 없음 | 기본 10GB + Full 좌석당 10GB | 사용량 기반 과금 | 활성 | 가입 |
| Organization | Viewer 5,000원 / Collaborator 25,000원 / Full 59,000원 | 서버 기준 상한 없음 | 기본 100GB + Full 좌석당 50GB | 사용량 기반 과금 | 비활성 노출 | 준비 중 |
| Enterprise | 현재 서버 기준 Organization과 동일 좌석 단가 | 서버 기준 상한 없음 | 기본 100GB + Full 좌석당 50GB | 사용량 기반 과금 | 비활성 노출 | 문의 예정 |

플랜 카드 하단에는 아래 보조 문구를 함께 둔다.

`Organization`과 `Enterprise`는 가격과 과금 구조는 공개하지만, 현재 공개
사이트에서는
비활성 상태로만 노출합니다.

### 5.3 요금 계산기

요금제 페이지에는 방문자가 좌석 수와 스토리지 가정을 넣고 예상 월 비용을 바로
볼 수 있는 요금 계산기를 포함한다. 가격이 모두 공개되더라도 실제 팀 구성에
따라 체감 비용이 달라지므로, 계산기는 공개 정보의 이해를 돕는 핵심 블록이다.

- 입력 항목:
  플랜 선택, Viewer 수, Collaborator 수, Full 수, 예상 추가 스토리지
- 출력 항목:
  월 예상 좌석 비용, 월 예상 스토리지 초과 비용, 총 월 예상 비용
- 비활성 동작:
  `Organization`, `Enterprise`를 선택하면 계산은 가능하지만 가입 CTA는
  비활성으로 유지
- 보조 문구:
  `실제 청구는 워크스페이스 구성과 사용량에 따라 달라질 수 있습니다`

### 5.4 과금 철학 설명 블록

가격 바로 아래에는 `왜 플랜만이 아니라 좌석 타입을 함께 보여주는가`를 짧고
명확하게 설명한다. 이 블록은 방문자가 가격표를 오해하지 않게 만드는 핵심
장치다.

- Starter는 최대 5명까지 무료로 시작하는 진입 플랜
- 유료 플랜은 `Viewer`, `Collaborator`, `Full` 좌석 단가를 조합하는 구조
- 스토리지는 플랜 기본 제공량과 `Full` 좌석 수에 따라 늘어나는 구조
- AI는 Starter만 월 100 크레딧 포함, 나머지는 사용량 기반 과금
- 스토리지 초과 과금은 1GB당 월 200원

### 5.5 비교표

비교표는 카드에서 못 보여준 기준을 정리한다. 다만 공개 정보와 상담 전환
정보는 분리해, 초기 페이지가 과도하게 복잡해지지 않게 한다.

권장 비교 항목은 아래와 같다.

- 플랜 타입
- Starter 최대 멤버 수
- 유료 플랜 좌석 단가
- 기본 스토리지
- `Full` 좌석당 추가 스토리지
- 스토리지 초과 과금 단가
- AI 과금 방식
- Starter 포함 AI 크레딧
- 권장 도입 방식
- 활성 여부

비교표 하단 보조 문구는 아래 톤을 유지한다.

`현재 공개 페이지에서는 플랜, 좌석 단가, 스토리지, AI 정책처럼 시작 판단에
필요한 기준과 예상 비용 계산기를 함께 안내합니다. 실제 좌석 조합과 추후
업그레이드 경로는 앱 내부 구독 흐름에서 이어집니다.`

### 5.6 도입 방식 섹션

요금제 페이지에서는 `어떻게 시작하는가`를 짧고 분명하게 보여줘야 한다.
결제 기능 구현 여부와 별개로, 공개 사이트의 메인 행동은 가입이라는 점을
오해 없이 노출해야 한다.

권장 3단계는 아래와 같다.

1. `가입`
2. `워크스페이스 생성과 플랜 선택`
3. `Starter 시작 또는 Team 선택`

보조 설명은 아래 수준으로 유지한다.

- Starter는 상시 무료 진입 플랜
- Team은 좌석 타입별 비용을 확인한 뒤 선택하는 유료 플랜
- 파일럿이 필요한 팀은 보조 `파일럿 신청` 경로로 문의한다
- `Organization`, `Enterprise`는 현재 노출만 하고 가입 CTA는 비활성으로 둔다

### 5.7 FAQ

FAQ는 가격 자체보다 도입 저항을 줄이는 질문 중심으로 구성한다. 초기 공개
질문은 아래 항목을 우선한다.

1. `ERP를 이미 쓰고 있어도 필요한가요?`
2. `PLM이나 MES를 몰라도 쓸 수 있나요?`
3. `Starter 무료 플랜으로 어디까지 사용할 수 있나요?`
4. `유료 플랜은 사람 수가 아니라 좌석 타입으로 과금되나요?`
5. `스토리지는 어떻게 계산되고 언제 추가 과금이 붙나요?`
6. `AI 사용량은 어떤 방식으로 과금되나요?`
7. `요금 계산기 금액과 실제 청구 금액은 왜 다를 수 있나요?`
8. `파일럿 신청은 어떤 경우에 필요한가요?`
9. `Organization`과 `Enterprise`는 왜 보이지만 바로 가입할 수 없나요?`

### 5.8 마지막 CTA

페이지 끝에서는 플랜 비교를 마친 방문자가 바로 행동할 수 있게 CTA를 다시
닫는다. 여기서는 `가입`을 메인 CTA로 유지하고, 파일럿 신청은 보조 문의로만
남긴다.

- 제목:
  `우리 팀에 맞는 시작 플랜을 바로 확인해보세요`
- 기본 CTA:
  `가입`
- 보조 CTA:
  `파일럿 신청`
- `Organization`, `Enterprise` 보조 링크:
  `준비 중`
- 보조 문구:
  `Starter와 Team은 지금 구조를 확인하고 가입할 수 있으며, 파일럿이 필요한
  팀은 별도 문의로 이어집니다`

## 6. 공개 노출 수준 기준

요금제 페이지는 도입 판단에 필요한 과금 정보를 모두 공개한다. 공개 가격 구조와
비활성 노출 범위를 함께 보여줘야 사용자 기대치가 맞는다.

### 공개

아래 항목은 페이지에서 바로 공개한다.

- 플랜명
- 좌석 단가
- Starter 최대 멤버 수
- 기본 스토리지
- `Full` 좌석당 추가 스토리지
- AI 과금 방식
- 스토리지 초과 단가
- 요금 계산기 입력 기준
- 활성/비활성 노출 상태
- 기본 CTA와 비활성 CTA 상태

### 상담 전환

아래 항목은 상담 과정에서 안내하는 편이 낫다.

- 실제 좌석 조합 추천
- 맞춤 구축 범위
- 보안 검토와 운영 정책
- 마이그레이션 지원 범위
- 월간 AI 한도 운영 정책

## 7. 카피 원칙

요금제 페이지는 `정액 플랜 비교`보다 `구조가 명확하다`에 초점을 둔다. 카피는
아래 원칙을 유지한다.

- 메인 CTA는 `가입`으로 쓴다.
- `파일럿 신청`은 보조 문의 경로로만 쓴다.
- `데모 요청`은 쓰지 않는다.
- `회사 단위 정액`, `Team 24.9만원` 같은 구식 표현은 쓰지 않는다.
- 서버 enum과 별개로 공개 문서 표기는 `Organization`, `Enterprise`로 통일한다.
- `PLM`, `MES` 같은 용어는 보조 설명에서만 제한적으로 쓴다.
- 가격 숫자만 강조하지 말고 좌석 타입, 스토리지 규칙, 비활성 상태를 함께 쓴다.

웹 구현에 필요한 최소 IA는 아래 정도면 충분하다. 구현 단계에서는 랜딩의
`요금` 섹션과 CTA 문구를 이 문서 기준으로 맞춘다.

- `/pricing` 독립 라우트
- 히어로
- 플랜 카드 4종
- 요금 계산기
- 비교표
- 도입 방식 섹션
- FAQ 아코디언
- 마지막 CTA

## 9. 남은 모호점

현재 문서들만 기준으로 보면 아래 항목은 아직 확정이 아니다. 실제 웹 구현 전
추가 판단이 필요하다.

- 요금제 페이지를 랜딩 헤더의 `요금` 앵커와 병행할지, 독립 URL만 둘지
- `가입` CTA가 앱의 어느 구체 라우트로 이동할지
- `파일럿 신청`을 모달로 열지 별도 페이지로 열지
