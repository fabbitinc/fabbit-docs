# 사이트 신청/전환 흐름 기획

이 문서는 Fabbit 공개 사이트의 신청 전환 경험을 정의한다. 범위는 공개
사이트의 CTA 진입점, 파일럿과 데모 공용 신청 폼, 완료 화면, 그리고 공개
경험 기준의 전환 흐름까지다.

이 문서는 공개 신청 경험만 다룬다. 내부 리드 등급 산정, CRM 처리, 앱 회원가입,
플랜 선택, 업그레이드, 웹 상태 머신 같은 운영 또는 구현 명세는 이 문서의
범위에 포함하지 않는다.

## 1. 문서 역할과 범위

공개 사이트의 목적은 셀프서브 가입 확대가 아니라 자격 있는 제조업 리드를
파일럿, 데모, 상담으로 연결하는 것이다. 따라서 이 문서는 `가입`보다 `신청`
경험을 기준으로 설계한다.

이 문서가 맡는 범위는 아래와 같다.

- 랜딩과 요금제 페이지의 CTA 진입점
- 파일럿, 데모 공용 신청 폼 구조
- 신청 완료 후 유저에게 보이는 상태와 문구
- 공개 사이트 기준의 최소 전환 흐름

이 문서가 맡지 않는 범위는 아래와 같다.

- 내부 리드 등급 산정 규칙
- 운영팀 후속 액션과 CRM 처리
- 앱 회원가입, 워크스페이스 생성, 플랜 선택
- 업그레이드와 구독 변경 처리
- `web` 상태 키, 이벤트 스키마, 데이터 모델 상세

앱 내부 진입 이후의 흐름은 [[10_제품/기획/app/index]]에서 다룬다.

## 2. 목표와 원칙

사이트 신청 흐름은 랜딩의 전환 효율과 운영 효율을 동시에 고려해야 한다.
다만 이 문서는 운영팀 내부 처리보다 공개 신청 경험 자체를 먼저 정리한다.

신청 흐름의 목표는 아래와 같다.

1. 방문자가 CTA를 눌렀을 때 무엇을 신청하는지 바로 이해하게 한다.
2. 리드 선별에 필요한 최소 정보만 받아 이탈을 낮춘다.
3. 제출 직후 다음 연락 방식이 무엇인지 명확히 안내한다.
4. 아직 없는 셀프서브 결제나 앱 내부 가입 흐름을 암시하지 않는다.

전환 설계 원칙은 아래와 같다.

- 1차 CTA는 `무료 파일럿 신청`이다.
- 2차 CTA는 `제품 데모 요청`이다.
- 두 CTA는 하나의 공용 신청 폼으로 연결한다.
- 폼은 `2단계` 구성을 기본안으로 둔다.
- 제출 자체는 막지 않고, 후속 검토는 별도 운영 문서에서 다룬다.
- 완료 화면은 신청 의도에 맞는 다음 연락 방식을 바로 알려준다.

## 3. CTA 진입점

공개 사이트에서 관리해야 할 주요 CTA 진입점은 아래와 같다. 이 진입점들은
모두 같은 폼으로 연결되지만, 시작 의도는 `pilot`과 `demo`로 구분한다.

| 위치 | CTA | request_type | 의도 |
|------|-----|--------------|------|
| 헤더 우측 | 무료 파일럿 신청 | `pilot` | 자격 있는 리드의 즉시 전환 |
| 히어로 기본 CTA | 무료 파일럿 신청 | `pilot` | 메인 전환 |
| 히어로 보조 CTA | 제품 데모 요청 | `demo` | 확신 부족 리드의 저마찰 전환 |
| 대표 시나리오 스포트라이트 | 제품 데모 요청 | `demo` | 제품 화면 관심 리드 전환 |
| 마지막 CTA 기본 버튼 | 무료 파일럿 신청 | `pilot` | 스크롤 완료 후 재전환 |
| 마지막 CTA 보조 버튼 | 제품 데모 요청 | `demo` | 비교 검토형 리드 전환 |

## 4. 공개 전환 흐름

기본 전환 흐름은 방문자가 CTA를 누른 뒤 공용 신청 폼에 진입하고, 제출 이후
완료 안내를 보는 구조다. 운영팀의 내부 검토와 리드 등급 산정은 이 문서의
공개 범위 밖에 둔다.

```text
검색 / 커뮤니티 / 소개 링크 유입
              ↓
        랜딩 또는 요금제 페이지 탐색
              ↓
     CTA 클릭 (`pilot` 또는 `demo`)
              ↓
        공용 신청 폼 진입
              ↓
   1단계: 회사와 적합도 입력
              ↓
   2단계: 운영 문제와 연락처 입력
              ↓
            제출
              ↓
      신청 완료 화면 노출
              ↓
      운영팀 후속 연락
```

이 흐름에서 중요한 점은 아래와 같다.

- `pilot`과 `demo`는 같은 폼을 사용한다.
- 신청 자체는 모든 리드에 열어둔다.
- 공개 화면은 신청 완료와 다음 연락 방식까지만 설명한다.
- 내부 분류, 후속 운영, 앱 진입 흐름은 별도 문서로 넘긴다.

## 5. 폼 구조

폼은 2단계 구성을 기본안으로 둔다. 초기 목표가 정보 최대 수집이 아니라
전환 확보와 최소 선별이기 때문이다.

### 5.1 1단계: 회사와 적합도

1단계에서는 회사와 세그먼트를 빠르게 확인할 수 있는 항목을 먼저 받는다.

| 필드 | 형식 | 목적 | 비고 |
|------|------|------|------|
| 회사명 | 텍스트 | 리드 식별 | 필수 |
| 업종 | 단일 선택 | `C25/C29/C28` 우선 판별 | 필수 |
| 직원 수 | 단일 선택 | 기본 세그먼트 구분 | 필수 |
| 역할 | 단일 선택 | 대표, 설계, 생산, 구매, 품질 맥락 파악 | 필수 |

업종 옵션의 기본안은 아래와 같다.

- 금속가공
- 기계장비
- 전기장비
- 기타 제조업
- 기타

직원 수 옵션의 기본안은 아래와 같다.

- 1~9명
- 10~49명
- 50~299명
- 300명 이상

### 5.2 2단계: 현재 운영 방식과 연락처

2단계에서는 현재 운영 문제와 후속 연락에 필요한 항목을 받는다.

| 필드 | 형식 | 목적 | 비고 |
|------|------|------|------|
| 현재 관리 방식 | 다중 선택 | 엑셀, 폴더, ERP, 기타 현재 상태 파악 | 필수 |
| ERP 사용 여부 | 단일 선택 | 업스트림 포지셔닝 확인 | 필수 |
| 가장 큰 문제 | 단일 선택 | 대표 pain 확인 | 필수 |
| 담당자명 | 텍스트 | 후속 연락 식별 | 가정 |
| 이메일 | 이메일 | 후속 연락 | 가정 |
| 전화번호 | 텍스트 | 빠른 연락 | 가정 |

현재 관리 방식 옵션의 기본안은 아래와 같다.

- 엑셀
- 폴더
- ERP
- 기타

가장 큰 문제 옵션의 기본안은 아래와 같다.

- BOM 수동 입력
- 도면 검색/버전 혼선
- 설계 변경 반영 지연
- 생산 연결/피드백 단절

## 6. request_type별 시작 카피

공용 폼이라도 상단 카피는 CTA 의도에 맞게 바꾼다. 방문자가 누른 버튼과
폼 제목이 어긋나면 이탈이 커지기 때문이다.

| request_type | 상단 제목 | 보조 설명 |
|--------------|-----------|-----------|
| `pilot` | 무료 파일럿 신청 | 파일럿 적합 여부를 빠르게 확인하고 다음 미팅을 준비하기 위해 필요한 정보만 받는다. |
| `demo` | 제품 데모 요청 | 회사 상황에 맞는 데모를 준비하기 위해 필요한 정보만 받는다. |

폼 우측 또는 하단에는 파일럿 우선 대상 안내를 함께 둔다.

- 10~49명 제조팀
- ERP는 있거나 도입 예정
- 도면과 BOM을 엑셀 또는 폴더로 관리하는 팀

## 7. 완료 경험

제출 완료 후 유저에게는 신청 의도에 맞는 다음 연락 방식을 명확히 보여준다.
내부 등급 산정 결과를 공개 화면에 직접 반영하는 것은 이 문서의 기본 범위로
두지 않는다.

| 상태 | 노출 조건 | 유저 메시지 방향 |
|------|-----------|------------------|
| `submitted_pilot` | `request_type=pilot` 제출 완료 | 파일럿 적합 여부를 확인한 뒤 연락 |
| `submitted_demo` | `request_type=demo` 제출 완료 | 데모 준비 후 연락 |
| `submitted_general` | request 구분 없이 공통 완료 화면 사용 시 | 검토 후 연락 |

유저 노출 문구의 기본안은 아래와 같다.

- `submitted_pilot`:
  `신청이 접수되었습니다. 파일럿 적합 여부를 확인한 뒤 연락드리겠습니다.`
- `submitted_demo`:
  `신청이 접수되었습니다. 회사 상황에 맞는 데모 일정을 준비해 연락드리겠습니다.`
- `submitted_general`:
  `신청이 접수되었습니다. 검토 후 적절한 방식으로 연락드리겠습니다.`

## 8. 공개 경험과 내부 운영의 경계

신청 문서는 공개 경험만 정의해야 한다. 아래 항목은 필요하더라도 다른 문서에서
다루는 편이 맞다.

- `A/B/C` 리드 분류 규칙
- 내부 후속 액션과 CRM 상태
- 앱 회원가입으로의 자동 전환 여부
- `web` 상태 키와 이벤트 스키마
- 구독 생성이나 플랜 선택 로직

즉, 이 문서는 `CTA를 눌러 어떤 폼을 보고 어떤 완료 화면을 보게 되는가`까지만
책임진다.

## 9. 정책

사이트 신청 흐름에서는 아래 정책을 지킨다.

- `가입`, `결제`를 전면 CTA로 쓰지 않는다.
- 파일럿과 데모는 분리 페이지보다 공용 폼을 우선한다.
- 아직 구현되지 않은 생산 기능은 `준비 중`, `로드맵`으로 분리한다.
- 신청 차단보다 신청 후 검토 방식을 택한다.
- 공개 폼 경험이 앱 내부 가입 흐름으로 오해되지 않게 한다.

## 10. MVP 범위

초기 신청/전환 흐름 MVP는 아래까지만 포함한다.

- 랜딩과 요금제 페이지의 반복 CTA
- 파일럿, 데모 공용 신청 폼 1개
- 2단계 입력
- 제출 완료 화면
- 최소 수준의 전환 추적

초기 제외 범위는 아래와 같다.

- 셀프서브 가입
- 셀프서브 결제
- 업종별 별도 신청 폼
- 앱 온보딩으로의 자동 분기

## 11. 남은 모호점

현재 문서만으로는 바로 결정하기 어려운 지점들은 아래와 같다.

- `연락처`의 실제 필수 필드가 확정되지 않았다.
- 폼 UI가 모달인지 별도 페이지인지 기준선이 없다.
- 완료 화면을 request type별로 나눌지 공통으로 둘지 확정되지 않았다.
- `ERP 사용 여부` 옵션 체계가 `있음`, `없음`, `도입 예정`인지 명확하지 않다.
