# 사이트 신청/전환 흐름 기획

이 문서는 Fabbit 공개 사이트의 신청 전환 기준선을 정의한다. 범위는 공개
사이트의 CTA 진입점, 파일럿/데모 공용 신청 폼, 리드 분기, 완료 상태,
후속 액션, 그리고 `web` 구현에 필요한 최소 상태와 이벤트 정의다.

현재 기준에서 사이트의 목적은 셀프서브 가입 확대가 아니라, 자격 있는 제조업
리드를 선별해 파일럿 또는 데모로 연결하는 것이다. 따라서 전환 흐름도
`가입`보다 `신청`, `데모`, `상담` 중심으로 설계한다.

> [!info] 기본안
> 이 문서는 기존 랜딩페이지 기획을 바탕으로 공용 신청 폼을 기본안으로 둔다.
> `무료 파일럿 신청`과 `제품 데모 요청`은 서로 다른 페이지로 분리하지 않고,
> 동일한 폼에서 시작 상태와 후속 액션만 다르게 둔다.

## 1. 목표와 원칙

사이트 신청 흐름은 랜딩의 전환 효율과 운영 효율을 동시에 만족해야 한다.
따라서 문서상 목표도 단순 제출 수보다 적합 리드 선별과 후속 연결에 둔다.

신청 흐름의 목표는 아래와 같다.

1. 방문자가 CTA를 눌렀을 때 무엇을 신청하는지 바로 이해하게 한다.
2. 리드 선별에 필요한 최소 정보만 받아 이탈을 낮춘다.
3. 제출 직후 운영팀이 파일럿과 데모 대상을 구분할 수 있게 한다.
4. 아직 없는 셀프서브 결제나 운영 중이 아닌 생산 기능을 암시하지 않는다.

전환 설계 원칙은 아래와 같다.

- 1차 CTA는 `무료 파일럿 신청`이다.
- 2차 CTA는 `제품 데모 요청`이다.
- 두 CTA는 하나의 공용 신청 폼으로 연결한다.
- 폼은 `2단계`를 기본안으로 둔다.
- 완료 후 내부적으로 `A/B/C` 리드 분류를 남긴다.
- 유저에게는 신청 완료와 다음 연락 방식을 명확히 안내한다.

> [!info] 가정
> 기존 문서의 `연락처`는 웹 구현 기준으로 `담당자명`, `이메일`, `전화번호`
> 세 필드로 분해해 다루는 안을 기본으로 둔다. 원문 문서에는 `연락처`만
> 정의되어 있으므로, 필수 여부는 구현 전에 확정이 필요하다.

## 2. CTA 진입점

공개 사이트에서 관리해야 할 주요 CTA 진입점은 아래와 같다. 이 진입점들은
모두 같은 폼으로 연결되지만, 시작 의도와 추적 이벤트는 분리해서 본다.

| 위치 | CTA | request_type | 의도 |
|------|-----|--------------|------|
| 헤더 우측 | 무료 파일럿 신청 | `pilot` | 자격 있는 리드의 즉시 전환 |
| 히어로 기본 CTA | 무료 파일럿 신청 | `pilot` | 메인 전환 |
| 히어로 보조 CTA | 제품 데모 요청 | `demo` | 확신 부족 리드의 저마찰 전환 |
| 대표 시나리오 스포트라이트 | 제품 데모 요청 | `demo` | 제품 화면 관심 리드 전환 |
| 마지막 CTA 기본 버튼 | 무료 파일럿 신청 | `pilot` | 스크롤 완료 후 재전환 |
| 마지막 CTA 보조 버튼 | 제품 데모 요청 | `demo` | 비교 검토형 리드 전환 |

모든 CTA는 아래 메타데이터를 함께 넘긴다.

- `request_type`: `pilot` 또는 `demo`
- `entry_point`: `header`, `hero_primary`, `hero_secondary`,
  `spotlight_demo`, `final_primary`, `final_secondary`
- `entry_page`: `site_landing`

## 3. 전환 흐름

기본 전환 흐름은 방문자가 CTA를 누른 뒤 공용 신청 폼에 진입하고, 제출 이후
내부 분류와 후속 액션으로 이어지는 구조다. 분기 기준은 제출 전 차단보다
제출 후 운영 처리에 둔다.

```text
검색 / 커뮤니티 / 소개 링크 유입
              ↓
        랜딩 페이지 스크롤
              ↓
     CTA 클릭 (`pilot` 또는 `demo`)
              ↓
        공용 신청 폼 진입
              ↓
   1단계: 회사/대상 적합도 입력
              ↓
   2단계: 운영 문제/연락처 입력
              ↓
            제출
              ↓
      리드 자격 분류 (`A/B/C`)
        ┌─────────────┼─────────────┐
        ↓             ↓             ↓
       A              B             C
  파일럿 미팅        데모 우선      자료 발송 후 nurture
  24시간 내 연락      일정 제안      후속 보류
```

이 흐름에서 중요한 점은 아래와 같다.

- `pilot`과 `demo`는 같은 폼을 사용한다.
- 폼 제출 자체는 모든 리드에 열어둔다.
- 분기는 제출 전 차단이 아니라 제출 후 분류로 처리한다.
- 랜딩의 메시지와 완료 화면 메시지는 서로 모순되면 안 된다.

## 4. 폼 구조

폼은 2단계 구성을 기본안으로 둔다. 초기 목표가 정보 최대 수집이 아니라
전환 확보와 최소 선별이기 때문이다. 단계별 피로도를 낮추면서도 운영팀이
후속 액션을 판단할 최소 데이터를 확보해야 한다.

### 4.1 1단계: 회사와 적합도

1단계에서는 리드 적합도를 빠르게 판단할 수 있는 항목을 먼저 받는다. 회사와
세그먼트를 먼저 확인해야 이후 후속 액션을 무리 없이 분기할 수 있다.

| 필드 | 형식 | 목적 | 비고 |
|------|------|------|------|
| 회사명 | 텍스트 | 계정/리드 식별 | 필수 |
| 업종 | 단일 선택 | `C25/C29/C28` 우선 판별 | 필수 |
| 직원 수 | 단일 선택 | `10~49`, `50~299`, `1~9` 분기 | 필수 |
| 역할 | 단일 선택 | 대표/설계/생산/구매/품질 맥락 파악 | 필수 |

업종 옵션의 기본안은 아래와 같다.

- 금속가공
- 기계장비
- 전기장비
- 기타 제조업
- 기타

직원 수 옵션의 기본안은 아래와 같다.

- 1~9명
- 10~49명
- 50~299명
- 300명 이상

### 4.2 2단계: 현재 운영 방식과 연락처

2단계에서는 현재 운영 문제와 후속 연락에 필요한 항목을 받는다. 리드의 실제
pain과 연락 가능성을 같이 확보해야 제출 이후 바로 대응할 수 있다.

| 필드 | 형식 | 목적 | 비고 |
|------|------|------|------|
| 현재 관리 방식 | 다중 선택 | 엑셀/폴더/ERP/기타 현재 상태 파악 | 필수 |
| ERP 사용 여부 | 단일 선택 | 업스트림 포지셔닝 판별 | 필수 |
| 가장 큰 문제 | 단일 선택 | pain 분류 | 필수 |
| 담당자명 | 텍스트 | 후속 연락 식별 | 가정 |
| 이메일 | 이메일 | 후속 연락 | 가정 |
| 전화번호 | 텍스트 | 빠른 연락 | 가정 |

현재 관리 방식 옵션의 기본안은 아래와 같다.

- 엑셀
- 폴더
- ERP
- 기타

가장 큰 문제 옵션의 기본안은 아래와 같다.

- BOM 수동 입력
- 도면 검색/버전 혼선
- 설계 변경 반영 지연
- 생산 연결/피드백 단절

### 4.3 request_type에 따른 시작 카피

공용 폼이라도 상단 카피는 CTA 의도에 맞게 바꾼다. 방문자가 누른 버튼과
폼 제목이 어긋나면 이탈이 커지기 때문에, request type에 따라 문맥을 맞춘다.

| request_type | 상단 제목 | 보조 설명 |
|--------------|-----------|-----------|
| `pilot` | 무료 파일럿 신청 | 파일럿 적합 여부를 빠르게 확인하고 다음 미팅을 준비하기 위해 필요한 정보만 받는다. |
| `demo` | 제품 데모 요청 | 회사 상황에 맞는 데모를 준비하기 위해 필요한 정보만 받는다. |

폼 우측 또는 하단에는 파일럿 우선 대상 안내를 함께 둔다.

- 10~49명 제조팀
- ERP는 있거나 도입 예정
- 도면/BOM을 엑셀 또는 폴더로 관리

## 5. 분기와 리드 분류

리드 분류는 제출 후 내부적으로 계산한다. 이 분류는 신청 가능 여부를 막는
장치가 아니라 운영 우선순위를 정하는 장치다.

| 등급 | 기준 | 내부 후속 액션 |
|------|------|----------------|
| A | 10~299명, 도면/BOM 엑셀 관리, ERP 있음 또는 예정 | 24시간 내 파일럿 미팅 |
| B | 타깃 업종이나 규모/상황이 일부 애매함 | 데모 우선 |
| C | 1~9명 또는 비타깃 업종 | 자료 발송 후 nurture |

분기 규칙의 기본안은 아래와 같다.

1. 직원 수가 `1~9명`이면 우선 `C` 후보로 둔다.
2. 업종이 `금속가공`, `기계장비`, `전기장비` 중 하나면 타깃 업종으로 본다.
3. 현재 관리 방식에 `엑셀` 또는 `폴더`가 포함되면 pain 적합도가 높다고 본다.
4. ERP가 `있음` 또는 `도입 예정`이면 상위 적합도로 본다.
5. 위 조건이 대부분 맞으면 `A`, 일부만 맞으면 `B`, 크게 벗어나면 `C`로 둔다.

> [!warning] 운영 규칙
> 분기 결과는 유저의 제출 가능 여부를 막는 용도가 아니다. 부적합 리드도
> 제출은 가능하게 두고, 완료 후 안내와 내부 후속 액션만 다르게 처리한다.

## 6. 완료 상태

제출 완료 후 유저에게 보여줄 상태는 아래 3종을 기준으로 둔다. 기본적으로는
신청 의도에 맞는 완료 메시지를 먼저 보여주고, 필요 시 내부 분류 결과를 더
구체적인 후속 메시지로 연결할 수 있다.

| 상태 | 노출 조건 | 유저 메시지 방향 | 내부 처리 |
|------|-----------|------------------|-----------|
| `submitted_pilot` | `request_type=pilot` 제출 완료 | 파일럿 적합 여부를 확인해 연락 | 리드 생성 |
| `submitted_demo` | `request_type=demo` 제출 완료 | 데모 준비 후 연락 | 리드 생성 |
| `submitted_general` | request 구분 없이 공통 완료 화면 사용 시 | 검토 후 연락 | 리드 생성 |

후속 분류가 UI에 바로 반영되어야 하면 아래 변형 상태를 쓸 수 있다.

- `qualified_a`
- `qualified_b`
- `qualified_c`

유저 노출 문구의 기본안은 아래와 같다.

- `A`: `신청이 접수되었습니다. 파일럿 적합 여부를 확인한 뒤 24시간 내에
  연락드리겠습니다.`
- `B`: `신청이 접수되었습니다. 회사 상황에 맞는 데모 일정을 준비해
  연락드리겠습니다.`
- `C`: `신청이 접수되었습니다. 현재 상황에 맞는 안내 자료를 먼저 보내드리고,
  필요한 경우 후속으로 연락드리겠습니다.`

## 7. web 구현 기준 상태 정의

`web` 구현에서는 아래 상태 집합을 최소 기준선으로 삼는다. 이 상태 집합은
폼 렌더링, 검증, 제출, 완료 메시지, 내부 분류 연결을 모두 포괄해야 한다.

| 상태 키 | 의미 |
|---------|------|
| `idle` | 폼이 열리기 전 기본 상태 |
| `opened` | CTA 클릭 후 폼이 열린 상태 |
| `step_1` | 1단계 입력 중 |
| `step_2` | 2단계 입력 중 |
| `validation_error` | 현재 단계 검증 실패 |
| `submitting` | 제출 요청 전송 중 |
| `submitted` | 제출 완료 |
| `qualified_a` | 내부 분류 결과 A |
| `qualified_b` | 내부 분류 결과 B |
| `qualified_c` | 내부 분류 결과 C |
| `submit_error` | 제출 실패 |

폼 데이터의 최소 모델은 아래와 같다.

| 키 | 설명 |
|----|------|
| `request_type` | `pilot` 또는 `demo` |
| `entry_point` | CTA 위치 식별값 |
| `company_name` | 회사명 |
| `industry` | 업종 |
| `employee_count` | 직원 수 |
| `role` | 역할 |
| `management_methods` | 현재 관리 방식 목록 |
| `erp_status` | ERP 사용 여부 |
| `primary_pain` | 가장 큰 문제 |
| `contact_name` | 담당자명 |
| `contact_email` | 이메일 |
| `contact_phone` | 전화번호 |
| `lead_grade` | 내부 계산 결과 `A/B/C` |

## 8. web 구현 기준 이벤트 정의

기존 랜딩페이지 기획에서 정의된 최소 이벤트는 아래와 같다. 초기 구현은 이
이벤트 집합만으로도 CTA 클릭과 폼 제출, 적합 리드 비율을 추적할 수 있어야
한다.

| 이벤트 | 발생 시점 | 필수 속성 |
|--------|-----------|-----------|
| `hero_cta_click` | 히어로 CTA 클릭 | `request_type` |
| `problem_section_view` | 문제 섹션 노출 | 없음 |
| `feature_section_view` | 핵심 기능 섹션 노출 | 없음 |
| `pricing_section_view` | 요금 섹션 노출 | 없음 |
| `lead_form_start` | 폼 첫 진입 | `request_type`, `entry_point` |
| `lead_form_submit` | 폼 제출 완료 | `request_type`, `entry_point` |
| `qualified_lead` | 리드 분류 완료 | `lead_grade` |

반복 CTA를 운영할 계획이라면 아래 속성을 공통으로 넣는다.

- `entry_point`
- `request_type`
- `industry`
- `employee_count`
- `erp_status`
- `lead_grade`

## 9. 정책

사이트 신청 흐름에서는 아래 정책을 지킨다. 이 정책은 마케팅 카피, 화면 상태,
후속 운영 모두에 공통으로 적용된다.

- `가입`, `결제`를 전면 CTA로 쓰지 않는다.
- 파일럿과 데모는 분리 페이지보다 공용 폼을 우선한다.
- 아직 구현되지 않은 생산 기능은 `준비 중`, `로드맵`으로 분리한다.
- 고객 사례나 절감 수치는 확보 전까지 신청 유도 문구에 넣지 않는다.
- 리드 선별은 신청 차단이 아니라 제출 후 분류로 처리한다.

## 10. MVP 범위

초기 신청/전환 흐름 MVP는 아래까지만 포함한다. 이 범위를 넘는 기능은 별도
검증 이후 다음 단계로 넘긴다.

- 랜딩 내 반복 CTA
- 파일럿/데모 공용 신청 폼 1개
- 2단계 입력
- 제출 완료 화면
- A/B/C 리드 분류
- 기본 이벤트 측정

초기 제외 범위는 아래와 같다.

- 셀프서브 가입
- 셀프서브 결제
- 자료 다운로드 게이트 페이지
- 업종별 별도 신청 폼
- 다국어 신청 페이지

## 11. 남은 모호점

현재 문서만으로는 바로 결정하기 어려운 지점들이 있다. 이 항목들은 `web`
구현 전에 정책으로 한 번 더 확정해야 한다.

- `연락처`의 실제 필수 필드가 확정되지 않았다.
- 폼 UI가 모달인지 별도 페이지인지 기준선이 없다.
- `hero_cta_click` 외 CTA 클릭 이벤트 네이밍이 통일되어 있지 않다.
- `A/B/C` 분류를 완료 화면에 바로 반영할지 정해지지 않았다.
- `ERP 사용 여부` 옵션 체계가 `있음/없음/도입 예정`인지 명확하지 않다.
