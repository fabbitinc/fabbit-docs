# 변경관리 기획

이 문서는 Fabbit PLM의 변경관리 모듈을 정의한다. 변경관리는 `이슈`로
현장 문제를 접수하고, `변경요청`으로 실제 반영 작업을 추적하는 2단계
구조를 기본으로 한다.

현재 구현 기준은 `Issue`와 `ChangeRequest` 두 엔티티다. 제조 도메인에서
자주 쓰는 ECR/ECO 완전 분리는 후속 범위로 두고, MVP에서는 부품 중심의
변경 추적과 협업 이력을 먼저 고정한다.

## 1. 문제와 목표

- 설계 변경, BOM 오류, 도면 누락, 양산 불량 피드백이 메신저와 엑셀에
  흩어져 있어 어떤 문제가 어떤 변경으로 이어졌는지 추적하기 어렵다.
- 변경 요청이 생겨도 담당자, 검토자, 첨부파일, 댓글, 반영 이력이 한
  화면에 모이지 않으면 생산 기준본 변경의 책임소재가 흐려진다.
- 부품 상세에서 바로 이슈를 남기지 못하면 변경관리 모듈이 따로 노는
  도구가 되고, 현장 사용성이 급격히 떨어진다.

목표는 아래 네 가지다.

1. **이슈 접수 표준화**: 부품 중심으로 설계/품질/생산 이슈를 한 형식으로
   등록한다.
2. **변경요청 상태 가시화**: `DRAFT -> SUBMITTED -> MERGED/CLOSED`
   흐름을 화면과 서버에서 동일하게 관리한다.
3. **반영 근거 추적**: 댓글, 첨부파일, 멘션, 상태 변경, 이슈-변경요청
   링크를 타임라인으로 남긴다.
4. **부품 변경 영향 연결**: 변경요청이 어떤 부품과 어떤 이슈를
   다루는지 상세 화면에서 바로 파악한다.

## 2. 주요 사용자

| 사용자 | 주요 관심사 |
|--------|-------------|
| 설계 엔지니어 | 부품/도면/BOM 관련 이슈 등록, 변경요청 작성, 첨부파일 공유 |
| 품질 엔지니어 | LOT 불량, 검사 결과, 시정조치 요청을 이슈로 남기고 추적 |
| 생산기술 담당자 | 현장 조립성 문제, 공정 편차, 작업표준 변경 요청 접수 |
| 설계 리더 | 변경요청 검토 현황 확인, 반영 여부 판단, 상태 전이 관리 |
| PLM 관리자 | 라벨 체계, 팀 배정, 권한 정책, 감사로그 운영 |

## 3. 메뉴명과 진입 경로

| 구분 | 권장 명칭 | 비고 |
|------|-----------|------|
| 상위 메뉴 | `변경관리` | PLM 기본 모듈 |
| 하위 메뉴 1 | `이슈` | 문제 접수와 원인 기록 |
| 하위 메뉴 2 | `변경요청` | 실제 반영 작업과 검토 추적 |
| 기본 CTA | `새 이슈 등록`, `새 변경요청 등록` | 목록 화면 기준 |

진입 경로 예시:

`부품 상세 > 변경관리 > 새 이슈 등록`
`변경관리 > 이슈 > 이슈 상세 > 변경요청 생성`
`변경관리 > 변경요청 > 변경요청 상세 > 제출`
`변경관리 > 변경요청 > 변경요청 상세 > 반영`

부품 상세에서 진입한 경우, 이슈/변경요청 작성 화면은 해당 부품을 기본
연결 대상으로 미리 채운다. BOM/도면/LOT 직접 링크는 후속 범위로 둔다.

## 4. 화면 구성

| 화면 | 목적 | 주요 CTA |
|------|------|----------|
| `이슈목록` | 접수된 이슈 검색, 상태 확인, 상세 진입 | `새 이슈 등록` |
| `이슈상세` | 이슈 본문, 연결 부품, 연결 변경요청, 타임라인 확인 | `변경요청 생성`, `이슈 수정`, `닫기/다시 열기` |
| `이슈편집` | 신규 이슈 작성 및 열린 이슈 수정 | `저장` |
| `변경요청목록` | 변경요청 검색, `crState` 확인, 상세 진입 | `새 변경요청 등록` |
| `변경요청상세` | 검토 현황, 연결 이슈, 타임라인, 상태 전이 관리 | `제출`, `반영`, `닫기`, `다시 열기` |
| `변경요청편집` | 변경요청 작성 및 `DRAFT/SUBMITTED` 상태 수정 | `저장`, `제출` |

모든 상세 화면은 `loading`, `empty`, `error`, `read only` 상태를 구분하고,
타임라인은 댓글과 활동 로그를 함께 보여준다.

## 5. 핵심 시나리오

### 시나리오 1: 부품 상세에서 이슈 접수

1. 사용자가 부품 상세 화면에서 `새 이슈 등록`을 누른다.
2. 작성 화면은 선택된 부품을 기본 연결 대상으로 채운다.
3. 제목, 본문, 담당자, 라벨, 첨부파일을 입력해 저장한다.
4. 이슈는 `OPEN` 상태로 생성되고, 상세 화면에서 타임라인이 시작된다.

### 시나리오 2: 이슈에서 변경요청 생성

1. 설계 리더가 열린 이슈 상세에서 `변경요청 생성`을 누른다.
2. 변경요청 작성 화면은 원본 이슈 번호와 부품 연결을 기본값으로 가져온다.
3. 담당자, 검토자, 라벨, 첨부파일을 설정하고 저장한다.
4. 변경요청은 `DRAFT`로 생성되고, 이슈-변경요청 링크가 양쪽 상세에
   동시에 보인다.

### 시나리오 3: 변경요청 제출과 검토

1. 작성자가 `DRAFT` 변경요청을 정리한 뒤 `제출`을 누른다.
2. 상태는 `SUBMITTED`로 바뀌고, 검토자는 상세 화면에서
   `APPROVED/REJECTED`를 남긴다.
3. MVP에서는 검토 결과가 `반영` 버튼을 자동으로 막지 않는다.
4. 반영 결정은 담당자나 리더가 최종 수행한다.

### 시나리오 4: 변경요청 반영과 이슈 자동 종결

1. 담당자가 `SUBMITTED` 변경요청에서 `반영`을 누른다.
2. 변경요청은 `MERGED`가 되고, 기본 `Issue.state`는 `CLOSED`가 된다.
3. 시스템은 연결된 일반 이슈를 검사한다.
4. 다른 미해결 변경요청(`DRAFT`, `SUBMITTED`)이 없으면 연결 이슈를
   자동으로 `CLOSED`로 전환한다.

### 시나리오 5: 반영 없이 종료 후 재개

1. 우선순위 하락이나 중복 요청으로 작업을 멈춰야 한다.
2. 사용자는 `DRAFT` 또는 `SUBMITTED` 변경요청을 `닫기` 처리한다.
3. 이후 다시 진행해야 하면 `다시 열기`를 눌러 `SUBMITTED`로 복귀한다.
4. `MERGED`는 재개하지 않는다.

## 8. 데이터 연결

| 엔티티 | 관계 | 설명 |
|--------|------|------|
| Issue | 1:N Comment | 댓글은 타임라인 항목으로 합쳐 조회 |
| Issue | N:M Assignee / TeamAssignee | 개인 담당자와 팀 담당자 동시 지원 |
| Issue | N:M Label | 분류와 우선순위 표시 |
| Issue | N:M Part | MVP 직접 연결 대상은 `부품`만 지원 |
| Issue | N:M ChangeRequest | 이슈 하나가 여러 변경요청과 연결 가능 |
| ChangeRequest | 1:N Reviewer / TeamReviewer | 검토자와 팀 검토자 분리 |
| ChangeRequest | N:M LinkedIssue | 원인 이슈, 연관 이슈 복수 연결 가능 |
| Timeline | Comment + Activity | 댓글과 활동 로그를 시간순 병합 |

### 데이터 모델 특징

- 이슈 번호는 `Issue`와 `ChangeRequest`가 같은 전역 시퀀스를 공유한다.
- 본문과 댓글은 TipTap JSON을 저장 포맷으로 사용한다.
- 멘션은 `userMention`, `issueMention` 구조를 사용하며 활동 로그와 알림을
  발생시킨다.
- 첨부파일은 soft delete로 제거하고, 요청당 최대 20개를 연결한다.

## 9. MVP 범위

### MVP 포함

- 이슈/변경요청 목록, 상세, 작성, 수정
- 이슈 상태 `OPEN/CLOSED`
- 변경요청 상태 `DRAFT/SUBMITTED/MERGED/CLOSED`
- 부품, 라벨, 담당자, 팀담당자, 검토자, 팀검토자 연결
- 첨부파일 배치 연결과 개별 삭제
- 댓글 작성/수정/삭제
- 댓글 + 활동 로그 통합 타임라인
- 이슈-변경요청 양방향 링크 동기화
- 변경요청 `MERGED` 후 연결 이슈 자동 종결
- TipTap 기반 본문/댓글 작성과 멘션 처리

### MVP 제외

- 검토 완료 여부를 기준으로 한 `반영` 강제 차단
- 팀 검토자 직접 리뷰 제출
- BOM/도면/LOT/공정 직접 링크 엔티티
- ECR/ECO 완전 분리 모델
- 변경 영향 diff 비교 화면
- 대량 상태 전이와 일괄 배정
- SLA, 마감기한, 자동 리마인더 대시보드

## 10. FE 구현 메모

- 이슈 목록 검색은 API 기준으로 제목 부분 일치만 보장한다. 숫자 검색은
  `lookup`에서만 지원하므로 상세 연결 picker와 목록 검색 UX를 분리한다.
- 변경요청 목록은 `state`와 `crState`를 모두 보여줘야 한다.
- 댓글 목록은 별도 API가 없고, 타임라인 API에서 `comment/activity`를 같이
  받아 렌더링해야 한다.
- 연결 편집은 `추가/삭제` 버튼이 아니라 최종 선택 목록을 서버에
  `sync`하는 방식이다.
- `MERGED`와 `CLOSED` 변경요청은 편집 화면에서 읽기 전용으로 처리한다.
- 팀 검토자는 배정 현황만 보여주고, 리뷰 제출 버튼은 개인 검토자에게만
  노출한다.
- 본문/댓글 에디터는 TipTap JSON과 멘션 노드를 그대로 보존해야 한다.
- 부품 상세에서 이슈/변경요청 작성으로 진입할 때는 부품 연결을
  미리 채워 사용자가 다시 찾지 않게 해야 한다.

## 11. 완료 조건

- 사용자는 부품 상세 또는 변경관리 메뉴에서 이슈를 생성할 수 있다.
- 사용자는 열린 이슈 상세에서 변경요청을 생성하고 양쪽 링크를 즉시 확인할
  수 있다.
- 사용자는 변경요청의 `crState`와 검토 현황을 같은 화면에서 볼 수 있다.
- 사용자는 타임라인에서 댓글과 상태 변경, 배정 변경, 파일 첨부 이력을
  시간순으로 확인할 수 있다.
- `MERGED` 이후 연결 이슈 자동 종결 규칙이 화면 문구와 서버 동작에서
  일치한다.
- `MERGED/CLOSED` 변경요청에서 편집 CTA가 숨겨지거나 읽기 전용으로
  전환된다.
- 댓글 수정/삭제, 리뷰 제출, 링크 동기화의 실패 조건이 화면에 명확히
  노출된다.

## 12. 후속 확장

- 검토자 전원 승인 이후에만 `반영` 가능한 승인 게이트
- 부품 리비전 릴리스와 변경요청 `MERGED`를 직접 연결하는 ECO 플로우
- BOM, 도면, LOT, 공정, 설비 직접 링크
- 변경 유형 템플릿 (도면 수정, BOM 정정, 대체품 승인 등)
- 변경 영향 범위 자동 계산
- 알림 구독, SLA, 대시보드, 감사로그 내보내기
