# 변경관리 정책

이 문서는 변경관리 모듈의 데이터 모델, 상태 전이, 권한, 유효성 검증,
API 계약을 정의한다. MVP의 기준 구현은 `Issue`와 `ChangeRequest`를
중심으로 하며, 제조 변경의 접수와 반영 추적을 우선 지원한다.

## 1. 데이터 모델

### 1.1 이슈 (`Issue`)

| 필드 | 타입 | 규칙 |
|------|------|------|
| `id` | UUID | 시스템 생성 |
| `number` | Integer | 전역 유일, 이슈/변경요청 공용 시퀀스 |
| `type` | Enum | `ISSUE` 고정 |
| `title` | String | 필수, trim 후 500자 이하 |
| `body` | TipTap JSON | 선택, 저장 시 JSON 유효성 검사 |
| `state` | Enum | `OPEN`, `CLOSED` |
| `closedAt` | Instant | `CLOSED` 전이 시 기록 |
| `createdBy`, `updatedBy` | UUID | 수행자 필수 |
| `assignees` | User[] | 개인 담당자 |
| `assignedTeams` | Team[] | 팀 담당자 |
| `parts` | Part[] | MVP 직접 연결 대상 |
| `labels` | Label[] | 분류/우선순위 표시 |
| `files` | File[] | soft delete 지원 |
| `commentsCount` | Integer | 목록/상세 요약용 |
| `linkedChanges` | ChangeRequest[] | 연결된 변경요청 배지 |

### 1.2 변경요청 (`ChangeRequest`)

| 필드 | 타입 | 규칙 |
|------|------|------|
| `id`, `number`, `title`, `body` | Issue 상속 | 공통 규칙 사용 |
| `type` | Enum | `CHANGE_REQUEST` 고정 |
| `state` | Enum | 기본 `Issue.state`와 함께 유지 |
| `crState` | Enum | `DRAFT`, `SUBMITTED`, `MERGED`, `CLOSED` |
| `mergedAt`, `mergedBy` | Instant, UUID | `MERGED` 전이 시 기록 |
| `reviewers` | Reviewer[] | 개인 검토자 |
| `reviewerTeams` | Team[] | 팀 검토자 |
| `linkedIssues` | Issue[] | 연관 일반 이슈 |

### 1.3 검토자/댓글/타임라인

| 엔티티 | 핵심 필드 | 규칙 |
|--------|-----------|------|
| `ChangeRequestReviewer` | `userId`, `reviewStatus`, `reviewedAt` | `(change_request_id, user_id)` unique |
| `ChangeRequestIssue` | `changeRequestId`, `issueId` | `(change_request_id, issue_id)` unique |
| `IssueComment` | `issueId`, `body`, `createdBy` | 댓글 본문 필수, 작성자만 수정/삭제 |
| `TimelineItem` | `type`, `action`, `detail`, `body` | 댓글과 활동 로그를 병합 |

### 1.4 관계 제약

- 이슈-변경요청 링크는 중복 생성할 수 없다.
- 개인 담당자/검토자와 팀 담당자/검토자가 중복 커버되면 개인 항목을
  제거한다.
- MVP에서는 `Part`만 직접 연결한다. BOM/도면/LOT는 본문/라벨/첨부로만
  표현한다.

## 2. 상태 관리

### 2.1 이슈 상태

```text
OPEN -> CLOSED
  ^       |
  |-------|
```

| 전이 | 허용 조건 | 실패 조건 |
|------|-----------|-----------|
| 생성 -> `OPEN` | 제목 유효, 수행자 존재 | 제목 누락, 수행자 없음 |
| `OPEN -> CLOSED` | 열린 이슈 | 이미 `CLOSED` |
| `CLOSED -> OPEN` | 닫힌 이슈 | 이미 `OPEN` |

- `CLOSED` 이슈는 제목/본문 수정이 불가하다.
- 이슈 자동 종결은 `변경요청 MERGE` 이벤트에서만 수행한다.

### 2.2 변경요청 상태

`ChangeRequest`는 `Issue.state`와 `crState`를 함께 가진다.

| `crState` | `Issue.state` | 편집 가능 여부 | 의미 |
|-----------|---------------|----------------|------|
| `DRAFT` | `OPEN` | 가능 | 작성 중 |
| `SUBMITTED` | `OPEN` | 가능 | 검토 중 |
| `MERGED` | `CLOSED` | 불가 | 반영 완료 |
| `CLOSED` | `CLOSED` | 불가 | 반영 없이 종료 |

상태 전이 규칙은 아래와 같다.

| 전이 | 허용 조건 | 실패 조건 |
|------|-----------|-----------|
| 생성 -> `DRAFT` | 제목 유효, 수행자 존재 | 제목 누락, 수행자 없음 |
| `DRAFT -> SUBMITTED` | 현재 `DRAFT` | `SUBMITTED/MERGED/CLOSED` |
| `SUBMITTED -> MERGED` | 현재 `SUBMITTED` | `DRAFT/MERGED/CLOSED` |
| `DRAFT -> CLOSED` | 현재 `DRAFT` | `MERGED/CLOSED` |
| `SUBMITTED -> CLOSED` | 현재 `SUBMITTED` | `DRAFT/MERGED/CLOSED` |
| `CLOSED -> SUBMITTED` | 현재 `CLOSED` | `DRAFT/SUBMITTED/MERGED` |

- `MERGED`와 `CLOSED` 변경요청은 수정 불가다.
- `reopen`은 `CLOSED -> SUBMITTED`만 허용한다.
- `merge`와 `close`는 `Issue.state`를 함께 `CLOSED`로 바꾼다.

### 2.3 리뷰 상태

| 상태 | 의미 | 전이 규칙 |
|------|------|-----------|
| `PENDING` | 미응답 | 검토자 배정 시 기본값 |
| `APPROVED` | 승인 | 검토자가 직접 제출 |
| `REJECTED` | 반려 | 검토자가 직접 제출 |

- 리뷰 제출 API는 `APPROVED` 또는 `REJECTED`만 허용한다.
- `PENDING` 제출은 실패한다.
- 팀 검토자는 배정 정보만 유지하고, MVP에서는 직접 리뷰를 제출하지 않는다.
- MVP에서는 검토 결과가 `MERGE`를 자동 차단하지 않는다.

### 2.4 연결 이슈 자동 종결

`MERGED` 시점에만 아래 규칙을 수행한다.

1. 현재 변경요청에 연결된 일반 이슈를 조회한다.
2. 각 이슈에 대해 다른 미해결 변경요청(`DRAFT`, `SUBMITTED`)이 남아 있는지
   확인한다.
3. 미해결 변경요청이 없고 이슈가 `OPEN`이면 자동으로 `CLOSED` 처리한다.

실패/예외 규칙:

- 이미 `CLOSED`인 이슈는 다시 닫지 않는다.
- `CLOSED` 변경요청만 남아 있는 경우는 미해결로 보지 않는다.
- 변경요청을 `CLOSED` 처리하는 것만으로는 이슈를 자동 종결하지 않는다.

## 3. 권한

MVP에서는 조직 멤버 공통 열람을 기본으로 두고, 액션 권한만 제한한다.

| 액션 | 조직 멤버 | 작성자/담당자 | 검토자 | 관리자 |
|------|-----------|---------------|--------|--------|
| 목록/상세 조회 | 가능 | 가능 | 가능 | 가능 |
| 이슈/변경요청 생성 | 가능 | 가능 | 가능 | 가능 |
| 제목/본문 수정 | 읽기 전용 | 가능 | 읽기 전용 | 가능 |
| 담당자/팀/라벨/부품/링크 sync | 읽기 전용 | 가능 | 읽기 전용 | 가능 |
| 이슈 닫기/다시 열기 | 읽기 전용 | 가능 | 읽기 전용 | 가능 |
| 변경요청 제출/반영/닫기/다시 열기 | 읽기 전용 | 가능 | 읽기 전용 | 가능 |
| 리뷰 제출 | 불가 | 불가 | 본인만 가능 | 가능 |
| 댓글 작성 | 가능 | 가능 | 가능 | 가능 |
| 댓글 수정/삭제 | 본인만 가능 | 본인만 가능 | 본인만 가능 | 가능 |

권한 추가 규칙:

- 댓글 수정/삭제는 작성자 본인만 가능하다.
- 리뷰 제출은 해당 변경요청에 개인 검토자로 배정된 사용자만 가능하다.
- FE는 액션 권한이 없을 때 버튼을 숨기거나 읽기 전용 문구를 노출해야 한다.

## 4. 유효성 검증

### 4.1 공통 필드

| 항목 | 검증 규칙 |
|------|-----------|
| `title` | 필수, trim 후 500자 이하 |
| `body` | null 허용, 존재 시 TipTap JSON 유효 문서 |
| `comment.body` | 필수, 공백만 입력 불가 |
| `fileIds` | 요청당 최대 20개 |
| `state`, `crState` 파라미터 | enum 이외 값이면 `VALIDATION_ERROR` |

### 4.2 연결 관계

| 항목 | 검증 규칙 |
|------|-----------|
| `labelIds` | 모두 존재해야 함 |
| `partIds` | 모두 존재해야 함 |
| `issueIds` | 일반 이슈만 허용 |
| `changeRequestIds` | 변경요청만 허용 |
| 담당자/검토자 | 중복 입력 불가 |
| 팀 배정 | 팀 멤버와 개인 배정이 겹치면 개인 항목 제거 |

### 4.3 리뷰/댓글

- `reviewStatus`는 `APPROVED`, `REJECTED`만 허용한다.
- 리뷰 시각(`reviewedAt`)은 리뷰 저장 시 자동 기록한다.
- 댓글 본문은 비어 있을 수 없다.
- 댓글 수정/삭제는 동일 이슈 소속 여부를 먼저 확인한다.

### 4.4 본문/멘션

- 본문과 댓글은 TipTap JSON만 저장한다.
- 멘션은 `userMention`, `issueMention` 포맷이어야 한다.
- 잘못된 문서 구조나 허용되지 않은 노드는 저장 실패다.

## 5. 비즈니스 규칙

### 5.1 생성과 수정

- 이슈 번호는 이슈/변경요청 공용 시퀀스에서 채번한다.
- 이슈 생성 후 상태는 항상 `OPEN`이다.
- 변경요청 생성 후 상태는 항상 `DRAFT`다.
- 변경요청 생성 시 `issueNumber` 1건을 바로 연결할 수 있다.
- 추가 이슈 연결은 생성 후 `syncIssues`로 처리한다.

### 5.2 검색과 lookup

- 목록 검색은 제목 부분 일치만 지원한다.
- lookup 검색은 제목 부분 일치와 번호 부분 일치를 모두 지원한다.
- 목록 필터는 `state`, 변경요청의 경우 `cr_state`를 함께 받는다.

### 5.3 sync 규칙

- 연결 API는 `추가/삭제`가 아니라 최종 목록 전체를 보내는 sync 방식이다.
- 응답은 `addedCount`, `removedCount`만 반환한다.
- sync 성공 시 양쪽 타임라인에 동일한 링크 변경 활동 로그를 남긴다.

### 5.4 첨부파일과 저장소

- 파일 첨부는 배치 연결한다.
- 같은 파일이 다른 리소스에 이미 연결돼 있으면 재첨부할 수 없다.
- 파일 삭제는 soft delete이며, 테넌트 저장소 사용량을 함께 조정한다.

### 5.5 타임라인과 활동 로그

- 타임라인은 댓글과 활동 로그를 생성 시각 기준 오름차순으로 합쳐 반환한다.
- 활동 로그는 상태 변경, 담당자 변경, 검토자 변경, 라벨 변경, 부품 변경,
  파일 첨부/삭제, 이슈-변경요청 링크 변경, 멘션을 포함한다.
- 사용자 멘션은 알림 이벤트를 발행한다.

## 6. API/이벤트 계약

### 6.1 이슈 API

| 목적 | 메서드 | 경로 |
|------|--------|------|
| 목록 | `GET` | `/api/v1/issues` |
| 상세 | `GET` | `/api/v1/issues/{issueNumber}` |
| lookup | `GET` | `/api/v1/issues/lookup` |
| 생성 | `POST` | `/api/v1/issues` |
| 수정 | `PATCH` | `/api/v1/issues/{issueNumber}` |
| 닫기/다시 열기 | `POST` | `/close`, `/reopen` |
| 담당/팀/라벨/부품/변경요청 sync | `PUT` | `/assignees`, `/assigned-teams`, `/labels`, `/parts`, `/changes` |
| 타임라인 | `GET` | `/api/v1/issues/{issueNumber}/timeline` |
| 댓글 | `POST/PATCH/DELETE` | `/comments` |
| 파일 | `POST/DELETE` | `/files` |

### 6.2 변경요청 API

| 목적 | 메서드 | 경로 |

| 생성 | `POST` | `/api/v1/changes` |
| 수정 | `PATCH` | `/api/v1/changes/{issueNumber}` |

| 이슈/담당/검토/라벨/부품 sync | `PUT` | `/issues`, `/assignees`, `/assigned-teams`, `/reviewers`, `/reviewer-teams`, `/labels`, `/parts` |

### 6.3 활동/이벤트

| 구분 | 값 |
|------|----|
| 타임라인 항목 타입 | `comment`, `activity` |

| 알림 이벤트 | 사용자 멘션 시 발행 |

## 7. BE 구현 메모

- 번호 시퀀스는 행 잠금 기반으로 배정해 중복 발급을 막아야 한다.
- `Issue`와 `ChangeRequest`는 같은 query 계층에서 enrichment해 목록/상세
  응답을 만든다.
- 타임라인은 댓글 목록 API를 별도로 만들지 않고, 활동 로그와 병합해
  반환한다.
- `merge` 후 자동 종결 로직은 현재 변경요청에 연결된 이슈만 대상으로 한다.
- 권한 플래그를 FE에 주고 싶다면 응답 확장보다 BFF 또는 별도 capability
  필드를 검토한다.
- TipTap 검증, 멘션 추출, 파일 소유권 검증은 공통 서비스로 유지한다.

## 8. 테스트 시나리오

- 이슈 생성 시 `OPEN` 상태와 전역 번호가 발급된다.
- 변경요청 생성 시 `DRAFT`와 `OPEN` 조합으로 생성된다.
- 닫힌 이슈 수정 시 실패한다.
- `DRAFT -> SUBMITTED -> MERGED` 전이가 성공한다.
- `MERGED/CLOSED` 변경요청 수정 시 실패한다.
- `CLOSED -> SUBMITTED` 재개가 성공하고, 다른 상태에서는 실패한다.
- 개인 검토자가 아닌 사용자의 리뷰 제출은 실패한다.
- `PENDING` 리뷰 제출은 실패한다.
- 댓글 수정/삭제는 작성자 본인만 성공한다.
- sync API는 최종 목록 기준으로 추가/삭제 수를 정확히 반환한다.
- `MERGED` 이후 다른 미해결 변경요청이 없을 때만 연결 이슈가 자동 종결된다.
- `CLOSED` 변경요청만 남아 있는 상태에서는 자동 종결 대상에서 제외된다.
- 파일 21개 첨부 시 검증 실패한다.
- 잘못된 TipTap JSON 저장 시 검증 실패한다.
