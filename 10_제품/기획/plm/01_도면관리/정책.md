# 도면관리 정책

이 문서는 도면 데이터 모델, 승인 상태, 변환 상태, 권한, 유효성 검증을
정의한다. 화면 맥락과 시나리오는
[[10_제품/기획/plm/01_도면관리/기획|기획]] 문서를 참고한다.

## 1. 데이터 모델

도면관리는 메타데이터, 원본 파일, 파생 아티팩트, 변환 잡을 함께 다룬다.
MVP에서는 현재 `server2` Drawing 구조를 유지하되, 부족한 제약은 명시적으로
추가한다.

### 1.1 Drawing

`Drawing`은 도면의 업무상 기준 엔티티다.

| 필드 | 타입 | 제약 | 설명 |
|------|------|------|------|
| `id` | UUID | 필수 | 도면 식별자 |
| `drawingNumber` | string | 필수, 조직 내 유일, 1~100자 | 도면번호 |
| `name` | string | 필수, 1~500자 | 도면명 |
| `version` | string | 필수, 1~50자 | 작업 버전 또는 리비전 표기 |
| `status` | enum | 필수 | `DRAFT`, `IN_REVIEW`, `APPROVED`, `RELEASED`, `OBSOLETE` |
| `conversionStatus` | enum | 필수 | `PENDING`, `PROCESSING`, `COMPLETED`, `FAILED` |
| `sourceType` | enum | 필수 | `CAD_2D`, `PDF_DOCUMENT`, `RASTER_IMAGE`, `CAD_3D` |
| `dimension` | enum | 필수 | `TWO_D`, `THREE_D` |
| `sourceFileId` | UUID | 필수 | 원본 업로드 파일 ID |
| `currentJobId` | UUID | 선택 | 현재 처리 중인 변환 잡 |
| `deletedAt` | datetime | 선택 | 소프트 삭제 시각 |

### 1.2 DrawingArtifact

`DrawingArtifact`는 뷰어와 다운로드를 위한 저장 객체를 정의한다.

| 타입 | 설명 | 필수 여부 |
|------|------|-----------|
| `SOURCE_ORIGINAL` | 사용자가 업로드한 원본 파일 | 필수 |
| `DERIVED_PDF` | 2D 통합 뷰어용 PDF | 2D 필수 |
| `DERIVED_WEBP` | 목록/상세 미리보기용 이미지 | 권장 |
| `DERIVED_GLB` | 3D 뷰어용 GLB | 3D 필수 |

### 1.3 DrawingProcessingJob

`DrawingProcessingJob`은 비동기 변환 이력을 남긴다. 최신 잡 1건의 상태를
상세 화면에 우선 노출한다.

| 필드 | 설명 |
|------|------|
| `pipelineKey` | 어떤 변환 파이프라인을 썼는지 |
| `profileKey` | 변환 프로필 |
| `status` | `REQUESTED`, `PROCESSING`, `COMPLETED`, `FAILED` |
| `attemptCount` | 재처리 횟수 |
| `failureReason` | 원문 실패 메시지 |
| `startedAt` / `completedAt` | 처리 시각 |

### 1.4 Part 연결

MVP는 현재 구현을 따라 `Part.drawingId` 기반으로 대표 도면 1건만 연결한다.
다만 제품 정책은 1:1을 명시하므로, 워크스페이스 기준 `parts.drawing_id`
고유 제약이 필요하다.

- 하나의 `Part`는 대표 도면을 최대 1건 가진다.
- 하나의 `Drawing`은 대표 연결 부품을 최대 1건 가진다.
- 새 도면을 같은 부품에 연결하면 기존 대표 도면은 해제된다.
- 후속 범위에서 `drawing_links` 테이블로 N:M 링크를 확장한다.

### 1.5 감사로그

추적성은 설계에 포함하되, 상세 조회 UI는 후속 범위로 둔다. 최소 이벤트
스키마는 기획 단계에서 아래처럼 고정한다.

- `drawing.created`
- `drawing.updated`
- `drawing.status_changed`
- `drawing.processing_requested`
- `drawing.processing_failed`
- `drawing.processing_retried`
- `drawing.part_linked`
- `drawing.part_unlinked`
- `drawing.deleted`

## 2. 상태 관리

도면은 승인 상태와 변환 상태를 별도로 가진다. 두 상태는 서로 다른 목적을
가진다.

## 2.1 승인 상태

승인 상태는 업무상 기준본 여부를 나타낸다.

| 상태 | 의미 | 편집 가능 여부 |
|------|------|----------------|
| `DRAFT` | 작성 중 | O |
| `IN_REVIEW` | 검토 대기 | X |
| `APPROVED` | 승인 완료, 배포 전 | X |
| `RELEASED` | 현장 기준본 | X |
| `OBSOLETE` | 폐기됨 | X |

### 2.1.1 승인 상태 전이

```text
DRAFT -> IN_REVIEW -> APPROVED -> RELEASED -> OBSOLETE
              \-> DRAFT (반려)
```

MVP에서는 위 전이만 허용한다.

| 전이 | 조건 | 실패 조건 |
|------|------|-----------|
| `DRAFT -> IN_REVIEW` | 필수 메타데이터 완료, 원본 파일 존재, `conversionStatus=COMPLETED` | 메타데이터 누락, 변환 미완료 |
| `IN_REVIEW -> DRAFT` | 검토자 반려 사유 입력 | 권한 없음 |
| `IN_REVIEW -> APPROVED` | 검토 권한 보유, 변환 성공 상태 유지 | 권한 없음, 변환 실패 |
| `APPROVED -> RELEASED` | 대표 연결 부품 1건 연결, 변환 성공 상태 유지 | 부품 미연결, 변환 실패 |
| `RELEASED -> OBSOLETE` | 대체 도면 배포 완료 또는 관리자 사유 입력 | 권한 없음, 사유 없음 |

### 2.1.2 상태별 버튼 규칙

상세 화면 버튼은 승인 상태와 권한에 따라 달라진다.

- `DRAFT`: `편집`, `검토 요청`, `삭제`
- `IN_REVIEW`: `승인`, `반려`
- `APPROVED`: `배포`, `반려`
- `RELEASED`: `폐기`, `원본 다운로드`
- `OBSOLETE`: `원본 다운로드`

## 2.2 변환 상태

변환 상태는 브라우저에서 도면을 열 수 있는 준비 여부를 나타낸다.

| 상태 | 의미 | 뷰어 노출 |
|------|------|-----------|
| `PENDING` | 요청됨, 대기 중 | X |
| `PROCESSING` | 변환기 실행 중 | X |
| `COMPLETED` | 뷰어 산출물 준비 완료 | O |
| `FAILED` | 변환 실패 | X |

### 2.2.1 변환 상태 전이

```text
PENDING -> PROCESSING -> COMPLETED
                     \-> FAILED
FAILED -> PENDING (재처리 또는 원본 교체)
```

변환 상태는 파일 등록과 원본 교체, 재처리에서만 바뀐다.

| 전이 | 조건 | 실패 조건 |
|------|------|-----------|
| `null -> PENDING` | 신규 도면 등록 또는 원본 교체 | 원본 파일 없음 |
| `PENDING -> PROCESSING` | 잡 클레임 성공 | 잡 중복 실행 |
| `PROCESSING -> COMPLETED` | 산출물 저장 성공 | PDF/GLB/WEBP 저장 실패 |
| `PROCESSING -> FAILED` | 변환기 실패, 시간 초과, 지원하지 않는 포맷 | 없음 |
| `FAILED -> PENDING` | 재처리 또는 원본 교체 | 권한 없음 |

### 2.2.2 실패 코드 매핑

MVP는 현재 `server2` 실패 코드를 그대로 사용자 문구와 연결한다.

| 코드 | 의미 | 사용자 문구 |
|------|------|-------------|
| `TIMEOUT` | 시간 초과 | 도면 변환 시간이 초과되었습니다 |
| `UNSUPPORTED_FORMAT` | 미지원 포맷 | 지원하지 않는 도면 형식입니다 |
| `CONVERTER_UNAVAILABLE` | 변환기 비가용 | 변환기를 사용할 수 없습니다 |
| `CONVERSION_FAILED` | 일반 변환 실패 | 도면 변환에 실패했습니다 |
| `UNKNOWN` | 미분류 오류 | 도면 처리 중 오류가 발생했습니다 |

## 3. 권한

도면은 설계팀이 쓰기, 생산·품질 부서가 읽기 중심으로 사용하는 구조다.
권한은 역할과 상태를 함께 본다.

| 액션 | 설계 엔지니어 | 설계 리더 | 생산/품질 | 관리자 |
|------|---------------|-----------|-----------|--------|
| 목록/상세 조회 | O | O | O | O |
| 신규 등록 | O | O | X | O |
| Draft 편집 | O | O | X | O |
| 검토 요청 | O | O | X | O |
| 승인/반려 | X | O | X | O |
| 배포 | X | O | X | O |
| 폐기 | X | O | X | O |
| 재처리 | X | O | X | O |
| 삭제 | O (`DRAFT`만) | O | X | O |

권한이 있어도 상태 제약을 넘을 수는 없다. 예를 들어 설계 엔지니어라도
`RELEASED` 도면은 편집할 수 없다.

## 4. 유효성 검증

유효성 검증은 프론트와 백엔드가 같은 기준을 사용해야 한다. 필드 검증과
상태 전이 검증을 분리한다.

### 4.1 필드 검증

| 항목 | 규칙 |
|------|------|
| 도면번호 | 필수, 공백 불가, 100자 이하, 조직 내 유일 |
| 도면명 | 필수, 공백 불가, 500자 이하 |
| 버전 | 필수, 공백 불가, 50자 이하 |
| 원본 파일 | 신규 등록 시 필수, `UPLOADED` 상태 파일만 허용 |
| 대표 연결 부품 | `RELEASED` 전환 전까지 필수 |
| 삭제 | `DRAFT` + 미배포 + 미연결일 때만 허용 |

### 4.2 파일 검증

지원 포맷은 파일 확장자로 판정한다.

- 2D CAD: `.dwg`, `.dxf`
- PDF 문서: `.pdf`
- 이미지: `.png`, `.jpg`, `.jpeg`, `.bmp`, `.tif`, `.tiff`, `.webp`
- 3D: `.step`, `.stp`, `.iges`, `.igs`, `.stl`, `.obj`, `.3mf`, `.fbx`,
  `.glb`, `.gltf`

지원하지 않는 확장자는 `VALIDATION_ERROR`로 처리한다.

### 4.3 저장 실패 코드

최소 오류 코드는 아래를 구분한다.

- `VALIDATION_ERROR`
- `DUPLICATE_DRAWING_NUMBER`
- `FILE_NOT_UPLOADED`
- `UNSUPPORTED_FORMAT`
- `INVALID_STATUS_TRANSITION`
- `PERMISSION_DENIED`
- `DRAWING_NOT_EDITABLE`
- `PART_LINK_REQUIRED`

## 5. 비즈니스 규칙

MVP는 현재 `server2`의 Drawing 서비스가 가진 실제 동작을 최대한 활용하되,
제품 제약을 명시적으로 덧붙인다.

### 5.1 뷰어 선택 규칙

- `dimension=THREE_D`이고 `glbKey`가 있으면 `GLB` 뷰어를 사용한다.
- 그 외에는 `PDF` 뷰어를 사용한다.
- 산출물이 아직 없으면 예상 뷰어 타입만 보여 주고 실제 뷰어는 비활성화한다.

### 5.2 저장소와 삭제 정책

- 도면 삭제는 `Drawing`과 관련 `File`을 모두 소프트 삭제한다.
- 원본, PDF, WEBP, GLB 파일 크기는 조직 저장 용량에 반영한다.
- `RELEASED`와 `OBSOLETE`는 삭제하지 않고 상태로 관리한다.
- 현재 `Part` 기준 삭제 API를 그대로 쓰면 공유 도면이 함께 지워질 수 있으므로,
  MVP 구현 전 1:1 제약 또는 역참조 검증을 먼저 넣어야 한다.

### 5.3 부품 연결 정책

- `Part`는 대표 도면을 최대 1건 가진다.
- 새 도면을 같은 `Part`에 연결하면 기존 대표 도면 연결을 해제한다.
- `RELEASED` 전환 전에는 대표 연결 부품이 비어 있을 수 있지만,
  `RELEASED` 시점에는 반드시 1건이어야 한다.
- 후속 범위에서 `BOM`과 `ECO`까지 링크를 확장한다.

### 5.4 배포본 수정 정책

- `RELEASED`는 인플레이스 수정하지 않는다.
- 수정이 필요하면 새 도면을 `DRAFT`로 등록하고 기존 도면을 `OBSOLETE`로
  전환한다.
- 정식 도면 리비전 엔티티와 새 Draft 자동 생성은 후속 범위다.

### 5.5 동시성 정책

- 같은 도면에 대해 동시에 2개의 활성 변환 잡을 두지 않는다.
- `currentJobId`가 살아 있는 동안 중복 재처리 요청은 무시하거나
  `PRECONDITION_FAILED`를 반환한다.
- 사용자 편집 잠금은 후속 범위다.

### 5.6 MVP 적용 범위

- 현재 `server2`가 이미 가진 비동기 변환 상태 조회는 MVP에 포함한다.
- 메타데이터 편집, 승인 상태 전이, 독립 목록/상세 API는 새로 구현한다.
- 1:1 대표 부품 연결 제약은 MVP에서 명시적으로 강제한다.

### 5.7 후속 정책

후속 정책은 도면관리 MVP가 안정화된 뒤 순차적으로 확장한다.

- 다중 부품/N:M 링크
- 도면 마스터/버전 분리
- 결재선, 서명, 코멘트 스레드
- 사용자 편집 잠금
- 감사로그 상세 UI

## 6. API/이벤트 계약

기존 `server2`는 부품 중심 등록 API와 처리 상태 조회 API만 노출한다.
독립 메뉴형 도면관리를 위해 아래 계약이 필요하다.

### 6.1 조회 API

조회 API는 목록, 상세, 변환 상태를 분리해 응답한다.

- `GET /api/v1/drawings`
  - 검색: `search`, `status`, `conversion_status`, `dimension`, `source_type`
  - 응답: 목록 행 + 대표 연결 부품 요약 + 썸네일 URL
- `GET /api/v1/drawings/{drawingId}`
  - 응답: 메타데이터, 상태, 변환 상태, 뷰어 URL, 원본 URL, 대표 연결 부품
- `GET /api/v1/drawings/{drawingId}/processing`
  - 기존 API 유지

### 6.2 쓰기 API

쓰기 API는 Draft 생성, 상태 전이, 재처리를 명시적으로 나눈다.

- `POST /api/v1/drawings`
  - 입력: `fileId`, `drawingNumber`, `name`, `version`, `partId`
- `PATCH /api/v1/drawings/{drawingId}`
  - 입력: `name`, `version`, `partId`
- `POST /api/v1/drawings/{drawingId}/submit-review`
- `POST /api/v1/drawings/{drawingId}/approve`
- `POST /api/v1/drawings/{drawingId}/reject`
- `POST /api/v1/drawings/{drawingId}/release`
- `POST /api/v1/drawings/{drawingId}/obsolete`
- `POST /api/v1/drawings/{drawingId}/retry-processing`
- `DELETE /api/v1/drawings/{drawingId}`

### 6.3 호환성 유지

기존 부품 상세 기능과 신규 도면 메뉴는 일정 기간 병행 운영한다.

- `POST /api/v1/parts/{partId}/drawings`는 기존 FE 호환을 위해 한동안 유지한다.
- 신규 FE는 도면 중심 API를 우선 사용한다.

### 6.4 이벤트

이벤트는 감사로그와 후속 알림 기능의 공통 소스로 쓴다.

- `drawing.created`
- `drawing.updated`
- `drawing.processing.completed`
- `drawing.processing.failed`
- `drawing.status.changed`
- `drawing.part.changed`

## 7. BE 구현 메모

지금 코드 기준으로 실제 구현이 필요한 부분을 명확히 남긴다.

- `DrawingService.createDrawing(UUID fileId)`는 파일명만 이름으로 쓰고
  `drawingNumber`를 받지 않는다. 등록 DTO를 확장해야 한다.
- `drawing_number`는 soft delete를 고려한 재사용 정책이 없어, 같은 번호를
  다시 등록할 때 DB 충돌 가능성을 먼저 정리해야 한다.
- `Drawing.changeStatus(...)`는 현재 제약이 없다. 상태 전이 유스케이스에서
  허용 전이와 실패 조건을 강제해야 한다.
- 목록/상세 조회는 `Drawing`, `DrawingServingProjection`, `Part`를 조합한
  전용 Query가 필요하다.
- `parts.drawing_id`에 고유 제약이 없으면 1:1 정책이 깨진다. DB 제약 또는
  서비스 검증이 필요하다.
- 현재 부품 교체 등록은 기존 도면을 자동 정리하지 않으므로 orphan 도면과
  orphan 파일 정리 전략이 필요하다.
- `retry-processing`은 현재 부재하므로 변환 잡 재요청 유스케이스가 필요하다.
- 감사로그는 후속이지만 이벤트 포맷은 지금 결정해 두어야 한다.

## 8. 테스트 시나리오

성공과 실패 시나리오는 상태 전이와 변환 흐름을 기준으로 설계한다.

- 신규 도면 생성 시 `status=DRAFT`, `conversionStatus=PENDING`으로 시작한다.
- 지원 포맷 파일이면 변환 잡이 생성되고 최종적으로 `COMPLETED`가 된다.
- 미지원 포맷이면 등록 또는 변환 단계에서 실패한다.
- `DRAFT -> IN_REVIEW`는 메타데이터와 변환 완료가 모두 있어야 성공한다.
- `APPROVED -> RELEASED`는 대표 연결 부품이 없으면 실패한다.
- `RELEASED` 도면에 편집 요청을 보내면 `DRAWING_NOT_EDITABLE`이 발생한다.
- 같은 부품에 새 도면을 연결하면 기존 대표 도면 연결이 해제된다.
- 변환 실패 도면에 `retry-processing`을 호출하면 `PENDING`으로 돌아간다.
- 삭제는 `DRAFT` 상태에서만 허용되고 관련 파일도 함께 소프트 삭제된다.
