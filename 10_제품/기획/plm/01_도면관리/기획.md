# 도면관리 기획

이 문서는 Fabbit PLM의 도면 등록, 변환, 검토, 배포, 부품 연결 흐름을
정의한다. 현재 `server2` Drawing 구현이 가진 `도면 메타데이터 + 원본 파일 +
파생 뷰어 산출물 + 비동기 변환 상태` 구조를 기준으로, 독립 메뉴형 도면관리
MVP를 정리한다.

상세 규칙은 정책 문서에, 화면 산출물은 `screens/`, 플로우 산출물은
`flows/`에 분리한다.

## 1. 문제와 목표

제조 현장의 도면은 공유폴더와 메신저로 흩어져 있어 최신 승인본 보장이
어렵다. 현장 작업자는 `DWG`, `STEP`, `PDF`, 이미지 도면을 제각각 열어야
하고, 어떤 도면이 어떤 부품의 기준본인지 역추적하기도 어렵다.

목표는 아래 네 가지다.

1. **최신 승인본 보장**: 현장에서 항상 `Released` 상태의 최신 도면만
   기준본으로 사용하게 한다.
2. **뷰어 표준화**: 2D는 `PDF`, 3D는 `GLB`로 변환해 브라우저에서 바로
   본다.
3. **부품 연결 가시화**: 도면 상세에서 대표 연결 부품과 다운로드 가능한
   원본 파일을 함께 확인한다.
4. **실패 원인 노출**: 변환 실패나 승인 불가 사유를 사용자가 즉시 이해하고
   조치할 수 있게 한다.

## 2. 주요 사용자

도면관리는 설계팀이 생성하고, 리더가 검토·배포하며, 생산·품질 부서는
읽기 전용으로 참조하는 구조를 기본으로 한다.

| 사용자 | 주요 관심사 |
|--------|-------------|
| 설계 엔지니어 | 도면 등록, 메타데이터 입력, 원본 교체, 검토 요청 |
| 설계 리더 | 검토, 승인, 배포, 폐기 전환, 변환 실패 대응 |
| 생산관리자 | 최신 배포본 조회, 원본 다운로드, 연결 부품 확인 |
| 품질 담당자 | 승인 상태 확인, 변경된 도면 추적, 증적 조회 |
| 시스템 관리자 | 지원 포맷, 변환기 상태, 저장 용량, 권한 정책 관리 |

## 3. 메뉴명과 진입 경로

도면관리는 PLM 코어 기능이므로 독립 상위 메뉴를 가진다. 다만 부품 상세에서
도면 상세로 바로 넘어가는 딥링크도 함께 제공한다.

| 구분 | 권장 명칭 | 비고 |
|------|-----------|------|
| 상위 메뉴 | `도면` | 제조 현장 용어를 그대로 사용 |
| 하위 메뉴 | `도면 목록` | 기본 진입 화면 |
| 화면 제목 | `도면 상세` | 뷰어 + 메타데이터 + 상태 액션 |
| 기본 CTA | `새 도면 등록` | 업로드 완료 파일 기준 등록 |

진입 경로 예시는 아래와 같다.

`도면 > 도면 목록 > 새 도면 등록`
`도면 > 도면 목록 > 도면 상세`
`부품 > 부품 상세 > 관련 도면 > 도면 상세`

## 4. 화면 구성

MVP 화면은 목록, 상세, 편집 3개로 고정한다. 잠금 기능은 후속 범위지만,
읽기 전용·권한 없음·변환 중 상태는 MVP에서 반드시 구분한다.

### 목록 화면

목록 화면은 도면번호, 도면명, 연결 부품, 승인 상태, 변환 상태를 한 번에
검색하고 운영 액션으로 진입하는 허브다.

- 주요 CTA: `새 도면 등록`, `상세 보기`
- 필수 상태: `loading`, `empty`, `error`
- 읽기 전용: 조회 권한만 있으면 접근 가능
- 잠금 상태: 후속 범위, MVP는 표시하지 않음

### 상세 화면

상세 화면은 도면 메타데이터, 뷰어, 대표 연결 부품, 변환 상태, 상태 전이
액션을 함께 제공한다. 도면 상태와 변환 상태를 동시에 보여 주는 것이 핵심이다.

- 주요 CTA: `편집`, `검토 요청`, `승인`, `배포`, `폐기`, `재처리`,
  `원본 다운로드`
- 필수 상태: `processing`, `failed`, `permission denied`, `read only`
- 읽기 전용: `Released`, 권한 없음, 변환 실패 후 편집 제한 상황
- 잠금 상태: 후속 범위, Draft 편집 잠금은 나중에 추가

### 편집 화면

편집 화면은 신규 등록과 `Draft` 도면 수정에만 사용한다. `Released` 도면은
직접 수정하지 않고 읽기 전용으로 본다.

- 주요 CTA: `저장`, `검토 요청으로 이동`
- 필수 상태: `saving`, `validation error`, `upload missing`, `leave guard`
- 읽기 전용: `Released`, `Obsolete`, 권한 없음
- 잠금 상태: 후속 범위, MVP에서는 상태 기반 읽기 전용만 제공

### 화면별 API 의존성

화면이 바로 구현 가능하도록 최소 API 의존성을 아래처럼 고정한다.

| 화면 | 필수 API | 비고 |
|------|----------|------|
| 목록 | `GET /api/v1/drawings` | 검색, 상태 필터, 변환 상태 필터 포함 |
| 상세 | `GET /api/v1/drawings/{id}` | 메타데이터, 뷰어 URL, 대표 연결 부품 조회 |
| 상세 | `GET /api/v1/drawings/{id}/processing` | 폴링 기반 변환 상태 갱신 |
| 편집 | `POST /api/v1/drawings` | 신규 Draft 생성 |
| 편집 | `PATCH /api/v1/drawings/{id}` | Draft 메타데이터 수정 |
| 상태 액션 | `POST /api/v1/drawings/{id}/*` | 검토 요청, 승인, 배포, 폐기, 재처리 |

## 5. 핵심 시나리오

MVP는 현재 `server2`의 데이터 구조를 크게 깨지 않으면서 독립 메뉴형 경험을
제공하는 데 집중한다.

### 시나리오 1: 신규 도면 등록과 비동기 변환

1. 설계 엔지니어가 `새 도면 등록`을 클릭한다.
2. 업로드 완료된 파일을 선택하고 도면번호, 도면명, 버전, 대표 연결 부품을
   입력한다.
3. 저장하면 도면이 `Draft`, 변환 상태가 `PENDING`으로 생성된다.
4. 시스템은 파일 확장자를 분류해 2D는 `PDF`, 3D는 `GLB` 뷰어 산출물을
   비동기로 만든다.
5. 상세 화면은 `PROCESSING` 배지를 보여 주고, 완료 시 뷰어를 자동으로
   활성화한다.

### 시나리오 2: 검토 요청, 승인, 배포

1. 작성자가 `Draft` 도면을 저장하고 검토 요청을 보낸다.
2. 시스템은 필수 메타데이터와 변환 완료 여부를 검증한다.
3. 검토자는 `In Review` 상태의 도면을 확인하고 `승인` 또는 `반려`한다.
4. 승인된 도면만 `Released`로 배포할 수 있다.
5. `Released`가 되면 현장에서는 이 도면만 기준본으로 참조한다.

### 시나리오 3: 변환 실패 대응

1. 지원하지 않는 포맷이거나 변환기가 비정상인 경우 도면 변환이 실패한다.
2. 상세 화면은 `FAILED` 배지, 실패 코드, 사용자용 안내 문구를 보여 준다.
3. 설계 엔지니어는 원본 파일을 교체하거나, 관리자는 `재처리`를 실행한다.
4. 재처리 시 변환 상태는 다시 `PENDING`으로 돌아간다.

### 시나리오 4: 배포본 교체

1. 배포된 도면을 수정해야 한다.
2. MVP에서는 기존 `Released` 도면을 인플레이스 수정하지 않는다.
3. 사용자는 새 도면을 `Draft`로 등록하고 대표 연결 부품을 새 도면으로
   교체한다.
4. 새 도면이 `Released` 되면 기존 도면은 `Obsolete`로 전환한다.

## 6. 사용자/시스템 플로우

플로우 문서는 등록, 승인 배포, 변환 상태를 각각 분리해 검토한다.

- 와이어프레임: `screens/목록.tsx`, `screens/상세.tsx`, `screens/편집.tsx`
- 플로우: `flows/신규도면등록.tsx`, `flows/승인배포.tsx`,
  `flows/변환상태.tsx`

플로우에서 확인해야 할 포인트는 아래와 같다.

- 누가 `Draft`를 만들고 누가 `Released`를 확정하는지
- 어떤 조건에서 `PENDING/FAILED`가 되는지
- 변환 실패 시 사용자가 어디에서 막히는지
- `Released` 교체를 현재 구현에서 어떻게 우회 처리하는지

## 7. 데이터 연결

도면관리는 메타데이터만 다루는 기능이 아니라 파일 저장, 변환 잡, 뷰어
서빙, 부품 참조를 함께 묶는 허브다.

| 엔티티 | 관계 | 설명 |
|--------|------|------|
| Drawing | 1:N | 도면은 원본/파생 산출물(`Artifact`)을 가진다 |
| DrawingArtifact | N:1 | 원본, PDF, WEBP, GLB 저장 키를 보관한다 |
| DrawingProcessingJob | 1:N | 최신 변환 잡 상태를 추적한다 |
| DrawingServingProjection | 1:1 | 뷰어 제공용 캐시 키를 보관한다 |
| File | 1:1 | 업로드 완료 파일이 원본 파일로 연결된다 |
| Part | 1:1 (MVP) | 대표 연결 부품 1건을 직접 연결한다 |
| Organization | 소속 | 저장 용량과 권한 정책을 조직 단위로 적용한다 |

### 지원 파일 범위

MVP는 현재 `server2` 분류기 기준 포맷을 그대로 따른다.

- 2D CAD: `dwg`, `dxf`
- 2D 문서: `pdf`
- 2D 이미지: `png`, `jpg`, `jpeg`, `bmp`, `tif`, `tiff`, `webp`
- 3D CAD/메시: `step`, `stp`, `iges`, `igs`, `stl`, `obj`, `3mf`, `fbx`,
  `glb`, `gltf`

## 8. MVP 범위

MVP는 “독립 메뉴형 도면 목록/상세/등록 + 비동기 변환 상태 + 대표 부품
연결”까지를 구현 범위로 본다.

### MVP 포함

- 독립 메뉴형 도면 목록, 상세, 편집 화면
- 도면번호, 도면명, 버전, 대표 연결 부품 입력
- 지원 포맷 업로드 파일 기반 도면 등록
- `Draft -> In Review -> Approved -> Released -> Obsolete` 상태 관리
- `PENDING -> PROCESSING -> COMPLETED -> FAILED` 변환 상태 노출
- 2D `PDF` 뷰어, 3D `GLB` 뷰어, 썸네일 표시
- 변환 실패 코드와 사용자용 안내 문구 노출
- 원본 다운로드
- 대표 연결 부품 1건 연결 및 교체
- 소프트 삭제와 저장 용량 반영

### MVP 제외

- 도면 마스터/버전 엔티티 분리
- 배포본에서 자동으로 새 Draft 생성하는 정식 리비전 흐름
- 다중 부품 연결, N:M 링크 관리
- CAD 메타데이터 자동 추출과 도면번호 자동 채번
- 결재선, 전자서명, 코멘트 스레드
- 사용자 편집 잠금
- 감사로그 상세 조회 UI
- 대량 업로드와 일괄 배포

## 9. FE 구현 메모

FE는 승인 상태와 변환 상태를 별도 배지로 보여 줘야 한다. `Released /
FAILED` 같이 서로 다른 축의 상태가 동시에 존재할 수 있기 때문이다.

- 목록은 `도면번호/도면명/부품번호` 통합 검색과 다중 필터를 지원한다.
- 상세는 `viewerType`에 따라 `PDF` 또는 `GLB` 컴포넌트로 분기한다.
- 상세는 `GET /drawings/{id}/processing` 응답을 주기적으로 폴링해 상태를
  갱신한다.
- 편집은 `Draft` 전용으로 열고, `Released`와 `Obsolete`는 읽기 전용으로
  노출한다.
- 변환 실패 시 뷰어 대신 실패 배너와 `재처리`, `원본 다운로드` 버튼을
  우선 배치한다.
- 부품 상세에서 도면 상세로 이동할 수 있는 딥링크를 유지한다.
- 공통 상태로 `loading`, `empty`, `error`, `read only`,
  `permission denied` UI를 준비한다.

## 10. 완료 조건

완료 조건은 사용자가 최신 도면을 찾고, 실패 원인을 이해하고, 기준본을
배포할 수 있는지로 판단한다.

- 사용자는 목록에서 도면번호, 도면명, 부품번호로 도면을 찾을 수 있다.
- 사용자는 상세에서 승인 상태와 변환 상태를 동시에 확인할 수 있다.
- 사용자는 지원 포맷 파일을 등록하면 변환 진행 상태를 즉시 확인할 수 있다.
- 변환 실패 시 사용자는 실패 코드와 대응 액션을 구분할 수 있다.
- `Released` 전환은 필수 메타데이터와 변환 완료 조건을 통과한 경우에만
  가능하다.
- 대표 연결 부품 교체 후 부품 상세의 관련 도면 정보도 함께 갱신된다.
- 2D와 3D 도면은 각각 맞는 뷰어로 열린다.
- `플로우` 탭에서 등록, 승인 배포, 변환 상태를 팀이 함께 검토할 수 있다.

## 11. server2 기준 구현 과제

현재 `server2`의 Drawing 구현은 부품 내부 부속 기능에 가깝다. 독립 메뉴형
도면관리로 올리기 위해 필요한 구현 과제는 아래와 같다.

1. 도면 목록/상세/편집 API 추가
2. `fileId` 단일 입력이 아닌 메타데이터 입력 DTO 확장
3. 승인 상태 전이 유스케이스와 권한 검증 추가
4. 대표 연결 부품 1:1 제약과 교체 정책 정리
5. 변환 실패 재처리 API 추가
