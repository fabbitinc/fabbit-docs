# 부품관리 기획

이 문서는 Fabbit PLM의 부품(Item) 생성·편집·상태 관리와 리비전 정책을
정의한다. 부품은 PLM의 최소 관리 단위이며, BOM·도면·변경관리의 기준
엔티티다.

상세 규칙은 정책 문서에, 화면 산출물은 `screens/`, 플로우 산출물은
`flows/`에 분리한다.

## 0. 문서 운영 원칙

부품관리는 기존 구조를 유지하되 문서 역할을 아래처럼 분리한다.

| 산출물 | 역할 | 우선순위 |
|--------|------|----------|
| `기획.md` | 기능 목표, 화면 책임, 사용자 시나리오, 업무 흐름 기준 | 최상위 |
| `정책.md` | 상태, 리비전, 잠금, 검증, 권한, 예외 규칙 | 최상위 |
| `flows/*.tsx` | 신규 등록, 리비전 상승, 상태 전이 흐름의 시각 검토 자료 | 기획 기준 보조 |
| `screens/*.tsx` | 화면 정보 구조와 UI 배치 검토 자료 | 기획 기준 보조 |
| `screens/*-*.md` | 와이어프레임 리뷰 시 필요한 화면별 보조 정책 | 보조 |

운영 원칙은 아래와 같다.

- 기획과 정책이 SSOT이며, 와이어프레임 상세 정책 문서는 이를 반복 설명하지
  않는다.
- 기획에 포함되지만 아직 제품에 반영되지 않은 기능은
  🔴 (미개발)로 표시한다.
- 플로우는 화면 기준이 아니라 `신규 등록`, `리비전 상승`, `상태 전이` 같은
  업무 기준으로 유지한다.
- 와이어프레임 상세 정책은 기본 CRUD 화면을 요약하고, 복잡한 화면에만
  세부 문서를 할당한다.
- 문서와 TSX가 충돌하면 항상 `기획.md`와 `정책.md`를 우선한다.

## 1. 문제와 목표

- 중소 제조업은 엑셀로 부품 목록을 관리하여 품번 중복, 상태 불일치,
  버전 혼동이 빈번하다.
- 부품이 어떤 단계에 있는지(설계 중인지, 양산 중인지, 단종인지) 시스템이
  추적하지 않으면 잘못된 기준본으로 생산에 들어가는 사고가 발생한다.
- 변경 이력 없이 직접 수정하면 "누가 언제 왜 바꿨는지" 추적이 불가능하다.

목표는 아래 세 가지다.

1. **부품 수명주기 가시화**: 부품이 현재 어떤 단계인지 한눈에 파악한다.
2. **리비전 기반 변경 추적**: 편집 내용이 리비전으로 관리되어 언제든
   이전 상태로 돌아갈 수 있다.
3. **유연한 변경 통제**: 변경관리(ECR/ECO)를 강제할 수도, 생략할 수도 있어
   조직 규모와 성숙도에 맞게 운영한다.

## 2. 주요 사용자

| 사용자 | 주요 관심사 |
|--------|-------------|
| 설계 엔지니어 | 부품 생성, 속성 편집, 도면 연결, 리비전 관리 |
| 설계 리더 | 리비전 승인, 상태 전환, 변경 영향 검토 |
| 생산관리자 | 양산 기준 부품 확인, Released 리비전 기준 작업지시 |
| 구매 담당자 | 부품 상태별 조달 가능 여부 확인, 공급사 정보 참조 |
| 시스템 관리자 | 변경관리 모드 설정, 채번 규칙, 속성 정의 |

## 4. 핵심 시나리오

### 시나리오 1: 신규 부품 등록 🔴 (미개발)

1. 설계 엔지니어가 `새 부품 등록`을 클릭한다.
2. 품번, 품명, 부품 유형(단품/조립품), 카테고리를 입력한다.
3. 저장하면 부품이 `Design` 상태로 생성되고, 리비전 `A`(Draft)가 함께
   만들어진다.
4. 도면 파일을 업로드하고, 속성(재질, 제조사 등)을 입력한다.

### 시나리오 2: 부품 편집과 리비전 상승 🔴 (미개발)

1. 기존 부품의 속성이나 도면을 수정해야 한다.
2. 상세 화면에서 `편집`을 누르면 현재 리비전 상태를 기준으로 분기한다.
3. 현재 리비전이 Draft면 같은 리비전 안에서 수정한다(리비전 상승 없음).
4. 현재 리비전이 Released면 새 리비전 Draft가 자동 생성되고, 그 위에서
   수정한다.
5. 편집 화면은 저장만 담당하고, 리비전 Release는 상세 화면에서 별도
   액션으로 처리한다.

### 시나리오 3: 변경관리 연동 편집 🔴 (미개발)

1. 조직이 변경관리 모드를 활성화한 상태다.
2. Released 리비전을 수정하려면 먼저 변경 요청(ECR)을 생성하거나 기존
   ECR에 연결해야 한다.
3. ECR 승인 후 ECO가 생성되고, ECO 안에서 새 Draft 리비전을 만들어
   수정한다.
4. ECO 완료 시 리비전이 Release된다.

### 시나리오 4: 부품 상태 전환 🔴 (미개발)

1. Prototype이 완료되어 양산 승인을 받는다.
2. 설계 리더가 부품 상태를 `Production`으로 전환한다.
3. Production 상태가 되면 Released 리비전이 작업지시의 기준본으로
   사용된다.

### 시나리오 5: 매핑 기반 부품 업로드

1. 사용자가 목록 화면에서 `부품 업로드`를 눌러 업로드 다이얼로그를 연다.
2. 업로드 대상에 맞는 매핑을 선택하고, Excel 또는 CSV 파일을 올린다.
3. 시스템은 파일 헤더를 읽어 필수 컬럼 누락 여부를 먼저 검증한다.
4. ROOT BOM 매핑이면 파일별 루트 컨텍스트를 보강한 뒤 업로드를 실행한다.
5. 업로드가 시작되면 배치 진행률을 보여주고, 완료 시 목록과 관련 화면을
   다시 조회한다.

### 시나리오 6: 상세 화면에서 도면 등록과 확인

1. 사용자가 부품 상세의 `속성` 탭으로 진입한다.
2. 도면이 없으면 업로드 영역에서 파일을 선택하거나 드래그앤드롭한다.
3. 시스템은 파일 업로드를 완료한 뒤 `file_id`로 도면 등록을 시작한다.
4. 도면 처리 상태가 `PENDING` 또는 `PROCESSING`이면 미리보기 준비 상태를
   보여주고 상세 데이터를 주기적으로 다시 확인한다.
5. 도면 처리가 완료되면 미리보기, 다운로드, 뷰어 진입 액션을 노출한다.
6. 도면 처리가 실패하면 실패 메시지를 보여주고 재업로드로 복구할 수 있어야
   한다.
7. 사용자는 같은 위치에서 도면 삭제를 실행할 수 있다.

### 6.1 화면 정책 문서 할당 기준

부품관리는 화면별 정책 문서를 같은 밀도로 유지하지 않는다. 기본 CRUD 성격이
강한 화면은 요약 문서 1개로 끝내고, 상태·리비전·연결 관계가 복잡한 화면만
세부 문서를 둔다.

| 화면 | 문서 전략 | 이유 |
|------|-----------|------|
| 목록 | `목록-요약.md` 1개 | 검색, 필터, 선택 액션, 업로드/분석 진입이 핵심인 조회 화면 |
| 업로드 다이얼로그 | `업로드다이얼로그-요약.md` 1개 | 매핑 선택, 파일 검증, 배치 진행 상태가 한 흐름으로 이어지는 복합 다이얼로그 |
| 상세 | 영역별 상세 문서 유지 | 상태 전이, 리비전, 도면 처리, 삭제, 연결 정보가 한 화면에 모이는 복합 화면 |
| 편집 🔴 (미개발) | 영역별 상세 문서 유지 | 잠금, 기본 정보, 속성, 변경 사유, 저장 상태 분기가 함께 보이는 복합 화면 |

### 6.2 플로우 기준

플로우는 와이어프레임보다 먼저 검토한다. 구현과 리뷰에서 기준으로 삼아야 할
업무 흐름은 아래 네 가지다.

1. `신규부품등록` 🔴 (미개발)
   목록 진입, 기본 정보 입력, `Item + Rev.A Draft` 생성, 편집 화면 저장까지를
   기준 흐름으로 본다.
2. `부품업로드`
   매핑 선택, 헤더 검증, ROOT BOM 루트 컨텍스트 보강, 업로드 실행, 배치
   진행률 확인까지를 기준 흐름으로 본다.
3. `리비전상승` 🔴 (미개발)
   상세 화면에서 편집을 눌렀을 때 `Draft 재사용 / Released 기준 새 Draft 확보`
   분기를 기준 흐름으로 본다.
4. `상태전이` 🔴 (미개발)
   `Design -> Prototype -> Production -> EOL -> Obsolete` 순방향 전이와
   선행 검증을 기준 흐름으로 본다.

와이어프레임은 이 흐름을 보여주는 참고자료이고, 흐름 해석은 이 섹션과
`정책.md`를 기준으로 맞춘다.

## 7. 데이터 연결

| 엔티티 | 관계 | 설명 |
|--------|------|------|
| Revision | 1:N | 부품은 여러 리비전을 가짐 |
| BomEntry | N:M | 부품은 다른 부품의 하위이자 상위일 수 있음 |
| Drawing | 리비전 경유 | 도면은 리비전에 연결 |
| ECR/ECO | N:M | 변경 요청/지시와 연결 |
| 공급사 | N:M | 부품별 복수 공급사 가능 |
| 작업지시 | N:1 | 양산 부품의 Released 리비전 참조 |
| 조직/프로젝트 | 소속 | 부품은 조직과 프로젝트에 소속 |

### 스키마 변경 사항

기존 Item 스키마에서 `status` 필드를 확장한다.

```
기존: ItemStatus { ACTIVE, OBSOLETE }
변경: ItemStatus { DESIGN, PROTOTYPE, PRODUCTION, EOL, OBSOLETE }
```

기존 데이터 마이그레이션: `ACTIVE` → `PRODUCTION`, `OBSOLETE` → `OBSOLETE`

## 8. MVP 범위

### MVP 포함

- 부품 CRUD (생성, 조회, 수정, 삭제)
  🔴 (미개발)
- 5단계 상태 관리와 순방향 전이
  🔴 (미개발)
- 리비전 생성, Release, Obsolete
  🔴 (미개발)
- 직접 편집 모드 (모드 A)
  🔴 (미개발)
- 기본 검색과 필터
- 매핑 선택 기반 부품 업로드와 배치 진행률 확인
- 감사 로그 🔴 (미개발)
- 단일 편집 화면 분기형 UX (신규 등록 / Draft 수정 / Released 이후 새 Draft
  편집) 🔴 (미개발)
- 다른 사용자가 Draft를 편집 중일 때 읽기 전용 잠금 진입
  🔴 (미개발)
- 커스텀 속성 현장 추가 및 저장
  🔴 (미개발)

### MVP 제외

- 변경관리 연동 모드 (모드 B) → 변경관리 기획 완료 후
- 리비전 비교 (diff)
- 커스텀 속성별 리비전 상승 설정
- 대량 상태 전환
- 부품 복제
- 역방향 상태 전이

## 9. FE 구현 메모

- 목록 화면은 검색, 필터, 상태 배지, 선택 액션, 행 진입 액션을 제공한다.
- 목록 화면은 품번/품명 검색, 카테고리, 수명주기 상태, 도면 유무, 하위 부품
  유무 필터를 지원하고, `유형` 필터는 🔴 (미개발)로 유지한다.
- 목록 화면 상단에는 `속성 분석`, `부품 업로드`, `새 부품` 🔴 (미개발)을
  함께 배치한다.
- 목록에서 부품을 선택하면 `프로젝트 연결`, `선택 내려받기`, `전체 내려받기`,
  `검색 결과 내려받기` 액션이 열린다.
- 목록 테이블은 `품번`, `품명`, `카테고리`, `Rev`, `상태`, `도면`,
  `하위 부품`을 기본 컬럼으로 사용하고, `최종수정일`, `수정자`는
  🔴 (미개발)로 유지한다.
- 업로드 다이얼로그는 `매핑 선택`, `필수 컬럼 확인`, `파일 헤더 검증`,
  `ROOT BOM 루트 컨텍스트 지정`, `덮어쓰기 옵션`, `배치 진행률 확인`
  단계를 한 흐름으로 제공한다.
- 업로드 다이얼로그는 매핑이 없거나 매핑 조회에 실패하면 `속성 분석으로 이동`
  CTA를 노출한다.
- 업로드는 `.xlsx`, `.xls`, `.csv` 다중 파일을 허용하고, 파일 추가 즉시
  헤더 검증 상태를 보여준다.
- 상세 화면은 현재 상태와 현재 리비전에 따라 버튼 노출과 읽기 전용 상태가
  달라져야 한다.
- 상세 화면의 `속성` 탭은 고정 속성 영역과 도면 미리보기 영역을 함께
  배치하고, 도면 전용 별도 탭은 두지 않는다.
- 도면 블록은 `미등록`, `처리 중`, `처리 실패`, `처리 완료` 상태를 모두
  구분해서 보여줘야 한다.
- 도면 업로드는 단일 파일 기준으로 동작하고, 등록 후 `PENDING`,
  `PROCESSING`, `COMPLETED`, `FAILED` 상태를 상세 화면에서 확인할 수
  있어야 한다.
- 도면 뷰어 액션은 `GLB`면 전용 3D 뷰어로 열고, 그 외 viewer URL이 있는
  경우에는 새 탭으로 연다.
- 도면 삭제는 상세 화면의 속성 탭 안에서 실행하고, 완료 후 목록과 상세
  데이터를 다시 조회해야 한다.
- 편집 화면은 `신규 등록`, `Draft 편집`, `Released 기준 새 Draft 생성 후
  편집`을 하나의 레이아웃 안에서 분기형으로 안내해야 한다.
  🔴 (미개발)
- 편집 화면의 완료 액션은 `저장`만 제공하고, `Release`와 `상태 변경`은
  상세 화면에서 처리한다. 🔴 (미개발)
- 편집 화면은 잠금 충돌 시 차단 모달 대신 읽기 전용으로 진입하며, 잠금
  보유자와 자동 해제 기준을 상단에 노출한다.
  🔴 (미개발)
- 커스텀 속성은 MVP에서도 사용자가 현장에서 직접 추가할 수 있어야 하며,
  이름 중복·빈 값 검증이 필요하다.
  🔴 (미개발)
- 공통 상태로 `loading`, `empty`, `error`, `locked`, `permission denied`
  UI를 준비한다. 🔴 (미개발)
- 목록 화면의 와이어프레임 정책은 요약 문서만 유지하고, 세부 규칙은
  `기획.md`와 `정책.md`에서 확인한다.
- 상세 화면과 편집 화면은 상태, 잠금, 검증, 저장 분기가 겹치므로 영역별 상세
  정책 문서를 유지한다. 🔴 (미개발)
- MVP에서는 변경관리 모드 B, 리비전 diff, 대량 상태 전환 UI를 만들지
  않는다.

## 10. 완료 조건

- 사용자는 목록 화면에서 품번/품명 통합 검색과 상태 필터를 함께 사용할 수
  있다.
- 사용자는 목록 화면에서 카테고리, 도면 유무, 하위 부품 유무를 조합해 검색할
  수 있다.
- 사용자는 목록 화면에서 선택한 부품을 프로젝트에 연결하거나 선택/전체/검색
  결과 기준으로 내려받을 수 있다.
- 사용자는 목록 화면에서 `부품 업로드`와 `속성 분석`으로 진입할 수 있다.
- 사용자는 업로드 다이얼로그에서 매핑을 선택하고 여러 파일을 검증한 뒤 배치
  업로드를 실행할 수 있다.
- ROOT BOM 매핑에서는 사용자가 파일별 루트 컨텍스트를 입력하거나 모든 파일에
  일괄 적용할 수 있다.
- 업로드가 완료되면 시스템은 목록과 관련 화면을 새로고침할 수 있어야 한다.
- 사용자는 상세 화면의 `속성` 탭에서 도면을 등록하고, 처리 상태와 실패
  사유를 확인할 수 있다.
- 사용자는 도면 처리 완료 후 미리보기, 다운로드, 뷰어 열기, 삭제를 같은
  영역에서 수행할 수 있다.
- 사용자는 상세 화면에서 현재 상태와 리비전에 맞는 액션만 볼 수 있다.
  🔴 (미개발)
- Released 리비전을 수정하려 할 때 시스템은 같은 리비전 직접 수정 대신 새
  Draft 생성 흐름으로 안내한다.
  🔴 (미개발)
- 사용자는 편집 화면에서 잠금 상태, 필수값 누락, 저장 실패 사유를 즉시
  구분할 수 있다. 🔴 (미개발)
- 사용자는 편집 화면에서 커스텀 속성을 추가·삭제할 수 있고, 저장 전
  검증 오류를 확인할 수 있다.
  🔴 (미개발)
- 상태 전이 제약은 화면 문구와 백엔드 검증이 일치해야 한다.
  🔴 (미개발)
- `플로우` 탭에서 신규 등록, 리비전 상승, 상태 전이를 팀이 함께 검토할 수
  있어야 한다.
