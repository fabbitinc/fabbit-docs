# 부품관리 기획

이 문서는 Fabbit PLM의 부품(Item) 생성·편집·상태 관리와 리비전 정책을
정의한다. 부품은 PLM의 최소 관리 단위이며, BOM·도면·변경관리의 기준
엔티티다.

상세 규칙은 정책 문서에, 화면 설계와 플로우는 Excalidraw 캔버스에 분리했다.

## 1. 문제와 목표

- 중소 제조업은 엑셀로 부품 목록을 관리하여 품번 중복, 상태 불일치,
  버전 혼동이 빈번하다.
- 부품이 어떤 단계에 있는지(설계 중인지, 양산 중인지, 단종인지) 시스템이
  추적하지 않으면 잘못된 기준본으로 생산에 들어가는 사고가 발생한다.
- 변경 이력 없이 직접 수정하면 "누가 언제 왜 바꿨는지" 추적이 불가능하다.

목표는 아래 세 가지다.

1. **부품 수명주기 가시화**: 부품이 현재 어떤 단계인지 한눈에 파악한다.
2. **리비전 기반 변경 추적**: 편집 내용이 리비전으로 관리되어 언제든
   이전 상태로 돌아갈 수 있다.
3. **유연한 변경 통제**: 변경관리(ECR/ECO)를 강제할 수도, 생략할 수도 있어
   조직 규모와 성숙도에 맞게 운영한다.

## 2. 주요 사용자

| 사용자 | 주요 관심사 |
|--------|-------------|
| 설계 엔지니어 | 부품 생성, 속성 편집, 도면 연결, 리비전 관리 |
| 설계 리더 | 리비전 승인, 상태 전환, 변경 영향 검토 |
| 생산관리자 | 양산 기준 부품 확인, Released 리비전 기준 작업지시 |
| 구매 담당자 | 부품 상태별 조달 가능 여부 확인, 공급사 정보 참조 |
| 시스템 관리자 | 변경관리 모드 설정, 채번 규칙, 속성 정의 |

## 3. 메뉴명과 진입 경로

| 구분 | 권장 명칭 | 비고 |
|------|-----------|------|
| 상위 메뉴 | `부품` | `Item`, `품번` 대신 사용자 언어 사용 |
| 하위 메뉴 | `부품 목록` | 메뉴에서는 짧게 |
| 화면 제목 | `부품 상세` | 상세 화면에서 사용 |
| 기본 CTA | `새 부품 등록` | `Item 생성`보다 부드러움 |

진입 경로 예시:

`부품 > 부품 목록 > 새 부품 등록`
`부품 > 부품 목록 > 부품 상세 > 편집`

## 4. 핵심 시나리오

### 시나리오 1: 신규 부품 등록

1. 설계 엔지니어가 `새 부품 등록`을 클릭한다.
2. 품번, 품명, 부품 유형(단품/조립품), 카테고리를 입력한다.
3. 저장하면 부품이 `Design` 상태로 생성되고, 리비전 `A`(Draft)가 함께
   만들어진다.
4. 도면 파일을 업로드하고, 속성(재질, 제조사 등)을 입력한다.

### 시나리오 2: 부품 편집과 리비전 상승

1. 기존 부품의 속성이나 도면을 수정해야 한다.
2. 현재 리비전이 Draft면 같은 리비전 안에서 수정한다(리비전 상승 없음).
3. 현재 리비전이 Released면 새 리비전 Draft가 자동 생성되고, 그 위에서
   수정한다.
4. 수정 완료 후 리비전을 Release하면 새 기준본이 된다.

### 시나리오 3: 변경관리 연동 편집

1. 조직이 변경관리 모드를 활성화한 상태다.
2. Released 리비전을 수정하려면 먼저 변경 요청(ECR)을 생성하거나 기존
   ECR에 연결해야 한다.
3. ECR 승인 후 ECO가 생성되고, ECO 안에서 새 Draft 리비전을 만들어
   수정한다.
4. ECO 완료 시 리비전이 Release된다.

### 시나리오 4: 부품 상태 전환

1. Prototype이 완료되어 양산 승인을 받는다.
2. 설계 리더가 부품 상태를 `Production`으로 전환한다.
3. Production 상태가 되면 Released 리비전이 작업지시의 기준본으로
   사용된다.

## 7. 데이터 연결

| 엔티티 | 관계 | 설명 |
|--------|------|------|
| Revision | 1:N | 부품은 여러 리비전을 가짐 |
| BomEntry | N:M | 부품은 다른 부품의 하위이자 상위일 수 있음 |
| Drawing | 리비전 경유 | 도면은 리비전에 연결 |
| ECR/ECO | N:M | 변경 요청/지시와 연결 |
| 공급사 | N:M | 부품별 복수 공급사 가능 |
| 작업지시 | N:1 | 양산 부품의 Released 리비전 참조 |
| 조직/프로젝트 | 소속 | 부품은 조직과 프로젝트에 소속 |

### 스키마 변경 사항

기존 Item 스키마에서 `status` 필드를 확장한다.

```
기존: ItemStatus { ACTIVE, OBSOLETE }
변경: ItemStatus { DESIGN, PROTOTYPE, PRODUCTION, EOL, OBSOLETE }
```

기존 데이터 마이그레이션: `ACTIVE` → `PRODUCTION`, `OBSOLETE` → `OBSOLETE`

## 8. MVP 범위

### MVP 포함

- 부품 CRUD (생성, 조회, 수정, 삭제)
- 5단계 상태 관리와 순방향 전이
- 리비전 생성, Release, Obsolete
- 직접 편집 모드 (모드 A)
- 기본 검색과 필터
- 감사 로그

### MVP 제외

- 변경관리 연동 모드 (모드 B) → 변경관리 기획 완료 후
- 리비전 비교 (diff)
- 커스텀 속성별 리비전 상승 설정
- 대량 상태 전환
- 부품 복제
- 역방향 상태 전이

## 9. 후속 확장

- 변경관리 연동 모드 (모드 B) 구현
- 리비전 diff 비교 화면
- 부품 복제 (리비전 포함/미포함 선택)
- 대량 상태 전환 (선택한 부품 일괄 Production → EOL)
- 부품 템플릿 (자주 사용하는 속성 세트 저장)
- 대체 부품 관계 관리 (EOL 시 대체품 자동 추천)
- AI 기반 유사 부품 검색
- 공급사 연동 강화 (조달 리드타임, 가격 이력)
