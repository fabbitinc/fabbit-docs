# 부품관리 정책

이 문서는 부품의 상태 전이, 리비전, 편집 모드, 예외 처리에 대한 도메인
규칙을 정의한다. 제품 맥락과 화면 설계는 [[10_제품/기획/plm/04_부품관리/기획|기획]]
문서를 참고한다.

## 1. 부품 상태 (Item Status)

기존 ACTIVE/OBSOLETE 2단계를 5단계 수명주기로 확장한다.

| 상태 | 영문 | 의미 | 편집 가능 여부 |
|------|------|------|----------------|
| 설계 | Design | 초기 설계 단계, 자유롭게 수정 가능 | 자유 편집 |
| 시제 | Prototype | 시제품 제작/검증 단계 | 편집 가능 (리비전 추적) |
| 양산 | Production | 양산 승인, 현장 기준본으로 사용 | 제한 편집 (리비전 필수) |
| 단종 예고 | EOL | 생산 종료 예정, 대체품 준비 기간 | 속성 편집만 허용 |
| 폐기 | Obsolete | 완전 폐기, 신규 사용 불가 | 편집 불가 (읽기 전용) |

### 1.1 상태 전이 규칙

```text
Design → Prototype → Production → EOL → Obsolete
  ↑
  └─────── (되돌림: 관리자 전용, 사유 필수) ──┘
```

- **순방향 전이**: 각 상태에서 다음 상태로만 이동할 수 있다.
- **역방향 전이**: Production 이전 단계로의 되돌림은 관리자만 가능하며
  사유를 필수 입력해야 한다.
- **건너뛰기**: Design → Production 같은 건너뛰기는 허용하지 않는다.
  단, Design → Prototype은 시제 없이 바로 양산하는 경우가 거의 없으므로
  정책적으로 막는다.
- **Obsolete에서 복원**: Obsolete 상태에서 Design으로 되돌리는 것은
  관리자 전용이며 새 리비전을 강제 생성한다.

### 1.2 상태 전이 시 검증

| 전이 | 검증 조건 |
|------|-----------|
| Design → Prototype | Released 리비전 1개 이상 |
| Prototype → Production | Released 리비전 존재 + 필수 속성 완료 |
| Production → EOL | 대체 부품 정보 입력 권장(필수 아님) |
| EOL → Obsolete | 진행 중인 작업지시 없음 확인 |
| 역방향 전이 | 관리자 권한 + 사유 입력 필수 |

### 1.3 상태별 제약 사항

| 상태 | BOM 편집 | 도면 교체 | 속성 편집 | 삭제 | 작업지시 연결 |
|------|----------|-----------|-----------|------|---------------|
| Design | O | O | O | O (하위 없을 때) | X |
| Prototype | O (리비전) | O (리비전) | O | X | X |
| Production | O (리비전+변경관리) | O (리비전+변경관리) | △ (일부) | X | O |
| EOL | X | X | △ (대체품 정보 등) | X | X (신규 불가) |
| Obsolete | X | X | X | X | X |

## 2. 리비전 정책

### 2.1 리비전 체계

- 리비전 번호는 알파벳 순서를 기본으로 한다: A, B, C, ...
- 조직 설정으로 숫자 방식(1.0, 1.1, 2.0)도 선택 가능하다.
- 하나의 부품에 동시에 존재할 수 있는 Draft 리비전은 최대 1개다.

### 2.2 리비전 상태

| 상태 | 의미 |
|------|------|
| Draft | 작성 중, 자유 편집 가능 |
| Released | 승인·배포된 기준본 |
| Obsolete | 이전 기준본, 읽기 전용 |

### 2.3 리비전 상승 조건

리비전이 새로 만들어지는(상승하는) 조건은 아래와 같다.

| 조건 | 리비전 상승 여부 | 설명 |
|------|------------------|------|
| Draft 리비전 내 수정 | 상승 안 함 | 같은 Draft 안에서 자유 수정 |
| Released 리비전 이후 수정 시도 | 새 Draft 자동 생성 | Released는 불변, 새 리비전에서 수정 |
| 변경관리 모드에서 ECO 생성 시 | 새 Draft 자동 생성 | ECO에 연결된 Draft 리비전 |
| Draft → Released 승격 | 리비전 번호 확정 | A(Draft) → A(Released) |

### 2.4 리비전 상승을 유발하는 변경 항목

모든 편집이 리비전 상승을 유발하지는 않는다. 변경 항목을 두 범주로
나눈다.

**리비전 상승 필요 (Revision-triggering changes)**:

- 도면 파일 교체
- BOM 구성 변경 (하위 부품 추가/삭제/수량 변경)
- 핵심 설계 속성 변경 (재질, 치수, 스펙 등)
- 부품 유형 변경

**리비전 상승 불필요 (Non-revision changes)**:

- 부품 설명, 메모 수정
- 카테고리, 태그 변경
- 공급사 정보 업데이트
- 관리 속성 변경 (담당자, 프로젝트 배정 등)

조직 관리자는 커스텀 속성별로 리비전 상승 필요 여부를 설정할 수 있다.

## 3. 편집 모드

Fabbit은 두 가지 편집 모드를 제공한다. 조직 설정에서 선택한다.

### 3.1 모드 A: 직접 편집 모드 (변경관리 비활성)

GitHub에서 브랜치 보호 규칙이 없는 것과 동일하다. 중소기업 초기 도입에
적합하다.

- Released 리비전 수정 시 → 새 Draft 리비전 자동 생성
- Draft에서 자유 편집 후 직접 Release
- 승인 절차 없이 즉시 기준본 교체
- 리비전 이력은 자동 기록

### 3.2 모드 B: 변경관리 연동 모드 (변경관리 활성)

GitHub에서 브랜치 보호 규칙을 활성화한 것과 동일하다. 품질 체계가
필요한 조직에 적합하다.

- Released 리비전 수정 시 → ECR 생성 또는 기존 ECR 연결 필수
- ECR 승인 → ECO 생성 → ECO 안에서 새 Draft 리비전 생성
- ECO 완료 시 리비전 Release
- 승인자 지정, 검토 이력, 영향 분석 포함

### 3.3 모드 전환 정책

- 모드 A → 모드 B: 즉시 전환 가능. 이미 존재하는 Draft 리비전은
  ECR 없이 Release 허용(경과 조치).
- 모드 B → 모드 A: 진행 중인 ECR/ECO가 있으면 완료 또는 취소 후 전환.
- 모드 A에서 모드 B로 전환 시, 이미 존재하는 Draft 리비전은 ECR 연결
  없이 Release할 수 있는 유예 기간(7일)을 준다.
- 유예 기간 이후에는 미완료 Draft도 ECR 연결이 필요하다.

## 4. 정책과 예외

### 4.1 동시 편집 제어

- 같은 리비전을 두 사용자가 동시에 편집할 수 없다. 먼저 편집을 시작한
  사용자가 잠금(Lock)을 획득한다.
- 잠금은 편집 완료 또는 30분 비활동 시 자동 해제된다.
- 관리자는 강제 잠금 해제가 가능하다.

### 4.2 삭제 정책

- Design 상태 + BOM 참조 없음 + 변경 이력 없음 → 물리 삭제 가능
- 그 외 → 물리 삭제 불가, Obsolete 상태 전환으로 대체
- 리비전 개별 삭제는 Draft 상태만 가능 (Released/Obsolete 삭제 불가)

### 4.3 품번 변경 정책

- 부품 생성 후 품번 변경은 Design 상태에서만 허용
- Prototype 이후에는 품번 변경 불가 (새 부품 생성으로 대체)
- 품번 변경 시 기존 품번은 이력에 기록
