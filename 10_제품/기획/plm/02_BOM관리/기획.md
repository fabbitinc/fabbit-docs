# BOM 관리 기획

이 문서는 Fabbit PLM의 구조화 BOM 조회, 계층 탐색, Where-Used 확인,
엑셀 내보내기, 합성 반영 이력 확인 경험을 정의한다. 현재 `server2`
구현은 `Part`와 `BomLink`를 중심으로 BOM을 조회하는 수준까지 제공하므로,
MVP는 읽기 전용 탐색에 집중하고 수동 편집과 리비전은 후속 범위로 분리한다.

상세 규칙은 정책 문서에, 화면 산출물은 `screens/`, 플로우 산출물은
`flows/`에 분리한다.

## 1. 문제와 목표

중소 제조업은 BOM을 엑셀과 도면 주석으로 관리하여 상하위 관계, 수량,
최신본 여부를 빠르게 확인하지 못한다. 부품 하나가 어디에 쓰이는지
즉시 찾지 못하면 설계 변경 영향 분석, 구매 검토, 생산 기준본 확인이
모두 느려진다.

목표는 아래 네 가지다.

1. 구조화 조회를 제공해 BOM을 상하위 트리로 즉시 탐색한다.
2. 부품 중심 Where-Used를 제공해 변경 영향 범위를 빠르게 확인한다.
3. 현재 화면 기준의 트리를 그대로 내보내 운영팀과 공유할 수 있게 한다.
4. BOM 데이터가 어떤 합성 작업으로 반영됐는지 운영 이력을 남긴다.

## 2. 주요 사용자

이 기능은 설계팀만 아니라 생산, 구매, 품질 담당자가 공통 기준본을
읽는 화면으로 설계한다.

| 사용자 | 주요 관심사 |
|--------|-------------|
| 설계 엔지니어 | 상하위 BOM 탐색, Where-Used 확인, 변경 영향 범위 파악 |
| 설계 리더 | 조립품 기준본 검토, 트리 내보내기, 반영 이력 확인 |
| 생산관리자 | 양산 기준 조립품의 하위 구성 확인, 역전개로 영향 품목 확인 |
| 구매 담당자 | 특정 부품이 어느 조립품에 쓰이는지 확인, 수량 검토 |
| 시스템 관리자 | 합성 반영 결과 모니터링, 실패 이력 점검 |

## 3. 메뉴명과 진입 경로

메뉴는 부품 관리의 하위 탐색 기능이 아니라 독립 업무 화면처럼 보이되,
부품 상세에서도 자연스럽게 진입할 수 있어야 한다.

| 구분 | 권장 명칭 | 비고 |
|------|-----------|------|
| 상위 메뉴 | `부품` | 기존 PLM 메뉴 유지 |
| 하위 메뉴 | `BOM 관리` | `BOM 트리`보다 업무 중심 표현 |
| 화면 제목 | `BOM 목록`, `BOM 상세`, `BOM 가져오기 이력` | 화면별 제목 |
| 기본 CTA | `BOM 보기`, `역전개 보기`, `엑셀 내보내기` | 모두 읽기 전용 액션 |

진입 경로 예시는 아래와 같다.

`부품 > BOM 관리 > BOM 목록 > BOM 상세`
`부품 > 부품 상세 > 연결 정보 > BOM 하위`
`부품 > 부품 상세 > 연결 정보 > Where-Used`
`부품 > BOM 관리 > BOM 가져오기 이력`

## 4. 화면 구성

MVP 화면은 읽기 전용 탐색에 집중한다. 수동 편집 화면은 이번 범위에 넣지
않고, 조회 화면 안에서 검색, 방향 전환, 내보내기, 이력 확인을 끝낼 수 있게
구성한다.

| 화면 | 목적 | 주요 CTA | 필수 상태 |
|------|------|----------|-----------|
| `목록.tsx` | BOM 루트 후보인 조립품을 검색·필터하고 상세 진입점을 제공 | `BOM 보기`, `가져오기 이력` | loading, empty, error, permission denied |
| `상세.tsx` | 선택한 부품의 정전개/역전개 트리와 1-depth 관계 요약 제공 | `정전개`, `역전개`, `엑셀 내보내기`, `부품 상세로 이동` | loading, empty, error, read only |
| `가져오기이력.tsx` | BOM 반영 합성 작업의 상태와 실패 메시지 확인 | `새로고침`, `작업 상세 보기` | loading, empty, error |

모든 화면은 기본적으로 읽기 전용이다. 잠금 상태는 수동 편집을 지원하는
후속 범위에서만 필요하므로, MVP에서는 편집 잠금 UI를 두지 않는다.

## 5. 핵심 시나리오

MVP에서 반드시 검증할 사용 시나리오는 아래 네 가지다.

### 시나리오 1: 조립품 기준 BOM 정전개 조회

1. 사용자가 `BOM 목록`에서 조립품을 검색한다.
2. 목록에서 행을 클릭하면 `BOM 상세`로 이동한다.
3. 시스템은 부품 요약, 직접 자식/직접 부모 요약, 정전개 트리를 함께
   조회한다.
4. 사용자는 트리에서 하위 부품의 수량과 상태를 한 번에 확인한다.

### 시나리오 2: 단품 기준 Where-Used 역전개 조회

1. 사용자가 부품 상세 또는 BOM 상세에서 특정 단품을 선택한다.
2. `역전개 보기`를 누르면 `direction=REVERSE`로 트리를 다시 조회한다.
3. 시스템은 해당 부품을 참조하는 상위 조립품 체인을 반환한다.
4. 사용자는 변경 영향 범위를 상위 제품 기준으로 파악한다.

### 시나리오 3: 현재 트리 엑셀 내보내기

1. 사용자가 `BOM 상세`에서 원하는 방향을 확정한다.
2. `엑셀 내보내기`를 누르면 현재 루트와 방향 기준으로 파일 다운로드를
   요청한다.
3. 시스템은 트리를 평탄화해 `level`, `part_number`, `quantity` 등의
   컬럼으로 반환한다.
4. 사용자는 운영 회의나 외부 공유에 바로 활용한다.

### 시나리오 4: 합성 반영 결과 확인

1. 관리자가 `BOM 가져오기 이력`으로 이동한다.
2. 시스템은 최신 합성 작업 목록을 상태와 오류 수와 함께 보여 준다.
3. `FAILED` 작업이 있으면 오류 메시지를 펼쳐 원인을 확인한다.
4. `COMPLETED` 작업은 반영된 관계 수를 보고 BOM 갱신 여부를 판단한다.

## 6. 사용자/시스템 플로우

상세 플로우는 별도 컴포넌트로 관리하고, 이 기획에서는 어떤 흐름을 팀이
검토해야 하는지만 정의한다.

- 와이어프레임: `screens/목록.tsx`, `screens/상세.tsx`,
  `screens/가져오기이력.tsx`
- 플로우: `flows/트리조회.tsx`, `flows/WhereUsed조회.tsx`,
  `flows/합성반영.tsx`

플로우별 초점은 아래와 같다.

- `트리조회`: 목록 진입, 루트 선택, 정전개 조회, 빈 상태 처리
- `WhereUsed조회`: 역전개 전환, 상위 조립품 확인, 루트 재설정
- `합성반영`: `PENDING -> PROCESSING -> COMPLETED | FAILED` 상태 전이

## 7. 데이터 연결

BOM 관리는 독립 마스터를 새로 만드는 대신 기존 `Part`, `BomLink`,
`SynthesisJob`을 조합해 읽기 경험을 만든다.

| 대상 | 사용 API | 주요 필드/용도 |
|------|----------|----------------|
| BOM 루트 목록 | `GET /api/v1/parts` | `search`, `category`, `lifecycle_state`, `has_children`, `project_id`, `childrenCount` |
| 루트 부품 요약 | `GET /api/v1/parts/{partId}` | `partNumber`, `name`, `revision`, `lifecycleState`, `childrenCount`, `parentsCount` |
| 직접 관계 요약 | `GET /api/v1/parts/{partId}/bom` | `children[]`, `parents[]`, `quantity`, `extendedProperties` |
| 계층 트리 | `GET /api/v1/parts/{partId}/bom/tree` | `direction`, `root`, `totalCount`, `children[]` |
| 엑셀 내보내기 | `GET /api/v1/parts/{partId}/bom/tree/export` | 현재 루트·방향 기준 `.xlsx` 다운로드 |
| 반영 이력 | `GET /api/v1/synthesis`, `GET /api/v1/synthesis/{jobId}` | `status`, `processedRows`, `relationshipsCreated`, `errors[]` |

추가 데이터 의존성도 있다. 합성 이력 화면에서 사람이 읽을 파일명을 보여 주려면
현재 응답의 `fileId`를 파일 메타데이터와 결합해야 하므로, FE는 별도 파일 조회
또는 BE 응답 확장을 고려해야 한다.

## 8. MVP 범위

이번 MVP는 읽기 경험을 견고하게 만드는 데 집중한다. BOM을 직접 수정하는
기능은 정책과 동시성 설계가 더 필요하므로 후속으로 남긴다.

### MVP 포함

- `has_children=true` 기준의 BOM 루트 목록 조회
- 품번·품명 검색, 카테고리/상태 필터, 프로젝트 컨텍스트 연계
- 루트 부품 요약과 직접 관계 요약 조회
- 정전개(`FORWARD`)와 역전개(`REVERSE`) 트리 조회
- 현재 방향 기준 엑셀 내보내기
- 합성 작업 이력 목록과 작업 상태 확인
- loading, empty, error, permission denied, read only 상태 처리

### MVP 제외

- BOM 링크 수동 추가/삭제/수량 편집
- BOM 리비전, 베이스라인, diff 비교
- 대체품, 옵션품, Effectivity 관리
- 트리 드래그 앤 드롭, 일괄 편집, 일괄 승인
- MBOM 변환, 공정 라우팅 연계
- 사이클 경고 전용 UI와 그래프 시각화

## 9. FE 구현 메모

FE는 기존 Part API를 재사용해 BOM 화면을 조립해야 한다. 화면별 데이터
요청 단위를 분리해 로딩 체감을 낮추는 것이 중요하다.

- 목록 화면은 `GET /api/v1/parts`를 사용하고 기본 필터로
  `has_children=true`를 넣는다.
- 상세 화면은 `GET /api/v1/parts/{id}`, `GET /api/v1/parts/{id}/bom`,
  `GET /api/v1/parts/{id}/bom/tree`를 병렬 호출한다.
- 방향 전환 시에는 `bom/tree`만 재조회하고, 루트 요약과 직접 관계 요약은
  그대로 유지한다.
- 역전개 진입은 부품 상세의 `Where-Used` 클릭 컨텍스트를 그대로 받아
  `direction=REVERSE` 초기값으로 연다.
- 내보내기 버튼은 트리 로딩 중, 오류 상태, 빈 루트 상태에서 비활성화한다.
- 합성 이력 화면은 `PENDING`, `PROCESSING` 작업에 대해 10초 간격 폴링을
  지원한다.
- BOM 화면에는 수동 편집 CTA를 노출하지 않는다. 편집은 후속 범위다.

## 10. 완료 조건

이 기능이 완료됐다고 판단하려면 아래 조건을 모두 만족해야 한다.

- 사용자는 `BOM 목록`에서 조립품을 검색하고 상세로 이동할 수 있다.
- 사용자는 `BOM 상세`에서 정전개와 역전개를 전환해도 루트 기준이 유지된
  상태로 결과를 볼 수 있다.
- 사용자는 직접 관계 요약과 계층 트리를 구분해 이해할 수 있다.
- 사용자는 현재 화면의 트리를 `.xlsx`로 내보낼 수 있다.
- 사용자는 `BOM 가져오기 이력`에서 작업 상태와 실패 메시지를 확인할 수
  있다.
- 수동 편집, 리비전, diff가 아직 미구현이라는 사실이 화면과 문서에
  명확히 드러나야 한다.

## 11. 후속 확장

후속 단계에서는 읽기 중심 BOM을 운영 중심 BOM으로 확장한다.

- BOM 링크 수동 편집과 변경 사유 기록
- BOM 베이스라인/리비전과 부품 리비전 연동
- BOM diff 비교 화면
- 대체품, 옵션품, phantom, scrap rate 속성 관리
- MBOM 파생과 공정 라우팅 연계
- 합성 결과 검토 후 선택 반영하는 승인형 워크플로우
