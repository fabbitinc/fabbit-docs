# BOM 관리 정책

이 문서는 BOM 조회, 방향 전환, 내보내기, 합성 반영 이력에 대한 도메인
규칙을 정의한다. 현재 MVP는 `Part`와 `BomLink` 기반의 읽기 전용 EBOM
탐색을 대상으로 하며, 수동 편집과 BOM 리비전은 후속 범위다.

## 1. 데이터 모델

MVP는 새 BOM 마스터를 만들지 않고 기존 엔티티를 조합해 화면을 구성한다.
핵심은 `BomLink`의 제약을 명확히 하고, 화면 모델이 어떤 응답을 신뢰하는지
정의하는 것이다.

### 1.1 핵심 엔티티

| 엔티티 | 역할 | 주요 필드 |
|--------|------|-----------|
| `Part` | BOM 루트와 노드의 기준 엔티티 | `id`, `partNumber`, `name`, `revision`, `category`, `lifecycleState` |
| `BomLink` | 상위 부품과 하위 부품의 연결 | `parentPartId`, `childPartId`, `quantity`, `extendedProperties` |
| `BomTreeNode` | 트리 조회 응답 노드 | `id`, `partNumber`, `name`, `revision`, `quantity`, `children[]` |
| `PartBomResponse` | 직접 부모/직접 자식 1-depth 요약 | `children[]`, `parents[]` |
| `SynthesisJob` | BOM 반영 배치의 운영 이력 | `status`, `processedRows`, `relationshipsCreated`, `errors[]` |

### 1.2 `BomLink` 필드 제약

`BomLink`는 현재 서버 구현에서 가장 직접적인 BOM 엔티티다. 필드 단위 제약은
아래와 같다.

| 필드 | 제약 |
|------|------|
| `parentPartId` | 필수, null 불가 |
| `childPartId` | 필수, null 불가 |
| `(parentPartId, childPartId)` | 복합 유니크, 중복 링크 불가 |
| `quantity` | 1 이상 정수 |
| `parentPartId == childPartId` | 금지, 자기 참조 불가 |
| `extendedProperties` | null/blank 입력 시 `{}`로 정규화 |

### 1.3 MVP에서 없는 필드

실무 BOM에 자주 필요한 속성도 있다. 다만 현재 구현에는 아래 필드가 없으므로
문서에서 후속 범위로 명시한다.

- 위치 순번(`find number`)
- 대체품 / 옵션품 구분
- 유효일자(`effectivity`)
- 스크랩율, 투입손실율
- BOM 노트의 별도 구조화 필드

## 2. 상태 관리

BOM 링크 자체는 MVP에서 별도 라이프사이클을 갖지 않는다. 대신 조회 화면은
`방향 상태`, 합성 반영은 `작업 상태`, 미래 확장은 `BOM 베이스라인 상태`로
구분해 관리한다.

### 2.1 BOM 링크 상태

`BomLink`는 현재 `Draft`, `Released` 같은 독립 상태를 가지지 않는다.
링크의 사용 가능 여부는 상위·하위 `Part.lifecycleState`에 간접적으로
영향받고, 화면은 링크를 항상 읽기 전용으로 취급한다.

따라서 MVP 정책은 아래와 같다.

- 링크 자체 상태 전이는 없다.
- 링크 편집 액션은 노출하지 않는다.
- 링크 가시성은 권한과 루트 부품 존재 여부로만 결정한다.

### 2.2 BOM 탐색 상태

화면이 유지해야 할 탐색 상태는 아래와 같다.

```text
READY -> LOADING -> READY
                 -> EMPTY
                 -> ERROR
```

- `READY -> LOADING`: 사용자가 루트를 바꾸거나 방향을 바꿀 때 전이한다.
- `LOADING -> READY`: 트리와 직접 관계 요약을 정상 수신하면 전이한다.
- `LOADING -> EMPTY`: 정전개에서 하위가 없거나 역전개에서 상위가 없으면
  전이한다.
- `LOADING -> ERROR`: 권한 부족, 잘못된 방향값, 네트워크 실패, 서버 예외가
  발생하면 전이한다.

실패 조건은 아래와 같다.

- `partId`가 없거나 존재하지 않으면 조회를 시작하지 않는다.
- `direction`이 허용값이 아니면 `VALIDATION_ERROR`로 실패한다.
- 권한이 없으면 `permission denied` 상태로 전환한다.

### 2.3 합성 작업 상태

현재 서버 구현에 존재하는 `SynthesisJobStatus`는 BOM 반영 이력의 공식 상태다.

```text
PENDING -> PROCESSING -> COMPLETED
                     -> FAILED
```

| 상태 | 진입 조건 | 종료 조건 | 실패 조건 |
|------|-----------|-----------|-----------|
| `PENDING` | 합성 작업이 생성됨 | 워커가 작업을 시작함 | 없음 |
| `PROCESSING` | 실제 행 처리 시작 | 모든 행 처리 완료 또는 중단 | 파싱/매핑/저장 오류 누적 |
| `COMPLETED` | 작업이 정상 종료됨 | 종료 상태 | 없음 |
| `FAILED` | 작업이 중단됨 | 종료 상태 | 재시도는 새 작업 생성으로만 가능 |

### 2.4 후속 BOM 베이스라인 상태

후속 범위에서는 BOM 자체의 기준본 관리를 위해 아래 상태를 추가한다.

```text
DRAFT -> RELEASED -> OBSOLETE
```

이 상태는 수동 편집과 diff 기획이 확정된 뒤 별도 문서로 분리한다.

## 3. 권한

BOM은 여러 부서가 공통으로 읽는 기준 데이터이므로 조회 권한을 넓게 두고,
운영성 있는 이력 확인과 향후 편집 권한은 좁게 둔다.

| 역할 | 목록/상세 조회 | 역전개 조회 | 엑셀 내보내기 | 합성 이력 조회 | 수동 편집 |
|------|----------------|------------|---------------|----------------|-----------|
| 설계 엔지니어 | O | O | O | O | 후속 |
| 생산/구매 조회자 | O | O | O | 프로젝트 권한 내 O | 후속 |
| 시스템 관리자 | O | O | O | O | 후속 |
| 외부 협력사 | MVP 제외 | MVP 제외 | MVP 제외 | MVP 제외 | MVP 제외 |

권한 상세 규칙은 아래와 같다.

- BOM 화면은 기본 읽기 전용이다.
- `합성 이력`은 최소한 관련 프로젝트 또는 조직 데이터를 볼 수 있는
  사용자에게만 노출한다.
- 수동 편집이 없는 MVP에서는 잠금, 선점, 승인 권한 정책을 만들지 않는다.

## 4. 유효성 검증

조회 기능도 입력 검증과 실패 코드 정리가 필요하다. 프론트와 백엔드는 아래
규칙을 동일하게 이해해야 한다.

| 항목 | 검증 규칙 |
|------|-----------|
| `partId` | 필수, 존재하는 `Part`여야 함 |
| `direction` | `FORWARD`, `REVERSE`만 허용, 빈 값은 `FORWARD` |
| `quantity` | 응답/저장 모두 1 이상이어야 함 |
| 트리 깊이 | 서버 기준 최대 30단계 |
| 엑셀 내보내기 | 현재 루트와 방향이 확정된 상태에서만 허용 |
| 합성 상태 폴링 | `PENDING`, `PROCESSING`만 반복 조회 |

오류 처리 기준은 아래와 같다.

- `VALIDATION_ERROR`: 잘못된 방향값, 잘못된 파라미터 형식
- `NOT_FOUND`: 존재하지 않는 `partId`
- `PERMISSION_DENIED`: 권한 부족
- `INTERNAL_SERVER_ERROR`: 엑셀 생성 실패 등 서버 내부 오류

## 5. 비즈니스 규칙

MVP의 제품 규칙은 "부품 중심 탐색"을 전제로 한다. BOM을 독립 문서처럼
관리하지 않고, 특정 부품을 루트로 잡아 전개하는 방식을 유지한다.

### 5.1 목록 규칙

- 기본 목록은 `has_children=true`인 부품을 대상으로 한다.
- 단품의 Where-Used 조회는 목록이 아니라 부품 상세 또는 검색 진입으로
  처리한다.
- 목록 정렬은 품번 오름차순 또는 최근 사용 맥락 기준을 후속으로 검토한다.
  MVP는 현재 Part API 기본 정렬을 따른다.

### 5.2 상세 규칙

- `BOM 상세`는 루트 부품 요약, 직접 관계 요약, 계층 트리를 분리해 보여 준다.
- 직접 관계 요약은 `GET /bom`, 계층 트리는 `GET /bom/tree`를 사용한다.
- 방향 전환은 같은 루트에서만 일어나며, 루트를 암묵적으로 바꾸지 않는다.
- 사용자가 트리 노드를 클릭해 새 루트로 이동하는 것은 허용하되, 현재 루트와
  방향을 상단에 명시해야 한다.

### 5.3 역전개 규칙

- `REVERSE`는 Where-Used와 동일 의미로 사용한다.
- 역전개 결과가 비어 있으면 오류가 아니라 정상 빈 상태로 처리한다.
- 역전개에서도 수량은 해당 부모가 현재 노드를 몇 개 포함하는지를 의미한다.

### 5.4 내보내기 규칙

- 엑셀 내보내기는 현재 조회 중인 트리만 반출한다.
- 컬럼 순서는 `level`, `part_number`, `name`, `revision`, `quantity`,
  `material`, `unit`, `category`, `lifecycle_state`를 유지한다.
- `mapping_id` 쿼리 파라미터는 현재 구현에서 예약 값으로만 보고, MVP UI에는
  노출하지 않는다.

### 5.5 합성 이력 규칙

- 합성 이력은 최신순으로 보여 준다.
- `FAILED` 작업은 오류 메시지를 펼쳐 볼 수 있어야 한다.
- `COMPLETED` 작업은 `relationshipsCreated`를 핵심 반영 지표로 사용한다.
- 재처리는 기존 작업 재실행이 아니라 새 합성 작업 시작으로 처리한다.

### 5.6 후속 규칙

아래 규칙은 이번 MVP에서 다루지 않는다.

- BOM 링크 수동 생성/수정/삭제
- BOM 변경 사유와 감사로그
- BOM baseline 승인/배포
- diff 비교와 영향 승인

## 6. API/이벤트 계약

현재 BOM 관리에 필요한 API는 대부분 이미 존재한다. 별도 실시간 이벤트는
없으므로, 합성 상태는 폴링으로 관리한다.

| 계약 | 설명 |
|------|------|
| `GET /api/v1/parts` | BOM 루트 목록. `has_children=true` 기본 사용 |
| `GET /api/v1/parts/{partId}` | 루트 요약과 직접 부모/자식 카운트 |
| `GET /api/v1/parts/{partId}/bom` | 직접 부모/직접 자식 1-depth 관계 |
| `GET /api/v1/parts/{partId}/bom/tree` | 정전개/역전개 계층 트리 |
| `GET /api/v1/parts/{partId}/bom/tree/export` | 현재 트리 엑셀 파일 |
| `GET /api/v1/synthesis` | 최신 합성 작업 목록 |
| `GET /api/v1/synthesis/{jobId}` | 개별 합성 작업 상세 |

폴링 정책은 아래와 같다.

1. `합성 이력` 화면 진입 시 목록을 1회 조회한다.
2. `PENDING`, `PROCESSING` 작업이 있으면 10초 간격으로 재조회한다.
3. 모든 작업이 종료 상태면 폴링을 중단한다.

MVP에서는 아래 계약이 없다.

- BOM 생성 API
- BOM 수정 API
- BOM 삭제 API
- BOM diff API

## 7. BE 구현 메모

백엔드는 현재 조회 중심 구현을 유지하되, FE가 운영 화면을 만들 때 막히는
부분을 최소한으로 보완해야 한다.

- `PartQuery.getBomTree()`의 깊이 제한 30단계는 문서와 UI 안내에 반영한다.
- `BomLink` 제약은 도메인 테스트로 계속 보호한다.
- `bom/tree/export`의 `mapping_id`는 현재 미사용이므로 실제 활용 여부를
  정리하거나 제거 여부를 판단해야 한다.
- `SynthesisListResponse`는 현재 `fileId`만 주므로, 파일명 표시가 필요하면
  응답 확장 또는 별도 파일 조회 계약을 추가해야 한다.
- BOM 목록에서 `parentsCount`나 최근 반영 시각이 필요하면 전용 조회 DTO를
  추가하는 것이 낫다.
- 수동 편집을 시작할 때는 `BomLink` 감사로그, 충돌 제어, baseline 설계를
  별도 유스케이스로 분리해야 한다.

## 8. 테스트 시나리오

테스트는 성공과 실패를 함께 다뤄야 한다. 특히 방향 전환과 합성 작업 상태는
화면 문구와 서버 동작이 어긋나기 쉽다.

- `GET /api/v1/parts?has_children=true` 호출 시 자식 BOM이 있는 루트만
  목록에 노출된다.
- 정전개 조회에서 하위 부품이 있으면 트리와 `totalCount`가 함께 반환된다.
- 역전개 조회에서 상위 부품이 없으면 오류가 아니라 빈 상태가 반환된다.
- `direction=INVALID` 요청은 `VALIDATION_ERROR`로 실패한다.
- 존재하지 않는 `partId`로 조회하면 `NOT_FOUND`가 반환된다.
- 엑셀 내보내기 파일에는 정의된 9개 헤더가 순서대로 포함된다.
- `BomLink.connect()`는 자기 참조와 `quantity <= 0` 입력을 거부한다.
- 합성 작업은 `PENDING -> PROCESSING -> COMPLETED | FAILED` 외의 전이를
  허용하지 않는다.
- `FAILED` 합성 작업은 오류 메시지 목록을 비워 두지 않는다.
- 권한 없는 사용자는 BOM 상세와 합성 이력 화면에서 `permission denied`
  상태를 본다.
