# PLM 기획 인덱스

이 문서는 `10_제품/기획/plm` 폴더의 상위 진입 문서다. 현재 `plm` 문서군은
Fabbit의 설계 기준본 관리 경험을 다루는 활성 기획 묶음이며, archive 후보가
아니다. 다만 `site/app` 경계가 정리된 이후 기준에 맞춰 상위 구조와 소유권을
계속 보강해야 하므로, 현 시점 판단은 `존치 + 재편 예정`이다.

## 1. 이 문서군의 역할

`plm` 문서군은 로그인 이후 앱 안에서 사용자가 다루는 설계 기준본 경험을
정리한다. 도면, BOM, 변경관리, 부품 같은 코어 데이터 관리 흐름을 이 폴더에서
다루며, 공개 사이트 메시지나 가입, 구독, 요금 정책은 직접 소유하지 않는다.

이 문서군이 맡는 범위는 아래와 같다.

- 도면 등록, 검토, 배포, 변환 상태 관리
- BOM 조회, 계층 탐색, where-used 확인
- 이슈, 변경요청, 변경 이력 추적
- 부품 기준본 관리와 연결 데이터 탐색

이 문서군이 직접 고정하지 않는 범위는 아래와 같다.

- 공개 랜딩, 공개 요금제, 공개 신청 경험
- 회원가입, 워크스페이스 생성, 플랜 선택, 업그레이드
- 고객 세그먼트 운영 기준과 플랜, 좌석, 과금 정책

## 2. 현재 위치 판단

현재 `plm` 문서군의 위치는 아래처럼 판단한다.

| 항목 | 판단 | 이유 |
|------|------|------|
| 문서 등급 | 활성 코어 기획 | 제품 로드맵과 직접 연결되는 핵심 도메인이다. |
| 구조 상태 | 존치 + 재편 예정 | 하위 문서는 존재하지만 상위 인덱스와 소유권 정리가 늦었다. |
| archive 여부 | archive 후보 아님 | 현재 제품 범위 안의 핵심 경험이므로 보관 문서로 돌리기 이르다. |
| 참조 방식 | 정책 참조형 | 세그먼트, 도입 경로, 플랜 정책은 별도 SSOT를 먼저 따른다. |

즉, `plm`은 유지 대상이다. 다만 지금 상태를 최종 구조로 보지 않고, `app`,
`mes`와의 경계가 더 분명하게 읽히도록 계속 재편하는 편이 맞다.

골격은 각 `기획.md`를 먼저 본다.

## 4. 인접 문서군과의 관계

`plm`은 독립 도메인이지만, 현재 구조에서는 다른 기획 문서군과 분명한 경계를
가져야 한다.

| 문서군 | 역할 |
|--------|------|
| [[10_제품/기획/site/index]] | 공개 방문자에게 제품을 소개하고 파일럿, 데모, 상담으로 전환한다. |
| [[10_제품/기획/app/index]] | 계정 생성 이후 온보딩, 플랜 선택, 구독 변경 흐름을 다룬다. |
| `plm` | 로그인 이후 설계 기준본을 만들고 유지하는 코어 제품 경험을 다룬다. |
| [[10_제품/기획/mes/index]] | PLM 기준본이 현장 실행으로 내려간 뒤 작업과 실적을 관리한다. |

간단히 말해 `site`는 외부 진입, `app`은 계정과 구독 시작, `plm`은 설계
기준본, `mes`는 실행 관리다.

## 5. 먼저 읽는 순서

`plm` 기획을 읽을 때는 아래 순서를 기본으로 잡는다.

1. [[10_제품/전략/아이템-정의]]에서 제품 정의와 상위 가치를 확인한다.
2. [[10_제품/전략/제품-로드맵]]에서 현재 구현 범위와 우선순위를 확인한다.
3. [[10_제품/정책/고객-세그먼트-도입-정책]]에서 어떤 고객을 위해 이 경험을
   설계하는지 확인한다.
4. 플랜, 좌석, 접근 경계가 얽히면 [[40_사업/정책/플랜-과금-정책]]을 참조한다.
5. 그 다음에 도면, BOM, 변경관리, 부품 상세 기획을 읽는다.
