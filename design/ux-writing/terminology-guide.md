# UX 라이팅 및 용어 전략 (Terminology Strategy)

> **핵심 가치:** "중학생도 BOM을 짤 수 있을 만큼 쉬운 제조 솔루션"
>
> 기존 PLM/ERP의 높은 진입 장벽("배우기 어렵다")을 허물기 위해, **'토스(Toss)' 스타일의 직관적인 언어**를 도입합니다. 단, 엔지니어링 데이터의 **정확성**은 타협하지 않습니다.

---

## 1. 명칭 전략 (Naming Convention)

**원칙:** 메인 UI 레이블은 **"기능의 목적(무엇을 하는가?)"**을 서술형으로 쉽게 풀어서 작성하고, 전문 용어는 툴팁이나 상세 설명에 보조적으로 사용합니다.

### 사례별 가이드

| 기능(전문 용어) | **메인 UI (권장)** | 보조 설명 / 툴팁 (전문성 보완) | 비고 |
| :--- | :--- | :--- | :--- |
| **Where-used**<br>(역전개) | **이 부품이 사용되는 곳**<br>또는 **사용처 확인** | "부품 역전개(Where-used) 정보를 확인합니다." | 직관성 최우선 |
| **Revision**<br>(리비전/개정) | **버전 기록**<br>또는 **변경 내역** | "Revision History 및 Diff를 제공합니다." | 정확한 리비전 코드(Rev.A)는 유지 |
| **ECO/ECN**<br>(설계 변경) | **설계 변경 요청** | "Engineering Change Order(ECO)를 생성합니다." | 약어 사용 지양 |
| **Pending**<br>(결재 대기) | **승인 대기 중** | "관리자 확인 후 확정됩니다." | 상태를 명확히 설명 |

## 2. 톤앤매너 (Tone & Manner)

### 2.1. 친절하고 구체적인 가이드 (Micro-copy)
- **Bad:** "데이터 없음"
- **Good:** "아직 등록된 부품이 없어요. 도면을 먼저 업로드해 볼까요?"
- **전략:** 에러 메시지나 빈 상태(Empty State)에서 사용자가 취해야 할 행동을 부드럽게 제안합니다.

### 2.2. 맥락 기반의 제안 (Contextual Action)
- 사용자가 막막해하지 않도록 다음 단계를 미리 제시합니다.
- 예: BOM 생성 완료 후 → "이제 발주서를 만들어볼까요?" 버튼 노출

### 2.3. 상태의 시각화
- 딱딱한 텍스트 대신 컬러와 문구로 상황을 직관적으로 전달합니다.
- 예: `Approval Pending` → `🟠 사장님 확인 대기 중`

---

## 3. 주의사항: 정확성과의 균형

용어는 쉬워지더라도 **핵심 엔지니어링 데이터**는 정확하고 표준화된 표기를 준수해야 합니다.

1.  **식별자(Identifier) 유지:**
    *   품번(Part Number)은 절대 생략하거나 모호하게 표현하지 않습니다.
    *   ❌ 나사
    *   ⭕ M4 볼트 (PN: BLT-M4-001)
2.  **버전(Version) 명시:**
    *   '최신', '구형' 같은 형용사보다 리비전 코드를 병기합니다.
    *   ❌ 최신 버전
    *   ⭕ Rev.02 (최신)

---

## 4. 기대 효과

1.  **진입 장벽 제거:** 현장 관리자와 고령의 사장님도 별도 교육 없이 즉시 사용 가능.
2.  **사용자 경험 차별화:** 딱딱한 레거시 PLM 대비 "내 업무를 도와주는 비서" 같은 느낌 전달.
3.  **세일즈 포인트:** "직원들이 바로 쓸 수 있는 툴"이라는 점을 강조하여 도입 설득력 강화.
